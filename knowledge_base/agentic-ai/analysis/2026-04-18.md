### 1.2 ADK v1.31.0 Session Rewind — PHASE 4 CLOSED-LOOP ROLLBACK OPPORTUNITY (P1 — Architecture)

**Finding**: ADK v1.31.0 introduced Session Rewind — the ability to undo agent actions within a session by rewinding to a checkpoint before a previous invocation. The agent can then retry with different parameters. This is a debugging and correctness feature: if an agent takes a wrong optimization step, the orchestrator can rewind rather than abort.

**Impact on our pipeline**:

Our Phase 4 closed-loop pipeline (`scripts/closed_loop.sh`) has a linear progression: GENERATE → VALIDATE → OPTIMIZE → DEPLOY. If OPTIMIZE takes a wrong direction (description change that doesn't improve trigger rate), the current behavior is:
- OPTIMIZE → VALIDATE retry loop (max 3 retries per `closed_loop.sh` state machine)
- After 3 failures → REPORT_FAILURE

ADK's Session Rewind introduces an alternative pattern:
- OPTIMIZE takes wrong step → **rewind to pre-step checkpoint** → retry with different strategy
- The session context (including what was tried and failed) is preserved, so the agent doesn't repeat the same mistake

This maps to a capability we don't currently have: **mid-session rollback without losing session context**. Our current git-based rollback (`git checkout -- .claude/agents/meta-agent-factory.md`) loses the optimizer's in-session learning about what didn't work.

**Gap**: Our optimizer has no mid-session rollback capability. Git-based rollback is coarse-grained (file-level, not session-level). The optimizer relies on iteration history in `eval/experiment_log.json`, which partially compensates by recording failed approaches — but the optimizer's in-context understanding of failure patterns is lost on each new `claude -p` invocation.

**Cross-pollination opportunity**: While we can't directly use ADK's Session Rewind (it's ADK-specific), the concept applies to our Agent SDK integration path. If Phase 5 uses Agent SDK sessions (`claude-agent-sdk[otel]`), session-level checkpointing could enable similar rewind behavior. The Agent SDK's `get_subagent_messages()` (v0.1.60) already provides transcript inspection, which is a prerequisite for understanding what state to rewind to.

**Recommendation**: P1 — (1) Log this as a Phase 5.3 design consideration: "session-level checkpointing for optimizer rewind." (2) Evaluate whether Agent SDK's session management can provide equivalent checkpoint/rewind semantics when we migrate from `claude -p` to Agent SDK in Phase 5. (3) Near-term: no code changes; the current retry loop is adequate for Phase 4.

### 1.3 Agent SDK OTEL Tracing — PHASE 5 OBSERVABILITY FOUNDATION (P1 — Infrastructure)

**Finding**: Agent SDK v0.1.60 added W3C distributed tracing via `pip install claude-agent-sdk[otel]`. This propagates `TRACEPARENT`/`TRACESTATE` to CLI subprocesses, connecting SDK and CLI traces end-to-end.

**Impact on our pipeline**:

Our current observability stack is file-based:
- Performance JSONs: `logs/performance/{agent}-YYYY-MM-DD.json`
- Session logs: `logs/sessions/{agent}-{date}.jsonl`
- Health dashboard: `scripts/health_dashboard.py`
- Anomaly alerter: `scripts/anomaly_alerter.py`

This works for Phase 4's single-agent sequential execution, but Phase 5's multi-agent topology (parallel specialists, orchestrator-worker patterns) will produce interleaved traces that file-based logging can't correlate.

With OTEL tracing, a Phase 5 pipeline run would appear as a single distributed trace:
```
topology-router → [parallel: factory-agent, validator-agent, optimizer-agent] → deploy-gate
```
Each span would carry timing, token usage, tool calls, and error context. An OTEL collector (Jaeger, Tempo, or even stdout JSON) gives us the same observability Anthropic described in their multi-agent research system blog: "comprehensive observability without conversation content monitoring."

**Gap**: No distributed tracing in our pipeline. Phase 5 ROADMAP mentions observability but doesn't specify OTEL. The Agent SDK feature bridges this gap.

**Recommendation**: P1 — (1) Add `claude-agent-sdk[otel]` to Phase 5 requirements in ROADMAP §5.3.2. (2) Add "Implement OTEL collector for multi-agent pipeline traces" as a new Phase 5 task. (3) Near-term: no code changes; this is a Phase 5 deliverable.

### 1.4 Task Budgets — STEWARD COST CONTROL MECHANISM (P1 — Operational)

**Finding**: Task Budgets entered public beta on Opus 4.7 (`task-budgets-2026-03-13`). An advisory token ceiling across a full agentic loop — the model sees a running countdown. Minimum 20K tokens. Model adjusts behavior (shorter reasoning, fewer tool calls) as budget depletes.

**Impact on our pipeline**:

Our steward agents (factory-steward, researcher) run as nightly crons with duration-based cost ceiling (`scripts/lib/cost_ceiling.sh` — 5× 30-day rolling average). This is a blunt instrument: it measures wall-clock time, not token consumption. A fast run that generates excessive tokens isn't caught; a slow run with efficient token use may trigger false alerts.

Task Budgets provide a finer-grained control at the model level:
- Set a token budget per steward session (e.g., 500K tokens for researcher, 200K for factory-steward)
- Model self-regulates: adjusts reasoning depth and tool call frequency as budget depletes
- Combines with our existing duration ceiling for defense-in-depth

**Gap**: No token-level budget enforcement. Duration ceiling is necessary but insufficient.

**Recommendation**: P1 — (1) Evaluate Task Budgets for factory-steward first (most predictable workload). (2) If available via `claude -p` CLI (verify), add `--task-budget <tokens>` flag to daily scripts. (3) If API-only, defer until Phase 5 Agent SDK migration. (4) **Caution**: Task Budgets may cause steward agents to truncate work prematurely on legitimately complex sessions. Set conservative initial budgets (2× observed median) and monitor for truncation artifacts.

### 1.5 Opus 4.7 "Fewer Subagents" — DELEGATION REGRESSION RISK (P2 — Monitoring)

**Finding**: Opus 4.7 documentation and behavioral analysis confirm: the model spawns fewer subagents by default. It prefers reasoning through problems rather than delegating to specialists. Described as "more direct/opinionated tone" and "fewer tool calls by default (prefers reasoning)."

**Impact on our pipeline**:

Our steward agents (`factory-steward`, `researcher`) rely on subagent spawning for:
- Factory-steward: delegates to meta-agent-factory, skill-quality-validator, autoresearch-optimizer
- Researcher: delegates to specialized search/fetch subtasks

If Opus 4.7 reduces delegation, steward agents might attempt to do factory/validation/optimization work themselves rather than properly delegating. This could manifest as:
- Steward spending 30+ minutes on a task that should have been delegated in 5 minutes
- Steward producing lower-quality output because it didn't use the specialized agent
- No error — just silent quality degradation

**Mitigation already partially in place**: Our steward skill uses explicit delegation instructions ("Use the meta-agent-factory agent to..."), which should override the model's default preference. L13 in ROADMAP documents this lesson: "every script-initiated `claude -p` call that depends on specific agent behavior must prefix the requirement with 'Use the <agent-name> agent to...'"

**Gap**: No empirical data on Opus 4.7's delegation behavior with our steward prompts. The L13 explicit naming pattern should mitigate, but hasn't been tested on 4.7.

**Recommendation**: P2 — (1) During the graduated Opus 4.7 rollout (§1.1), monitor factory-steward session logs for delegation patterns. (2) Compare 4.7 sessions vs 4.6 sessions: count of subagent tool calls, duration per delegated task. (3) If delegation drops >30%, add stronger delegation prompting to steward skill.

## 2. Cross-Pollination Opportunities

### 2.1 ADK Session Rewind → Our Optimizer Checkpoint Pattern (P1)

**Direction**: Google ADK → Our Pipeline

ADK's Session Rewind (undo agent actions, retry) addresses the same problem our autoresearch-optimizer faces: how to recover from a wrong optimization direction without losing session context. The concept translates to:

**Current state**: Optimizer runs 4 parallel branches (A/B/C/D), evaluates each, commits the best. If all 4 fail, it starts a fresh iteration from scratch.

**With rewind-style checkpointing**: Before each branch modification, snapshot the current state. If all branches fail, rewind to the snapshot and try a fundamentally different optimization strategy (e.g., shift from adding trigger terms to strengthening exclusion rules).

**Implementation path**: When we migrate to Agent SDK sessions (Phase 5), implement session checkpointing using the SDK's session management. Until then, our `eval/experiment_log.json` provides a partial substitute (records what was tried).

### 2.2 Anthropic OTEL Tracing → ADK Observability Gap (P2)

**Direction**: Anthropic → Google

Agent SDK v0.1.60's W3C OTEL tracing is a capability ADK doesn't yet offer natively (ADK relies on Google Cloud Trace integration, which is vendor-locked). If we build an ADK-based agent for Phase 5, we'd face an observability gap on the Google side. The cross-pollination: design our Phase 5 observability to be OTEL-native (not vendor-specific), so it works with both Agent SDK traces and any future ADK OTEL support.

### 2.3 Anthropic Task Budgets → ADK Resource Limits (P2)

**Direction**: Bidirectional

Anthropic's Task Budgets (advisory token ceiling across agentic loops) and ADK's BashTool resource limits (process-level CPU/memory constraints) address the same problem from different angles:
- Task Budgets: model-level cost control (token ceiling)
- ADK resource limits: infrastructure-level safety (process constraints)

**For our pipeline**: Adopt both patterns. Phase 4/5 steward agents should have Task Budgets (model-level) AND duration ceilings (infrastructure-level) AND memory monitors (per ROADMAP §5.4 watchdog).

### 2.4 Google AgentEngineSandboxCodeExecutor → Anthropic Managed Agents Containers (P3)

**Direction**: Convergence

ADK v1.31.0's `AgentEngineSandboxCodeExecutor` (Vertex AI managed sandbox) mirrors Anthropic's Managed Agents container execution model. Both vendors are converging on the same architecture: **agent code execution in managed, isolated sandboxes**. This validates our Phase 5 credential isolation design (`knowledge_base/agentic-ai/evaluations/credential-isolation-design.md`) — the industry direction is sandboxed execution with vault-based credential access, not env-var inheritance.

### 2.5 Anthropic Four-Pattern Multi-Agent Taxonomy → Phase 5 TCI Router (P1)

**Direction**: Anthropic → Our Pipeline

Anthropic now officially documents four multi-agent patterns: Teams, Subagents, Three-Agent Harness, Orchestrator-Worker. Our Phase 5 TCI router currently routes to two tracks (parallel Track A, flagship Track B). The taxonomy suggests expanding to four dispatch targets:

| TCI Score | Anthropic Pattern | Our Dispatch |
|-----------|------------------|-------------|
| ≤ 2 | Subagents | Single agent with inline delegation |
| 3–4 | Teams | Parallel Track A (independent specialists) |
| 5–6 | Three-Agent Harness | Sequential pipeline (plan→generate→evaluate) |
| ≥ 7 | Orchestrator-Worker | Track B flagship with synchronous cohort |

**Note**: Yesterday's analysis (§1.4) proposed three-topology dispatch. Today's data refines to four, matching the official taxonomy. However, implementing 4 dispatch targets before Phase 5 starts adds complexity without validated benefit — defer the 4-way split to Phase 5 design sprint, keep 2-track (A/B) as Phase 4 baseline.

## 3. Threat Assessment

### 3.1 Opus 4.7 Tokenizer — Silent Cost Inflation (MEDIUM)

**Threat**: The new tokenizer produces 1.0–1.35× more tokens for same content. At our current fleet volume (2.3h/day, ~low single-digit USD/day), a 35% worst-case increase is operationally tolerable. But it silently inflates:
- Eval suite token budgets (more tokens per test = higher per-test cost)
- Prompt cache hit rates (longer cached prompts = higher cache-miss probability after TTL)
- Compaction trigger thresholds (more tokens per turn = earlier compaction)
- Cost ceiling baselines (30-day rolling averages will drift upward after 4.7 adoption)

**Mitigation**: After 4.7 adoption, update cost ceiling baselines (`scripts/lib/cost_ceiling.sh`) with a 1.35× headroom multiplier for the first 30 days. Monitor actual token ratios and adjust. The prompt caching 1h TTL env var (`ENABLE_PROMPT_CACHING_1H` from v2.1.105) partially offsets cache-miss probability.
