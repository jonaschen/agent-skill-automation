## 1. Gap Analysis: Findings vs. Pipeline State

### 1.1 OWASP MCP Security Crisis — QUANTIFIED THREAT to Phases 2-4 (P0, REINFORCED)

**Finding**: OWASP MCP Top 10 published with hard metrics: 30+ CVEs in 60 days, 84.2% tool poisoning success rate with auto-approval, 34/100 average security score across 17 popular servers, zero servers declare tool permissions.

**Impact on our pipeline**:
- **April 5 ADOPT item** (MCP content validation in `mcp_config_validator.sh`) is **independently validated** by the OWASP framework. Our implementation covers MCP03 (tool poisoning) and partially MCP04 (supply chain). However, OWASP identifies 7 additional attack categories we haven't addressed:
  - **MCP01 (Token mismanagement)**: Our pipeline doesn't audit how generated Skills handle API tokens passed to MCP servers. A generated Skill could inadvertently expose tokens through MCP server logs.
  - **MCP05 (Command injection)**: Tool descriptions can embed shell commands that execute when the LLM formats tool arguments. Our static scanner catches injection *phrases* but not injection *payloads* disguised as legitimate parameters.
  - **MCP09 (Shadow servers)**: No mechanism to detect if an agent connects to an unauthorized MCP server at runtime (only validated at config time).
- **Three attack variants now documented**: (1) direct poisoning (hidden instructions exfiltrate files), (2) tool shadowing (malicious servers override trusted tool behavior), (3) rug pulls (servers alter definitions after initial approval). Our static validation catches (1) partially but not (2) or (3).
- **`mcp-scan` tool** (`uvx mcp-scan@latest`) provides hash-based tool pinning — this catches rug pulls (variant 3) that our static scanner cannot.

**Gap**: Our April 5 MCP content validation addresses ~30% of the OWASP Top 10 attack surface. Missing: token hygiene (MCP01), runtime shadow server detection (MCP09), and rug pull detection (variant 3).

**Recommendation**: P0 — Integrate `mcp-scan` hash-based tool pinning into CI/CD gate as a complement to our static content validation. This is a low-effort, high-impact addition that addresses the rug pull vector. Add MCP01 (token hygiene) audit as a P1 follow-up.

### 1.2 Three-Agent Harness Validates Our Architecture — OPPORTUNITY to Adopt Sprint Contract Pattern (P2)

**Finding**: Anthropic's Planner/Generator/Evaluator harness blog with concrete metrics ($124-200/complex output, 3-6h execution time) is going viral — Swarms framework issue #1500, "harness engineering" emerging as a named discipline.

**Impact on our pipeline**:
- **Architecture validation**: Our `meta-agent-factory` (Generator) / `skill-quality-validator` (Evaluator) separation is independently validated by Anthropic's engineering team. The key insight — "self-evaluation bias is the critical failure mode: agents tend to confidently praise their own work even when quality is obviously mediocre" — is exactly why we separated generation from validation in Phase 1.
- **Sprint contract pattern**: Before each sprint, the Generator negotiates a "sprint contract" with the Evaluator defining success criteria. Our pipeline lacks this — the factory generates blindly, and the validator evaluates against fixed criteria. A contract pattern would:
  - Allow the factory to declare what it *intends* to produce (expected trigger patterns, permission model)
  - Allow the validator to pre-commit grading criteria before seeing the output
  - Reduce false failures from mismatched expectations
- **Context reset insight**: Sonnet 4.5 exhibited "context anxiety" near limits; Opus 4.6 sustained continuous sessions. Our steward agents run on Opus 4.6 — context resets may not be needed. But the structured handoff artifacts pattern (files, not conversations) is worth adopting for our optimizer's iteration state.
- **Cost benchmark**: $124-200 for complex multi-agent outputs. Our pipeline cost per Skill generation+validation+optimization is likely in the $5-15 range (much simpler task) — confirms our pipeline is cost-competitive.

**Gap**: No pre-evaluation contract between factory and validator. The validator evaluates against fixed criteria that may not match the factory's intent for a specific Skill.

**Recommendation**: P2 — For Phase 4 closed-loop robustness, implement a lightweight "generation contract" where `meta-agent-factory` outputs a manifest alongside the SKILL.md (expected trigger count, permission model, target domain), and `skill-quality-validator` validates against the manifest in addition to fixed criteria.

### 1.3 Agent SDK `terminal_reason` — DIRECTLY APPLICABLE to Phase 4 Closed Loop (P1)

**Finding**: TypeScript SDK v0.2.91 added `terminal_reason` field exposing why the query loop terminated (`completed`, `aborted_tools`, `max_turns`, `blocking_limit`).

**Impact on our pipeline**:
- **Phase 4 (Closed Loop)**: Our `closed_loop.sh` currently treats agent termination as binary (success/failure based on exit code). `terminal_reason` would enable differentiated retry logic:
  - `completed` → proceed to next stage
  - `max_turns` → retry with extended budget or escalate to human
  - `aborted_tools` → investigate tool availability, don't retry blindly
  - `blocking_limit` → resource constraint, back off and retry later
- **Routing regression diagnosis**: The active routing regression (T=0.658) causes positive prompts to "not-trigger" — understanding whether this is a `max_turns` or `completed` termination could help diagnose whether the model is running out of reasoning budget or genuinely misrouting.

**Gap**: Our pipeline doesn't consume `terminal_reason`. When we move to SDK-based agent execution (currently CLI-based), this becomes actionable.

**Recommendation**: P1 — Add `terminal_reason` consumption to the Phase 4 closed-loop refactor spec (already planned as state machine, P2). The retry logic differentiation should be part of the state machine's OPTIMIZE->VALIDATE retry counter.

## 2. Cross-Pollination Opportunities

### 2.1 Harness Engineering's "Assumption Encoding" → Our Pipeline Design Principle (NEW — HIGH VALUE)

**What Anthropic's harness blog teaches**: "Every component in a harness encodes an assumption about what the model can't do — stress test those assumptions as models improve."

**What this means for us**: Each component in our pipeline encodes assumptions:
- `skill-quality-validator` assumes the factory can't self-evaluate (validated by Anthropic)
- `autoresearch-optimizer` assumes descriptions can't be optimized in a single pass (valid — stochastic routing requires iterative refinement)
- `changeling-router` assumes the model can't auto-identify the right expert role (partially valid — keyword matching handles easy cases, but semantic disambiguation is still needed)
- `agentic-cicd-gate` assumes the model might regress after changes (valid — the routing regression proves this)

**Applicability**: When Capybara/Mythos ships, we should **stress-test these assumptions**. A sufficiently capable model might:
- Self-evaluate accurately enough to merge factory+validator (test this, don't assume)
- Optimize descriptions in fewer iterations (reduce the 50-iteration cap)
- Auto-identify expert roles without a routing table (simplify changeling-router)

This is a design principle, not an action item: **our pipeline should be designed to simplify as models improve, not to accumulate complexity.**

### 2.3 Anthropic's Context Reset vs. Continuous Session → Our Steward Agent Design (NEW)

**What Anthropic found**: Sonnet 4.5 exhibited "context anxiety" near context limits — quality degraded. Opus 4.6 sustained continuous sessions without degradation. Their solution: structured handoff artifacts (files on disk) rather than in-context state for multi-session workflows.

**What we have**: Our steward agents run 30-90 minute sessions on Opus 4.6. They rely on automatic compaction when context gets large. The `get_context_usage()` API (Agent SDK v0.1.56) could enable proactive management.

**Applicability**: Our stewards already use file-based state (performance JSONs, logs, steering notes). But the optimizer accumulates iteration state in-context. Adopting the structured handoff pattern:
- After each optimizer iteration, write best-so-far state to a file
- On context pressure (via `get_context_usage()`), reset context and reload from file
- This prevents compaction from discarding the best iteration result

Priority: P2 — implement when we move stewards to SDK-based execution.

### 3.1 MCP Attack Surface Expansion — UPGRADED from P0 to CRITICAL AWARENESS

**Previous assessment** (April 5): P0 — MCP tool poisoning is the first demonstrated attack vector.
**Current assessment**: The OWASP Top 10 framework expands the threat model from "tool poisoning" to a **10-category attack surface**. Our April 5 mitigation (content scanning) addresses ~30% of this surface.

**New threat vectors from OWASP**:
- **Tool shadowing** (variant 2): A malicious MCP server declares a tool with the same name as a trusted tool. The LLM preferentially uses the malicious version. Our allowlist approach doesn't prevent this because both servers are "allowed" — the conflict is at the tool name level.
- **Rug pulls** (variant 3): MCP servers change tool definitions after initial approval. Our static validation is point-in-time. Hash-based pinning (via `mcp-scan`) is the mitigation.
- **MCP09 Shadow servers**: Agents connecting to unauthorized MCP servers at runtime. Our validation is config-time only.

**Recommendation**: This is the highest-priority fix for the pipeline. The stabilization window is the right time to tackle it.

## 4. Strategic Observations

### 4.1 The "Harness Simplification Principle" Is Our Long-Term Design Compass

Anthropic's key insight — harness components encode model limitation assumptions that should be stress-tested and simplified as models improve — provides a strategic framework for our pipeline evolution:

**Current state** (Opus 4.6):
- Factory needs external validation → separate validator (correct)
- Descriptions need iterative optimization → optimizer with 50-iteration cap (correct)
- Routing needs explicit tables → changeling-router with keyword + semantic (correct)
- Deployment needs automated gating → CI/CD gate (correct, model-independent)

**Projected state** (Capybara/Mythos or Opus 5.x):
- Factory may self-evaluate accurately → test merging factory+validator into one agent
- Descriptions may need fewer iterations → reduce cap, potentially to single-pass
- Routing may work without explicit tables → test zero-shot role selection
- Deployment gating → always needed (trust-but-verify principle, model-independent)

**Design implication**: Build Phase 5-7 components with explicit assumption annotations. When a new model ships, test each assumption. If the model handles it natively, simplify the pipeline. This prevents the "complexity ratchet" where components accumulate but never get removed.
