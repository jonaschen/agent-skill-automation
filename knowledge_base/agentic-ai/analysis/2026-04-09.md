### 1.1 Claude Managed Agents — ARCHITECTURE EVALUATION Required (P1 — Strategic)

**Finding**: Anthropic launched Claude Managed Agents (public beta, April 8) — a fully managed agent runtime with:
- Secure cloud containers with pre-installed packages
- Built-in tools: Bash, file ops (read/write/edit/glob/grep), web search/fetch, MCP servers
- SSE streaming for real-time event consumption
- Stateful sessions with persistent file systems
- Rate limits: 60 creates/min, 600 reads/min per org
- Research preview features: multiagent, outcomes, memory

**Impact on our pipeline — three-layer assessment**:

**Layer 1 — Phase 4 (Immediate)**: Our 8 daily agents run via local cron → `claude` CLI. Managed Agents offers an alternative: instead of cron jobs on a local machine, agents could run as managed sessions triggered by external events (webhooks, schedules). However:
- Container startup latency is unknown — our cron agents need to start within seconds
- 60 creates/min is sufficient for our 8-agent fleet, but session persistence model unclear
- Our agents rely on local filesystem access (reading/writing to project repos) — Managed Agents containers are isolated
- **Verdict**: Not a viable Phase 4 replacement. Our local cron approach provides filesystem access, zero container overhead, and proven reliability. Managed Agents is designed for a different use case (cloud-hosted agent tasks, not local repository operations).

**Layer 2 — Phase 5 (Strategic)**: Managed Agents' multiagent research preview is directly relevant to Phase 5 topology. If Anthropic provides managed multi-agent orchestration as a cloud service:
- It could replace our custom A2A message bus (task 5.3) for cloud-hosted agent teams
- The `scrum-team-orchestrator` (task 5.3) may have a managed alternative
- But: we'd lose control over routing topology (TCI-based routing is our differentiator)
- **Verdict**: Monitor multiagent preview documentation. Our Phase 5 should be designed as a **hybrid** — local orchestration for development pipeline tasks, with Managed Agents as an optional cloud deployment target for customer-facing agent teams.

**Layer 3 — Phase 7 (Long-term)**: Managed Agents is a potential deployment target for AaaS Skills:
- Customers could run Skills in managed containers without infrastructure
- The branding rules ("Claude Agent" allowed, "Claude Code" not) enable white-label deployment
- The `ant` CLI with YAML resource versioning could serve as a declarative deployment interface
- **Verdict**: Record as Phase 7 deployment option alongside Conway and standalone SKILL.md distribution.

**Gap**: No evaluation framework for comparing local cron execution vs. Managed Agents hosting. No cost model for Managed Agents usage at our agent fleet scale.

**Recommendation**: P1 — Two actions:
1. **Phase 5 design note**: Record "Managed Agents multiagent preview" as an alternative to custom A2A bus. When multiagent documentation is released, perform a time-boxed 2-hour compatibility assessment: can our 6-message-type bus map to Managed Agents' multiagent primitives?
2. **Phase 7 design note**: Add Managed Agents as a third deployment target alongside Conway `.cnw.zip` and standalone SKILL.md. The three targets serve different markets: developer (SKILL.md), enterprise (Managed Agents), power-user (Conway).

### 1.2 Effort Level Default Change — COST IMPACT ASSESSMENT (P1 — Operational)

**Finding**: Claude Code v2.1.94 raised the default effort level from `medium` to `high` for API-key/Bedrock/Vertex/Foundry/Team/Enterprise users (non-free tier).

**Impact on our pipeline**:
- **Direct cost increase**: Our 8 daily agents all run via API keys. Every agent session now defaults to `high` effort instead of `medium`. Based on Anthropic's documentation, `high` effort produces longer, more detailed reasoning — meaning more output tokens per request.
- **Quality improvement**: Higher effort should improve agent output quality, particularly for complex tasks (factory-steward architectural decisions, researcher synthesis, reviewer analysis). This may reduce retry/rework cycles.
- **Estimated impact**: Without explicit benchmarks, higher effort typically increases output tokens by 30-80%. For our fleet running ~8 sessions/day averaging ~30-60 min each, this could increase daily token costs by 30-80%.
- **Mitigation options**:
  - Set `CLAUDE_CODE_EFFORT=medium` in daily scripts to restore previous behavior
  - Set `CLAUDE_CODE_EFFORT=high` only for agents that benefit from deeper reasoning (researcher, factory-steward, reviewer) and `medium` for routine stewards
  - Accept the increase and monitor via existing `cost_ceiling.sh` — our 5x rolling-average ceiling should absorb gradual increases

**Gap**: No per-agent effort level configuration. All agents inherit the global default.

**Recommendation**: P1 — Two immediate actions:
1. **Monitor first**: Let the cost ceiling run for 3-5 days at the new default. The 5x rolling-average ceiling should absorb the increase without triggering alerts, and we get free quality improvement.
2. **If costs spike**: Add `export CLAUDE_CODE_EFFORT=medium` to steward scripts that don't benefit from deeper reasoning (android-sw-steward, arm-mrs-steward, bsp-knowledge-steward) while keeping `high` for factory-steward, researcher, and reviewer.

## 2. Cross-Pollination Opportunities

### 2.1 Context-Aware Persistent Policy Approvals → Our Permission System (HIGH VALUE — NEW)

**What Google built**: Gemini CLI v0.38.0-preview.0 introduces context-aware persistent policy approvals — agents remember previously approved actions based on context, reducing repeated approval prompts.

**What we have**: Static permission rules in `check-permissions.sh` (binary allow/deny per agent class) plus `post-tool-use.sh` (monitors tool calls with initiator-type enforcement). Permission decisions are stateless — each tool call is evaluated independently.

**Convergence opportunity**: Add a **permission cache** to `post-tool-use.sh`:
- When a human approves a tool call in interactive mode, cache the (tool, parameters_pattern, context) tuple
- Subsequent matching tool calls auto-approve without re-prompting
- Cache expires per session (not persistent across sessions — safety boundary)
- Already partially implemented: `CLAUDE_INITIATOR_TYPE=cron-automated` auto-restricts in cron context

**Why this matters**: As Phase 5 introduces multi-agent teams, repeated permission prompts for the same tool patterns will create friction in the human-in-the-loop flow. Context-aware approvals would make HITL practical at scale without compromising safety.

**Priority**: P2 — Phase 5 design note. Not needed for Phase 4 (cron agents run unattended).

## 4. Strategic Observations

With Managed Agents' launch, the agentic AI landscape now has two distinct deployment models:

1. **Local-first** (CLI-native): Claude Code, Gemini CLI → agents run on developer machines, access local filesystems, integrated into development workflows
2. **Cloud-hosted** (managed): Managed Agents, Vertex AI Agent Builder → agents run in managed containers, accessed via API, integrated into production services

Our pipeline is firmly in camp #1 (local-first). This is correct for Phase 4 — our agents operate on local repos and need filesystem access. For Phase 7, we should support both deployment models: SKILL.md for local consumption, Managed Agents/Agent Engine for cloud deployment.

**Key insight**: The SKILL.md format is the portability layer. It works in Claude Code (local), Managed Agents (cloud), and potentially Gemini CLI (cross-platform). This is our strategic moat — the agent definition is runtime-agnostic.
