## 1. Gap Analysis: Findings vs. Pipeline State

### 1.1 Managed Agents Brain-Hands Architecture — PHASE 5 DESIGN REFERENCE (P1 — Architectural)

**Finding**: Anthropic published their Managed Agents architecture: stateless harness (brain) invokes sandboxes (hands) as tools via `execute(name, input) → string`. Three-component virtualization: Session (append-only log), Harness (decision loop), Sandbox (execution). Key properties:
- 60% median TTFT reduction from lazy container provisioning
- >90% P95 TTFT reduction
- Credentials never reach sandboxes — vault-based MCP proxies or resource-bundled auth
- Multi-brain, multi-hand scaling — harnesses and sandboxes scale independently
- Stateless harnesses resumable via `wake(sessionId)` + `getSession(id)`
- Context managed via durable Session log with `getEvents()` retrieval interface

**Impact on our pipeline**:

| Our Current Architecture | Managed Agents Pattern | Gap |
|-------------------------|----------------------|-----|
| Monolithic agent sessions (cron → long-running Claude) | Stateless harness + ephemeral sandbox | No brain/hands separation |
| State lives in conversation context | Append-only Session log with flexible retrieval | State implicitly coupled to context window |
| Credentials via environment variables | Vault-based MCP proxies | Weaker isolation |
| Single agent per task | Multi-brain, multi-hand (interchangeable tools) | No execution multiplexing |
| Fixed topology (parallel OR sequential) | Harness connects to any hand as needed | Static routing |

**Phase 5 translation**: The brain-hands decoupling maps directly to our topology-aware-router:
- **Brain** = topology-aware-router (decides what to execute, routes work)
- **Hands** = specialized agents executing in isolation (factory, validator, optimizer)
- **Session** = task state stored externally (not in any agent's context window)

The `execute(name, input) → string` interface is essentially what our agents already do when calling subagents — but Managed Agents makes the harness completely stateless and resumable. If our Phase 5 router can be rebooted mid-task and resume from a session log, it gains crash-safety for free without needing Vercel WDK-style deterministic replay.

**Key insight — lazy provisioning for steward agents**: Our nightly stewards currently initialize full Claude sessions upfront (cold start + context injection + MCP initialization) regardless of how much work is available. A lazy provisioning approach would: (1) lightweight check for pending work, (2) only spin up full session if work exists, (3) skip entirely on empty queues. The 60% TTFT improvement suggests significant gains are possible.

**Gap**: Phase 5 currently has no external session state design. The topology-aware-router stores all state in-conversation.

**Recommendation**: P1 — (1) Add "external session state" as a Phase 5.3 requirement — task state must survive router restart. (2) Evaluate `wake(sessionId)` pattern for steward resume-after-crash. (3) Consider lazy provisioning for nightly agents — pre-check work queue before full session spin-up.

Spec version remains 2025-11-25; next release tentatively June 2026. If Triggers & Events lands, MCP servers could emit events that activate agents without polling/scheduling.

**Impact on our pipeline**:

Our entire daily agent fleet (13+ runs) uses cron scheduling. Event-driven activation would enable:

| Cron Model (current) | Event-Driven Model (potential) |
|---------------------|-------------------------------|
| Fixed schedule regardless of work | Activate only when relevant events occur |
| May run with nothing to do | Always has work when activated |
| Missed events between runs | Real-time response to events |
| Simple, well-understood | Complex event routing and dedup needed |
| Cost: fixed daily compute | Cost: proportional to actual events |

**Concrete scenarios where event-driven wins**:
- `arm-mrs-steward`: activated when ARM releases a new spec update (instead of nightly check)
- `bsp-knowledge-steward`: activated when a new Linux kernel release or ARM TRM is published
- `agentic-ai-researcher`: activated when Anthropic/Google ships a new release (instead of daily sweep)
- `factory-steward`: activated when a new research ADOPT item is written or when performance drops below threshold

**However, event-driven doesn't replace all cron use cases**:
- Phase-driven autonomous work (stewards advancing ROADMAP tasks) still needs scheduled time
- Research sweeps that scan for "what's new" need periodic execution, not just reaction
- Quality review (project-reviewer) should run on a schedule to catch gradual drift

**Gap**: No event source infrastructure. No webhook receivers. No event-driven agent activation mechanism.

**Recommendation**: P2 — (1) Track MCP Triggers & Events spec drafts when they appear (likely pre-June). (2) Don't build custom event infrastructure yet — wait for spec. (3) Design Phase 5.4 to accommodate hybrid scheduling: cron for periodic work + event triggers for reactive work. (4) Identify which of our 13 daily runs would benefit most from event-driven activation (researcher, stewards with external dependency tracking).

### 1.3 Credential Isolation Gap — ENVIRONMENT VARIABLES VS. VAULT-BASED MCP PROXY (P1 — Security)

**Finding**: Managed Agents uses two credential isolation patterns:
1. **Resource-bundled auth**: git tokens initialize repos during sandbox setup, then sandbox operates without direct credential access (push/pull works via pre-configured git config)
2. **Vault-based credentials**: OAuth tokens stored in secure vaults, accessed only via MCP proxy servers — credentials never enter the sandbox environment

Our pipeline: all secrets via environment variables (`ANTHROPIC_API_KEY`, `GITHUB_TOKEN`, etc.) directly in agent process environments. If an agent's execution is compromised (command injection, tool poisoning), all env vars are accessible.

**Severity assessment**:
- Our agents run on a single machine with Jonas's credentials
- A command injection in any daily script exposes the full environment
- CVE-2026-35020 (TERMINAL env var injection) was mitigated with `unset TERMINAL`, but the pattern shows env vars are attack surface
- The `permissions.deny` override fix in v2.1.101 shows even permission systems can have bypasses

**Mitigation timeline**:
- **Immediate** (this sprint): No action needed — our threat model is single-user, single-machine. The attack would require compromising Claude Code itself, which is already trusted.
- **Phase 5** (multi-agent): If agents spawn subagents that execute untrusted code (e.g., generated skills being tested), credential isolation becomes critical.
- **Phase 7** (AaaS): Multi-tenant execution absolutely requires vault-based isolation.

**Gap**: No credential isolation mechanism designed for Phase 5/7.

**Recommendation**: P1 for Phase 5 design — (1) Add credential isolation to Phase 5 security requirements. (2) When Phase 5 executes generated skills in test sandboxes, those sandboxes must not inherit parent credentials. (3) Reference Managed Agents' MCP proxy pattern as the target architecture.

## 2. Cross-Pollination Opportunities

### 2.1 Managed Agents `execute(name, input) → string` ↔ Our Subagent Tool-Call Pattern

**Observation**: Managed Agents formalizes what our pipeline already does informally — parent agents invoke specialized agents and get string results back. But Managed Agents adds three properties we lack:
1. **Stateless harness**: Router doesn't hold state; can crash and resume
2. **Lazy provisioning**: Execution environments initialized on-demand
3. **Failure isolation**: Failed sandbox ≠ failed harness; harness treats failure as tool-call error

**Action**: Phase 5 topology-aware-router should be designed as a stateless harness that:
- Reads task state from external log on startup
- Invokes agents (hands) via tool calls
- Treats agent failures as retryable tool errors (not fatal)
- Can be `wake()`-ed with any pending task ID

### 2.2 ADK Lazy Scan Dedup ↔ Managed Agents Session Replay ↔ Our State Machine Skip

**Observation** (builds on yesterday's workflow convergence note): Three independent implementations of the same idea — "resume execution without re-running completed steps":
- **ADK v2.0 alpha.3**: Lazy scan dedup on graph workflow resume
- **Managed Agents**: `getSession(id)` + `getEvents()` — pick up from previous position
- **Our pipeline**: `closed_loop.sh` state machine conditional skip (≥0.95)

All three converge on: durable execution state + skip-completed-steps on resume. Our state machine is the simplest (file-based JSON), Managed Agents is the most sophisticated (append-only log with flexible retrieval). Phase 5 should adopt the append-only event log pattern for task orchestration.

### 2.3 ADK BashTool Metacharacter Blocking ↔ Our `cmd_chain_monitor.sh` Detection

**Observation**: Both ecosystems are hardening shell execution security, but at different layers:
- **ADK v1.29.0**: Pre-execution blocking (metacharacters rejected before shell runs)
- **Our pipeline**: Post-execution detection + advisory logging (metacharacters detected but not blocked)

Yesterday's analysis recommended evaluating pre-execution blocking. Today's update: our `cmd_chain_monitor.sh` now has `METACHAR_MODE` env var supporting future "block" mode. The 30-day baseline collection (started Apr 11) will provide data for calibrating a blocking allowlist. Timeline: block mode can be enabled ~May 11 (after baseline + Phase 4 deadline).

### 2.4 Managed Agents Context Management ↔ Our Steward Long-Running Sessions

**Observation**: Managed Agents stores context in a durable Session log *outside* the LLM context window, with `getEvents()` for selective retrieval. Our stewards dump everything into the conversation context, relying on Claude Code's compaction when limits approach.

**Implication for steward efficiency**: If steward agents wrote structured events to an external log (file-based) and only loaded relevant recent events into context at session start, they could:
- Start faster (less context to load)
- Focus on current work (not re-reading prior conversation)
- Resume after crash without losing state

This is essentially what our performance JSON files already do for metrics — extend the pattern to task state.
