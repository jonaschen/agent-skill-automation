## 1. Gap Analysis: Findings vs. Pipeline State

### 1.1 CVE-2026-35020 — IMMEDIATE EXPOSURE ASSESSMENT Required (P0)

**Finding**: CVSS 8.4 HIGH — OS command injection via `TERMINAL` environment variable in Claude Code CLI and Agent SDK. Local attack vector. `shell=true` parses the env var unsafely.

**Impact on our pipeline**:
- **Direct exposure**: Our 8 daily agent runs all invoke Claude Code via cron scripts (`scripts/daily_*.sh`). Each runs in the local user context where `TERMINAL` is typically set by the shell environment.
- **Attack surface**: If any upstream process (CI runner, container orchestrator, or compromised dependency) modifies `TERMINAL` before our cron jobs execute, arbitrary commands run with the agent's user privileges.
- **Mitigation status**:
  - Our scripts don't explicitly sanitize `TERMINAL` before invoking `claude`
  - The `cmd_chain_monitor.sh` monitors command chains but wouldn't detect injected commands from env var parsing — they'd appear as legitimate CLI startup behavior
  - The `cost_ceiling.sh` would eventually detect anomalous duration but not prevent execution

**Gap**: No environment variable sanitization in our cron scripts. No verification that we're running the patched Claude Code version.

**Recommendation**: P0 — Three immediate actions:
1. **Verify Claude Code version** is up-to-date across the execution environment
2. **Add `unset TERMINAL` or `export TERMINAL=xterm`** (safe default) to the preamble of all 8 daily scripts as a defense-in-depth measure
3. **Audit the cron environment** — verify no upstream process can inject env vars into our cron context

## 2. Cross-Pollination Opportunities

### 2.1 AWS IAM Context Keys → Our Pipeline Permission Model (HIGH VALUE — NEW)

**What AWS built**: Two IAM context keys (`aws:ViaAWSMCPService`, `aws:CalledViaAWSMCP`) that enable IAM policies to differentiate agent actions from human actions. Example: deny `ec2:TerminateInstances` when called via MCP, allow when called directly by a human.

**What we have**: `eval/check-permissions.sh` enforces binary agent-level permissions (review agents can't Write, execution agents can't Task). The `post-tool-use.sh` hook monitors tool calls but doesn't distinguish the initiator chain.

**Convergence opportunity**: Port the AWS pattern to our pipeline:
- Add `initiator_type` (human/automated/delegated) to tool permission context
- Allow `post-tool-use.sh` to enforce different policies per initiator:
  - `human-interactive`: standard permissions (current behavior)
  - `cron-automated`: restricted permissions (e.g., block destructive git ops)
  - `orchestrator-delegated`: intermediate permissions (subagent-specific)

**Why this matters**: As our pipeline becomes more autonomous (Phase 4 closed loop), distinguishing human-initiated from automated actions becomes a safety requirement, not a feature. The CVE-2026-35020 exposure demonstrates that automated environments have different threat surfaces.

Priority: P1 — Implement during Phase 4 security hardening. AWS provides the proven pattern.
