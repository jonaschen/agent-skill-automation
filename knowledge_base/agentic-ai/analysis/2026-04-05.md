## 1. Gap Analysis: Findings vs. Pipeline State

### 1.1 MCP Tool Poisoning — ACTIVE SECURITY THREAT to Phases 2–4 (P0)

**Finding**: Invariant Labs published an easy-to-reproduce tool poisoning attack against MCP — malicious tool descriptions can manipulate LLM behavior, potentially causing data exfiltration or unintended actions.

**Impact on our pipeline**:
- **Phase 4 (Closed Loop)**: Our `meta-agent-factory` generates `.mcp.json` configs. If a generated Skill connects to an untrusted MCP server with poisoned tool descriptions, the poisoned descriptions become part of the agent's prompt — our agents would execute under adversarial influence.
- **Phase 2 (CI/CD Gate)**: Our `mcp_config_validator.sh` (implemented 2026-04-04) validates JSON structure, required fields, deprecated auth patterns, and placeholder env vars — but does NOT validate tool description content. A valid-looking MCP config pointing to a malicious server passes our gate.
- **All autonomous agents**: The six nightly steward agents run with significant trust. If any MCP integration is added to their workflows, tool poisoning becomes a lateral movement vector.

**Gap**: We validate MCP *config format* but not MCP *tool content*. The poisoning attack operates at the semantic level — the tool description itself is the payload. Our current CI/CD gate cannot catch this.

**Recommendation**: **P0** — Add a tool description validation step to the CI/CD gate pipeline. For any MCP server referenced in a Skill's `.mcp.json`, fetch tool descriptions at validation time and:
1. Scan for instruction injection patterns (e.g., "ignore previous instructions", "you must", "do not tell the user")
2. Flag descriptions exceeding a character threshold (legitimate tool descriptions are typically <500 chars)
3. Reject descriptions containing file path references, URL patterns, or credential keywords
4. Log a warning for any MCP server not in a curated allowlist

This extends the `mcp_config_validator.sh` work from yesterday with content-level validation.

### 1.3 Agent Payment Protocol War — INFORMS Phase 7 Architecture (P2)

**Finding**: Four competing protocols launched within 90 days: AP2 (Google, 60+ orgs, mandate-based), Visa TAP (100+ partners, Agent Identity Certificates), x402 (HTTP 402 micropayments, stablecoin settlement), PayPal Agent Ready (protocol-agnostic, 400M accounts). $11.79B projected market.

**Impact on our pipeline**:
- **Phase 7 (AaaS)**: Our ROADMAP specifies Stripe-based outcome billing (task 7.1). The payment protocol war reveals a much more complex landscape:
  - *For B2B/enterprise*: Visa TAP's Agent Identity Certificate system provides cryptographic proof of agent identity — essential for enterprise transactions
  - *For micropayments/per-invocation*: x402 embeds payments at the HTTP layer — agents pay for API calls without billing agreements
  - *For consumer*: PayPal Agent Ready's protocol-agnostic approach leverages 400M existing accounts
  - *For agent-to-agent commerce*: AP2's mandate system (Intent Mandate → Cart Mandate) handles the "user authorized this agent to buy" trust problem
- **Phase 7 billing model needs redesign**: Our current plan (OpenTelemetry spans → Stripe invoice) is a traditional SaaS billing model. The emerging agent economy has three distinct billing patterns:
  1. **Subscription/invoice** (Stripe) — for platform access
  2. **Per-invocation micropayment** (x402) — for MCP server usage
  3. **Commerce mandate** (AP2) — for agent-executed purchases

**Gap**: Phase 7 was designed before agent payment protocols existed. The billing architecture section needs to account for all three billing patterns, not just Stripe. Our competitive advantage in AaaS depends on supporting the payment model that wins.

**Recommendation**: Add an evaluation task to Phase 7: "Survey agent payment protocols (AP2, TAP, x402, PayPal) and design a billing adapter layer that supports multiple settlement rails." Priority P2 — no immediate action needed, but the architecture decision must be informed by how the protocol war settles.

## 4. Strategic Observations

### 4.1 The Three-Layer Agent Protocol Stack Is Solidifying

The agent ecosystem now has a clear three-layer protocol architecture:

```
Layer 3: Payments  — AP2 (mandates) + Visa TAP (identity) + x402 (micropayments) + PayPal (unification)
Layer 2: Tools     — MCP (model-to-tool, 6,400+ servers, Linux Foundation)
Layer 1: Agents    — A2A (agent-to-agent, v1.0.0, Linux Foundation)
```

**Our position**: We operate at Layer 2 (MCP integration in Skills). We have no Layer 1 integration (our inter-agent communication is via files and environment variables). We have no Layer 3 integration (Phase 7 planned Stripe-only).

**Strategic implication**: For Phase 5, we should evaluate adopting A2A v1.0.0 for our `scrum-team-orchestrator`'s message bus instead of the currently planned custom 6-message-type schema. A2A provides:
- Signed Agent Cards (identity verification)
- gRPC transport (high-performance)
- Task lifecycle management (tasks/list, filtering, pagination)
- Linux Foundation governance (stability guarantee)

The cost of a custom bus is maintenance burden; the cost of A2A adoption is additional complexity. But A2A gives us Layer 1 interoperability "for free" — our agents could be discovered and orchestrated by external A2A-compatible systems.

### 4.2 Edge AI Has Crossed the Production Threshold

Three converging signals:
1. **Gemma 4 E2B**: 86.4% tool use, <1.5GB, native function calling, Apache 2.0
2. **AICore Developer Preview**: Production API path for on-device Android with hardware acceleration (Google, MediaTek, Qualcomm)
3. **140M+ devices**: Gemini Nano 4 on Android, forward-compatible with Gemma 4 code

Our Phase 6 was designed when FunctionGemma 270M (85% accuracy *after fine-tuning*) was state-of-the-art. Gemma 4 E2B achieves 86.4% *without fine-tuning* at a larger but still edge-viable size. The fine-tuning step we planned is now optional, not mandatory.

**Updated Phase 6 recommendation**: Replace FunctionGemma with Gemma 4 E2B/E4B as the edge model target. Update `eval/edge_readiness.py` criteria to reflect Gemma 4's capabilities. This is a simplification — fewer pipeline steps, better accuracy, production-ready hardware acceleration. Already noted in ROADMAP Phase 6.4 (evaluation confirmed 2026-04-04).

### 4.3 Persistent Agents Are the Next Platform Shift

Conway (Anthropic), Gemini CLI Memory Manager (Google), and the broader shift from session-based to event-driven agent architectures signal a platform transition. Our cron-based steward model is aligned with batch processing patterns but misses:
- Real-time event response (a GitHub issue filed at 2pm won't be triaged until the next nightly run)
- Continuous monitoring (our agents check health on a schedule, not when anomalies occur)
- Always-on availability (our agents are dormant between cron runs)

This isn't a gap that needs immediate fixing — batch processing is correct for nightly analysis/improvement cycles. But for Phase 7 AaaS customers, always-on agents will be an expected capability, not a nice-to-have.
