**Finding**: CVE-2026-40933 (disclosed April 16, 2026) — authenticated RCE via MCP STDIO adapters in Flowise AI. The vulnerability bypasses input sanitization (validateCommandInjection, validateArgsForLocalFileAccess) because allowed commands like `npx` can be combined with code execution arguments (`-c touch /tmp/pwn`). CVSS critical (AV:N/AC:L/PR:L/UI:N/S:C/C:H/I:H/A:H). Fixed in Flowise 3.1.0. **12,000+ instances actively exposed** per The Hacker News. Active exploitation confirmed.

**Impact on our pipeline**: None directly — we don't use Flowise. However, this extends the STDIO vulnerability family:

- The attack pattern (bypassing command allowlists via argument injection) is relevant to our `cmd_chain_monitor.sh` binary allowlist design

**Gap**: Our `cmd_chain_monitor.sh` uses a 60+ binary allowlist. The Flowise bypass shows that allowlisting the *binary* (e.g., `npx`) isn't sufficient if the *arguments* enable code execution. Our allowlist currently checks binary names only, not argument patterns. This is an acceptable Phase 4 risk (our monitor is detect-only, not blocking), but Phase 5 `PreToolUse` blocking mode should include argument pattern validation.

**Assessment**: No immediate action required. Add to MCP vulnerability tracking. Phase 5.5 defensive architecture should incorporate argument-pattern validation alongside binary allowlisting. Reference: Flowise bypass of `validateCommandInjection`.

## 2. Cross-Pollination Opportunities

### 2.1 Flowise Bypass Pattern → cmd_chain_monitor.sh Enhancement

**Flowise context**: CVE-2026-40933 bypassed command allowlists by using allowed binaries (`npx`) with code-execution arguments (`-c`). The allowlist checked the binary name but not the argument semantics.

**Our context**: `cmd_chain_monitor.sh` uses a 60+ binary allowlist that checks command names. In detect-only mode (current), this is informational. When Phase 5 enables blocking mode (`METACHAR_MODE=block`), argument-blind allowlisting would have the same vulnerability as Flowise.

**Cross-pollination**: Phase 5.5 `PreToolUse` blocking mode should add argument pattern flags to the binary allowlist. Specifically, flag these high-risk argument patterns on otherwise-safe binaries:
- `npx -c`, `node -e`, `python3 -c`, `ruby -e` (inline code execution)
- `bash -c`, `sh -c` (already caught by chain detection, but worth explicit flagging)
- `curl | bash`, `wget -O- | sh` (download-and-execute chains)

This is a Phase 5 enhancement, not Phase 4.
