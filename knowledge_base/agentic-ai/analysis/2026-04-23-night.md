# Agentic AI Analysis — 2026-04-23 (Night — Full)

## Findings

### Finding 1: MCP Tool Hooks Create a New Automation Primitive (S1 — HIGH)

CC v2.1.118's `type: "mcp_tool"` hook type enables hooks to invoke MCP tools directly without going through the agent conversation loop. This is architecturally significant:

**Current state**: Our pipeline's hooks are shell commands (`type: "command"`). They execute scripts but can't interact with MCP servers directly. To trigger an MCP tool, you need an agent session.

**New capability**: A hook can now call an MCP server tool as a side-effect. This creates a pre/post-processing pipeline that operates outside agent context:
- Pre-commit hook → MCP tool validates code against external rules
- Post-tool hook → MCP tool logs tool invocations to external system
- Pre-response hook → MCP tool enriches context from external knowledge base

**S1 relevance**: This is a building block for the "capability diff" concept (S1 priority). A hook could:
1. Fire on `session_start` → invoke an MCP tool that checks for new platform releases
2. Compare release features against an assumption registry
3. Generate upgrade task proposals before the agent session begins

**Gap**: Our pipeline doesn't use MCP tool hooks yet. This is a Phase 4 opportunity, not Phase 5.

**Assessment**: MODERATE-HIGH impact. The primitive is powerful but our pipeline currently runs agents via `claude -p` scripts, not interactive sessions with hooks. To leverage MCP tool hooks, we'd need either (a) hook-capable session management in our cron scripts, or (b) a dedicated MCP server that our hooks can call. Option (b) is more feasible and aligns with Phase 5.6 (Remote MCP).

### Finding 2: SDK ServerToolUseBlock Fix Reveals Silent Data Loss (MODERATE)

SDK Py v0.1.65 fixed `server_tool_use` and `advisor_tool_result` content blocks being silently dropped. Messages carrying only server-side tool calls arrived as empty `AssistantMessage(content=[])`.

**Implication for our pipeline**: Our pipeline processes SDK-returned messages in the Agent SDK integration (factory, optimizer). If any agent used server-side tools (advisor, web_search), those tool call results were invisible. This is a **data integrity bug** that was silently degrading message fidelity.

**Pipeline risk**: LOW for current pipeline (our agents primarily use local tools via Claude Code, not server-side SDK tools). But any future SDK-based agent integration must upgrade to v0.1.65+ to avoid this.

**Assessment**: MODERATE. No immediate pipeline impact but a correctness-critical fix for any SDK integration path.

### Finding 3: Thinking Display Control Provides Opus 4.7 Observability (S1 — LOW-MODERATE)

SDK v0.1.65's `ThinkingConfig.display` field lets callers override Opus 4.7's default `"omitted"` thinking display. This is relevant to #49562 (token burn) and our Opus 4.7 migration decision.

**Current state**: Opus 4.7's adaptive thinking uses tokens that are invisible to callers when display is `"omitted"`. The token burn reported in #49562 is partly attributable to this invisible thinking.

**New capability**: Setting `display` to a non-omitted value (presumably `"summarized"` or similar) gives visibility into thinking token usage without exposing raw thinking content.

**S1 relevance**: This is an observability improvement, not a cost fix. For our shadow eval pipeline, it could help diagnose whether Opus 4.7's token burn is in thinking or in generation — informing the GO/NO-GO decision.

**Assessment**: LOW-MODERATE. Useful for diagnostics but doesn't change the fundamental token economics of Opus 4.7.

### Finding 6: Agent Identity (SPIFFE-Based) — FIRST PRODUCTION AGENT IDENTITY SYSTEM (S2/Phase 5 — HIGH)

Google's Agent Identity assigns every agent a unique cryptographic ID based on the SPIFFE (Secure Production Identity Framework for Everyone) standard. Strongly attested, tied to agent lifecycle, mapped directly to resource URI.

**Why this matters for our pipeline**:
- Our Phase 5 multi-agent design has no agent identity model. Agents are identified by file path and name, not cryptographic identity.
- For Phase 5 inter-agent security (especially when agents route to each other via the orchestrator), knowing "which agent is this?" with cryptographic certainty is a prerequisite for secure delegation.
- SPIFFE is an established CNCF standard — this isn't Google-proprietary, it's cloud-native-compatible.

**Assessment**: HIGH. Phase 5 should evaluate SPIFFE-based identity or equivalent. This finding should go to the factory as a Phase 5 design input.

### Finding 9: Mariner "Teach and Repeat" — First Vendor Implementation of Persistent Agent Learning (S1 — MODERATE)

Mariner's "Teach and Repeat" feature lets users teach workflows once, then Mariner repeats them autonomously using persistent cross-session memory. This is the first major vendor implementation of user-demonstration-based agent learning.

**Relevance to S1**: Our S1 priority asks "how can agents detect when a new platform capability makes their current approach suboptimal?" Mariner's approach is different — it learns from user demonstrations, not platform capabilities. But the persistent cross-session memory pattern is relevant: our agents could store "what worked" across sessions and automatically prefer successful strategies.

**Assessment**: MODERATE. Not directly actionable but informative for S1 architectural thinking. No ADOPT needed.
