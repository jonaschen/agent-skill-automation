### 1.2 Three-Agent Harness Canon — PHASE 4 CLOSED LOOP VALIDATED (P1 — Architecture)

**Finding**: Anthropic's Apr 4 engineering blog "Harness Design for Long-Running Apps" formalized a Three-Agent Harness:
1. **Planning Agent** — task decomposition, strategy, success criteria
2. **Generation Agent** — code/design/implementation
3. **Evaluation Agent** — independent grading (explicitly: "separation of evaluation from generation is a strong lever")

Concrete parameters from the blog:
- **Structured handoff artifacts** replace shared rolling context
- Explicitly contrasts with compaction: "can make models overly cautious about approaching token limits"
- Concrete evaluator: **Playwright MCP evaluator** scoring 4 axes (design quality, originality, craft, functionality)
- **5–15 refinement iterations per run**, sometimes **~4 hours of autonomous operation** (exactly our Phase 4 end-to-end target!)

**Impact on our pipeline**:

| Our Agent | Harness Role | Alignment |
|-----------|--------------|-----------|
| meta-agent-factory | Generation | ✅ Already generates complete SKILL.md deliverables |
| skill-quality-validator | Evaluation | ✅ Separate agent, independent eval, JSON report as handoff artifact |
| autoresearch-optimizer | Planning / Re-Planning | ✅ Analyzes failures, plans description refinements as strategy |
| closed_loop.sh | Harness orchestrator | ✅ State machine coordinates all three |

This is a remarkable alignment. Our pipeline was designed independently but converged on the canonical pattern. Three specific refinements to make the mapping explicit:

1. **Handoff artifact contracts** — Validator → Optimizer today passes a JSON report with `trigger_rate`, `ci_lower`, `ci_upper`, `security_score`, `recommendations`. Factory → Validator passes a SKILL.md path. These are implicit contracts. The blog's explicit framing suggests we should make them **versioned, schema-validated contracts** so that agent replacement (e.g., if we later swap validator for a Mythos-class evaluator) doesn't break the harness.

2. **4-axis scoring gap** — The blog's Playwright evaluator scores design quality, originality, craft, functionality. Our validator scores trigger rate + security + permissions. We have no equivalent to "originality" (novel solution vs. rote pattern) or "craft" (description precision, no redundant clauses). This could be a P2 enhancement once Phase 4 pilot stabilizes at 80% deployment rate.

3. **5–15 iteration convergence budget** — The blog reports 5–15 iterations as typical. Our optimizer budget is ≤ 50 iterations per ROADMAP Phase 3 acceptance criteria. Gap: we may be over-budgeting. Post-upgrade to Opus 4.7 (+6.8pp SWE-bench), convergence should be faster — consider dropping iteration ceiling to 20 as a forcing function for description quality.

**Gap**: Handoff artifact contracts are implicit. Evaluator axes are incomplete vs. canonical set. Iteration budget may be over-sized for Opus 4.7 era.

**Recommendation**: **P1** — Open a sprint-contract-manifest task (already scaffolded in ROADMAP §10 Immediate Next Actions #10) for factory → validator. Add explicit versioned JSON schema. Document the Three-Agent Harness mapping in ROADMAP Phase 4 section as an architectural win. Defer the 4-axis scoring enhancement to post-I/O (P2).

### 1.5 Programmatic Tool Calling GA — RESEARCHER + FACTORY TOKEN EFFICIENCY (P2 — Opportunity)

**Finding**: Programmatic Tool Calling is now GA on Opus 4.7 via `code_execution_20260120`. Confirmed compatibility: Opus 4.7, 4.6, 4.5; Sonnet 4.6, 4.5 (5 models). Available on Claude API + Microsoft Foundry; Bedrock/Vertex still require beta header. Key constraint: MCP connector tools cannot be called programmatically yet. Key economic benefit: **intermediate tool results do NOT count toward input/output tokens** — only the final code execution result + Claude's response count.

BrowseComp + DeepSearchQA benchmarks confirmed programmatic calling was the key unlock for multi-step web research agents — not just an efficiency win, but a capability gate.

**Impact on our pipeline**:

- **agentic-ai-researcher**: Our nightly researcher performs WebSearch → WebFetch chains for each tracked topic (14 topic files, ~3 searches × 3-5 fetches per topic = ~50+ tool calls per sweep). Programmatic calling could batch these into a single code-execution block per topic: ~10× token reduction on intermediate results while keeping final synthesis quality identical.
- **meta-agent-factory**: Today, generating SKILL.md + `scripts/` + `references/` + tests is a multi-round-trip operation. Programmatic calling could produce all four artifacts in one code execution. Risk: our Write-tool allowlist security model assumes discrete file writes; programmatic tool calling would bypass post-tool-use hooks. Needs security review.

**Gap**: Neither agent uses programmatic calling. Security implication on post-tool-use.sh hook coverage is unanalyzed.

**Recommendation**: **P2** — (1) Add programmatic-tool-calling evaluation as a Phase 4 post-stabilization task. (2) Pilot on researcher first (low security blast radius — read-only tools). (3) Before factory pilot, audit post-tool-use.sh hooks for whether programmatic-execution container writes trigger the hooks at all — if not, that's a security regression vector that must be mitigated (likely via explicit `code_execution_20260120` policy in settings.local.json).

## 2. Cross-Pollination Opportunities

### 2.1 Anthropic Three-Agent Harness ↔ Google ADK Multi-Agent Pattern (CONVERGENCE SIGNAL)

Anthropic's Three-Agent Harness (Planning/Generation/Evaluation) is architecturally close to ADK's declarative workflow composition (v1.30.0 supports multi-agent coordination via the Agent Registry). Both vendors are converging on **separation of concerns by role** and **structured inter-agent message passing**. Their transport differs (Anthropic uses handoff artifacts in conversation context; Google uses A2A messages), but the role decomposition is identical.

**Cross-pollination**: Our factory → validator → optimizer contracts should be designed as **transport-agnostic schemas** (JSON Schema) so they could carry over either an A2A Task payload (if we post-I/O adopt A2A per ROADMAP §5.3) or Anthropic-native handoff artifacts. The 6-message-type bus design (ROADMAP §5.3) already anticipates this generalization.

### 2.2 Pre-I/O Freeze Synchronization ↔ Anthropic Flat Pricing (UNUSUAL STABILITY WINDOW)

Both vendors are withholding commercial-surface changes right now, for different reasons:
- **Anthropic**: Opus 4.7 priced flat to avoid triggering a price war (strategic)
- **Google**: Entire agent stack frozen waiting for I/O relaunch (tactical)

Net result: **4-week planning window with unusually high stability**. Our ROADMAP can plan through mid-May without expecting pricing structure, API shape, or capability envelope to shift underneath. This is the right moment to execute infrastructure-heavy tasks (Opus 4.7 fleet upgrade, sprint contract manifest, three-topology dispatch design) that would be costly to redo if the ground shifted.

### 2.3 MCP Triggers & Events ↔ A2A Async Direction (PROTOCOL CONVERGENCE)

Both Anthropic's MCP (Triggers & Events WG) and Google's A2A (async message bus direction) are moving toward **event-driven agent communication**. If both ship webhook/callback primitives in 2026, the distinction between "MCP tool call" and "A2A agent message" blurs — they become two transport options for the same semantic event.

**Cross-pollination**: Our 6-message-type bus (ROADMAP §5.3) should be designed semantically (TASK_ASSIGNMENT, PARTIAL_OUTPUT, REVIEW_REQUEST, etc.) with transport pluggability. Today's implementation might use direct subagent calls; tomorrow's might use A2A; day-after-tomorrow's might use MCP Triggers. The 6-type taxonomy is transport-independent.

### 2.4 Anthropic `/less-permission-prompts` ↔ Gemini CLI Context-Aware Approvals

Yesterday's analysis noted Gemini CLI v0.38.0 context-aware approvals. Today's `/less-permission-prompts` is the Anthropic-side equivalent — both vendors are solving the same UX problem (permission-prompt interruption frequency) with transcript-analysis-based allowlist generation.

## 3. Threat Detection

### 3.1 Opus 4.7 Eval Regression Risk (P0 — Actionable)

**Threat**: Opus 4.7 is a step-change model. Our trigger-routing eval baseline (T=0.895 / V=0.900) was established on 4.6. An uncalibrated fleet upgrade could silently shift routing decisions — including on the very description we optimized through G8 Iter 1 and Iter 2.

**Historical precedent**: L7 (Lessons Learned in ROADMAP) — adding agents caused T to drop from 0.895 to 0.658 before vocabulary deconfliction + eval pattern fix. Model upgrades are a similar class of perturbation.

**Mitigation**: Shadow eval before rollout (§1.1 recommendation). Treat this as a controlled change, not a routine upgrade. The mitigation is cheap (~15 minutes of eval compute) relative to the blast radius (10 agents × 15 nightly runs × quality exposure).

### 3.2 I/O 2026 Breaking Changes — A2A v1.1 and ADK v2.0 (P1 — Hedged)

**Threat**: If I/O 2026 (32 days away) ships A2A v1.1 with wire-format changes AND Phase 5 had already committed to A2A v1.0 integration, we'd need to re-implement. The 36-day A2A stable window (longest ever) is a silent signal that v1.1 is under private development.

**Mitigation**: Already in place via ROADMAP §5 HOLD directive (no ADK/A2A integration until post-I/O + gate-removal criteria). Sweep 2026-04-17 continues to report zero pre-I/O leaks on A2A, so the mitigation remains well-calibrated. No additional action needed.

### 3.3 Mariner / Agent Builder Rebrand Breaking References (P2 — Documentation)

**Threat**: If I/O 2026 formally rebrands Mariner → Gemini Agent AND Agent Builder → AI Applications (already underway), our KB files (`project-mariner.md`, `vertex-ai-agents.md`) will reference deprecated product names. Search traffic, link rot, and sweep query terms may all need re-tuning.

**Mitigation**: Post-I/O rename pass across KB, including INDEX.md. Event-driven sweep queries in agent definition (`knowledge_base/agentic-ai/analysis/google-io-2026.md` dedicated analysis) will catch this. The agent-definition "Event-Driven Sweep Queries" section already has placeholders for Android XR and Gemma agent edge.

### 3.4 Mythos Consortium Exclusion (P2 — Strategic)

**Threat**: Mythos access is invitation-only via Project Glasswing (11 organizations). We are not in the consortium. If Mythos becomes the reference model for security-critical agentic workloads, our benchmark baselines become comparatively weaker (we'd always be benchmarking Opus 4.6 vs. competitor Mythos-level models on hardened red-team evals).

**Mitigation**: No action until public API is announced. Today: record the capability gap as a known latent risk. If Mythos rolls out publicly before I/O (low probability per current signals), re-evaluate eval baselines urgently.

### 3.5 Programmatic Tool Calling Bypass of post-tool-use Hooks (P1 — Security)

**Threat**: `code_execution_20260120` runs tool calls inside a container. Our `post-tool-use.sh` hook inspects Bash/Write tool invocations at the harness level. If programmatic tool calling invokes tools inside the container that don't trip the harness-level hook, our cmd_chain_monitor / MCP depth monitor / metachar detection could all be silently bypassed.

**Mitigation**: Analyze hook coverage before enabling programmatic calling for any fleet agent. If gap confirmed, add container-level instrumentation OR explicitly deny `code_execution_20260120` in `settings.local.json` until gap is closed. This is P1 because it's a security-architecture question, but not urgent (no agent currently uses programmatic calling).
