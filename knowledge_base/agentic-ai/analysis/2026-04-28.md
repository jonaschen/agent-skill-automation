### Finding 2 — Subagent Resumability Now Documented (New Phase 5 Primitive)

- **Signal**: The Agent SDK subagents documentation now describes a complete resumability contract: `agentId` returned in the Agent tool result, parent can `resume: sessionId` and reference the `agentId` to continue an interrupted subagent. Subagent transcripts persist in separate files, are unaffected by main-conversation compaction, and have a default 30-day cleanup. This is the **first long-running multi-agent pattern available without external state stores**.
- **Assessment**: **HIGH for Phase 5 / S2.** Subagent resumability changes the architectural calculus for our planned Phase 5 watchdog circuit-breaker. Our prior Phase 5 design assumed that a failed subagent had to be re-run from scratch (which was expensive given LLM API costs and token windows). With resumability, a watchdog can pause a misbehaving subagent, escalate, and resume from the same conversation point after operator approval. This is a substantively better architecture — it inverts the "kill and retry" pattern into "pause, inspect, resume" — and it appeared in documentation only between Apr 24 and Apr 28.
- **Impact on pipeline**: Phase 5 design index needs an addendum. Recommend a factory P2 design note that revises the watchdog state machine to incorporate the Apr 28 resumability primitive. The S2 paper should also cite this — it's a citable primitive for the heterogeneous orchestration paper's recovery-pattern section.
- **Source**: https://code.claude.com/docs/en/agent-sdk/subagents

### Finding 3 — `PostToolUse.duration_ms` Closes a Long-Standing S1 Eval Gap

- **Signal**: CC v2.1.119 added `duration_ms` (and on `PostToolUseFailure`) to the hook input payload. Per-tool execution timing is now available to hooks without external instrumentation. OTEL has parallel additions: `tool_use_id` + `tool_input_size_bytes` on `tool_result`.
- **Assessment**: **MODERATE for S1 immediately, HIGH for the autoresearch optimizer's eval surface long term.** Our `eval/run_eval_async.py` currently times the whole eval prompt-to-result loop but cannot attribute time to individual tool calls within a Claude session. With `duration_ms` exposed, the optimizer can reason about *which* tool calls dominate latency in the failing/passing distribution. This is a precondition for the optimizer evolving from "trigger rate optimization" to "trigger rate + per-tool latency joint optimization."
- **Impact on pipeline**: Worth wiring into the optimizer once Jonas upgrades CC to v2.1.119. Pre-CC-upgrade, no work is unblocked. Post-upgrade, file as a P2 enhancement: "Surface `PostToolUse.duration_ms` in optimizer eval logs."

### Finding 6 — Dispatch-Primitive Comparison Verdict: Transpiler Feasible

- **Signal**: The P0 deliverable `experiments/dispatch-primitive-comparison.md` was produced today as the S3 day-7 work-around. **Verdict: partially portable — transpiler feasible with name-binding + feature-degradation rules.** 5/12 axes fully portable, 2/12 partially, 5/12 not portable.
  - **Portable axes**: parameter schema (`prompt`), context passing (fresh-context convergence — both vendors independently arrived at the same isolation model), basic return contract, recursion ban (both vendors forbid sub-sub-agents identically), tool allowlist concept.
  - **Not portable**: tool surface (Claude has one dispatcher with `subagent_type` discriminator; Gemini has N tools, one per subagent) — solvable at compile time. Resume (Claude-only). Background/async (Claude-only). Six AgentDefinition fields are Claude-only (`maxTurns`, `effort`, `permissionMode`, `background`, `disallowedTools`, `memory`). Wildcards in tool grants (Gemini-only).
- **Assessment**: **HIGH for S3 architectural feasibility.** This is the first piece of S3 evidence that's positive. The Anthropic ↔ Google dispatch primitives **converge on the semantics that matter most** (fresh-context isolation, recursion ban, prompt-as-only-channel) and **diverge on details that a transpiler can absorb at compile time** (dispatcher-vs-named-tool surface). The "subagent_type" parameter is the most asymmetric field — Claude treats it as a parameter, Gemini encodes it in the tool name itself — but a transpiler emitting `Agent(subagent_type=X, prompt=Y)` for Claude and `X(prompt=Y)` for Gemini handles it cleanly. Resume and background dispatch are real Claude advantages with no Gemini equivalent today; for those, the transpiler must either drop them with a warning or polyfill.
- **Impact on pipeline**: S3 architectural risk is materially reduced. The transpiler approach is now the preferred S3 design path. Implementation remains gated on Gemini CLI install for empirical validation, but the architectural surface is mapped well enough to begin Phase 5 design refinement (e.g., layering S3 dispatch abstraction over the TCI router).
- **Source**: `knowledge_base/agentic-ai/experiments/dispatch-primitive-comparison.md`

### Cross-Pollination Opportunities

#### CP1 — Pause-Inspect-Resume Pattern as Phase 5 Design Centerpiece

The combination of **subagent resumability** (Finding 2) and **CC v2.1.120's auto-update breakage** (Finding 1, indirectly) suggests an architectural pattern not yet articulated in either vendor's documentation:

- A **watchdog that pauses misbehaving subagents** (e.g., detects token-burn anomaly, suspicious tool-call sequence, or human-flagged behavior) and **routes to human inspection** before deciding to resume vs. abort.
- This composes with the **fresh-context dispatch model** (both vendors): the resumed subagent can be re-prompted with corrective context the human supplies.

Neither vendor publishes this pattern explicitly. It's the natural composition of two primitives: (a) Anthropic's resumability + (b) topology-aware routing (our planned TCI). The S2 paper has space for this as a novel proposed pattern — let's call it **"Inspect-Resume"** to distinguish from the more common "kill+retry" recovery loop.

**Value**: This is a publishable pattern. Worth a Phase 5 design note that explicitly maps the watchdog to the resumability primitive.

#### CP2 — Dispatch-Primitive Convergence as a Citable S3 Anchor

The dispatch comparison reveals **five fully-portable axes** where Anthropic and Google independently converged on identical semantics:
1. Fresh-context isolation
2. Prompt as the only data channel from parent to subagent
3. Subagent recursion ban
4. Allowlist-based tool permissions
5. Description-driven discovery (both use `description` to drive matching)

This is the **strongest cross-vendor convergence pattern** documented to date in the agentic AI space. It is more substantive than the MCP convergence (which is a wire protocol; semantic alignment is partial). Convergence on five semantic axes for sub-agent dispatch suggests these are **emergent design constraints** rather than vendor-specific choices.

**S2 paper relevance**: This is paper-grade material. The "convergent evolution in dispatch primitives" framing is novel and supportable from documented sources.
