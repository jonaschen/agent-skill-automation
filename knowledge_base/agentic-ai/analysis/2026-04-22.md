## Gap Analysis Findings

### Finding 1: Deep Research Agent API — MCP Convergence Signal (S2/S3)

**Observation**: Google shipped two Deep Research models (Apr 21) via the Interactions API with native MCP server integration (auth headers, tool restrictions), collaborative planning, async execution (60-min max), and $1-7/task pricing. This is Google's first autonomous agent-as-API-endpoint.

**Why this matters for our pipeline**: The MCP integration is the headline. Google's newest agent product can connect to remote MCP servers — the same protocol our entire pipeline is built on. This validates our foundational design choice and creates a cross-vendor bridge that didn't exist a week ago.

**S2 impact — HIGH**: Deep Research's architecture maps remarkably well to patterns we're designing for Phase 5:

| Deep Research Pattern | Our Phase 5 Equivalent | Alignment |
|----------------------|----------------------|-----------|
| Collaborative planning (user reviews plan before execution) | Three-Agent Harness planning phase | Direct match |
| Async execution via Interactions API (`background=true`) | Phase 5.3.1 async task dispatch | Same pattern |
| MCP server integration for external tools | Pipeline MCP for tool federation | Identical protocol |
| 60-minute max execution timeout | Steward session 44-min average | Similar envelope |

**S3 impact — MEDIUM-HIGH**: Deep Research proves that MCP is becoming the vendor-neutral tool integration layer. If Google's research agents and our Claude-based pipeline both speak MCP, the S3 "tool portability" problem narrows significantly. The remaining S3 gap is agent definition format and orchestration protocol — but tool access is arguably the harder problem, and it's converging.

**Gap**: Our MCP integration is local-only (STDIO transport, local servers). Deep Research uses remote MCP servers with auth. Phase 5 should evaluate whether our pipeline's MCP servers could be made remotely accessible — this would enable external agents (including Google's) to call our tools.

### Finding 2: Agent-as-API Pricing Models Diverge — Phase 7 Design Input

**Observation**: Two agentic pricing models now exist in production:
- **Anthropic Managed Agents** (Apr 8): $0.08/session-hour + token costs. Time-based. Pay for think-time + execution time.
- **Google Deep Research** (Apr 21): $1-7/task flat. Outcome-based. Pay per completed research task regardless of duration.

**Phase 7 relevance**: Our Phase 7 (AaaS Commercialization) specifies "outcome billing" — charging per completed task rather than per token or per minute. Google's Deep Research validates this pricing model in production. Anthropic's Managed Agents show the alternative (time-based). The market is actively testing both models simultaneously — neither has won.

**Implication**: Phase 7 pricing design should support both models. A task that takes 5 minutes of LLM time costs $0.007 via Managed Agents (plus tokens) vs. $1-7 via Deep Research. The pricing spread is enormous, suggesting different market segments: Deep Research targets end-users willing to pay for convenience; Managed Agents targets developers building atop the API. Our AaaS model should probably offer both tiers.

**No action now** — Phase 7 is distant. File as design input.

## Cross-Pollination Opportunities

### 1. Deep Research Collaborative Planning ↔ Phase 5 Three-Agent Harness

Deep Research's collaborative planning feature (user reviews and optionally refines the investigation strategy before the agent executes) is architecturally identical to the "planning phase" in our Three-Agent Harness pattern. The key insight: Deep Research proves this pattern works at scale with real users. Our Phase 5.1 (TCI Router) should incorporate a planning confirmation step for high-complexity tasks — not just plan generation, but user-visible plan review before execution.

**Concrete design input**: Deep Research exposes the plan via the Interactions API (streamable). Our pipeline's equivalent would be the sprint-orchestrator's task decomposition output. Currently, the sprint-orchestrator plan is consumed programmatically by worker agents. Deep Research's model suggests exposing plans to human review for tasks above a TCI threshold (e.g., TCI > 0.7).

### 2. Interactions API Async Pattern ↔ Phase 5.3.1 Task Dispatch

Google's Interactions API uses `background=true` for async agent execution with polling/streaming for status. This is the same pattern our Phase 5.3.1 needs for async task dispatch to sub-agents. The API design choices (background flag, status polling, 60-min timeout, streaming partial results) are directly applicable to our dispatcher design.

**Cross-reference**: Anthropic's Managed Agents use a similar async pattern (session-based, status endpoint). The convergence of two vendors on the same async-agent-execution pattern strongly validates the design direction for Phase 5.3.1.

### 3. MCP as Cross-Vendor Tool Layer ↔ S3 Portability

Before Deep Research, MCP was Anthropic's protocol adopted by third parties. Now Google's own agent product natively integrates MCP servers. This changes the S3 calculus:

| Before Apr 21 | After Apr 21 |
|---------------|-------------|
| MCP = Anthropic ecosystem tool | MCP = multi-vendor tool protocol |
| S3 tool portability: open question | S3 tool portability: MCP is the answer |
| S3 focus: agent format + tools + orchestration | S3 focus: agent format + orchestration (tools converging) |

The remaining S3 gaps are agent definition format (SKILL.md vs. Gemini Skills) and orchestration protocol (agent-to-agent coordination). Tool access is converging on MCP. This simplifies the S3 problem surface by approximately one-third.

## Threats to Architecture

### 1. Remote MCP Exposure (MEDIUM, Security)

Deep Research connects to remote MCP servers with auth headers. If our Phase 5 design includes remote MCP exposure (to enable cross-vendor agent interop), we inherit a new attack surface: our MCP servers become externally reachable endpoints. Our current security model (STDIO transport, local-only, `mcp_config_validator.sh` for static analysis, `mcp_server_allowlist.json`) assumes local trust. Remote MCP requires:
- Authentication (API keys or OAuth at minimum)
- Rate limiting
- Input validation on all tool calls
- Network-level access control

This is not an immediate threat (Phase 5 hasn't started), but it should be noted in the Phase 5 design index as a security consideration.

### 2. Vendor Lock-In Asymmetry (LOW, Strategic)

Deep Research's MCP integration is "connect to remote MCP servers" — it's a consumer of MCP services. Our pipeline is both a provider and consumer of MCP services. If we expose our tools via MCP to enable Google agent interop, we become more portable (good for S3) but also more dependent on MCP protocol stability. MCP is still at spec v1.0 with no formal governance beyond Anthropic. A breaking MCP spec change would affect our pipeline disproportionately because we're deeper into the ecosystem than typical adopters.

Mitigation: the MCP spec has been stable (no breaking changes observed in this sweep cycle). The risk is theoretical but worth noting for Phase 5 design.

### 3. Shadow Eval Readiness (LOW, S1)

Per-test logging is now implemented (commit 79c98c7). The factory queue also includes the failure analysis template (A4-afternoon). When #49562 is patched and the shadow eval re-runs, we should be able to characterize failures immediately. The remaining risk: if the patch ships during a period when `daily_shadow_eval.sh` doesn't fire (it only runs when migration is pending), the re-run depends on the migration detection logic correctly identifying the new Opus 4.7 build as a migration target. This should be verified — the script should detect any Opus 4.7 patch as a trigger, not just a new model ID.
