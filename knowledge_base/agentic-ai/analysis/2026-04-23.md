## Gap Analysis Findings

### Finding 1: ADK v2.0.0b1 Graph Orchestration — Phase 5 TCI Design Impact (S2)

**Observation**: ADK v2.0.0b1 introduces `Workflow(BaseNode)` graph orchestration — a fundamentally different model from both CC's Agent tool trees (dispatch → results) and ADK v1.x's agent hierarchies (parent → child delegation). Key primitives:

| Primitive | What It Does | Our Equivalent |
|-----------|-------------|----------------|
| `BaseNode` | Root-level workflow container (replaces Agent as root) | `closed_loop.sh` state machine |
| `NodeRunner` | Isolated per-node execution with scoped context | `claude -p` per-agent sessions |
| `DefaultNodeScheduler` | Orchestrates node execution order via `ctx.run_node()` | `topology-aware-router` (planned) |
| `ReAct` loop node | Explicit reasoning-action cycles replacing implicit agent flows | No equivalent — our agents use implicit ReAct via Claude's native behavior |
| Conditional edges | Dynamic branching based on node output | TCI threshold routing (static at dispatch) |

**Why this matters for Phase 5**: Our TCI routing design makes a binary decision at dispatch time (Track A parallel vs. Track B flagship) based on a 4-dimension coupling score. ADK v2.0's graph model allows **dynamic mid-execution routing** — a node's output determines the next node. This is structurally more flexible.

**Gap assessment — MODERATE, not critical**:

1. **Our TCI is static-at-dispatch; ADK's graph is dynamic.** For our current pipeline tasks (skill generation, eval, optimization), static routing suffices — the pipeline is linear (GENERATE → VALIDATE → OPTIMIZE → DEPLOY) with well-defined transitions. Dynamic routing adds value for tasks with unpredictable branching (e.g., "if eval passes, deploy; if eval fails with type X error, optimize description; if type Y error, restructure tools"). Our `closed_loop.sh` already handles this via conditional logic, but it's imperative code, not declarative graph.

2. **`ReAct` as explicit node vs. implicit behavior.** ADK v2.0 makes the reasoning-action loop an explicit, configurable graph node rather than emergent model behavior. This is architecturally cleaner but practically equivalent for our use case — Claude already does implicit ReAct. The explicit ReAct node matters more for systems using weaker models that need structured prompting to achieve ReAct patterns.

3. **The TCI comparison framework (factory queue item) is now actionable.** v2.0.0b1 provides concrete code to compare against, not just alpha speculation. The framework's five dimensions can be filled from the beta release notes and code. **However**, the beta explicitly warns "not for production" and "not backwards compatible." Filling the framework now risks it being invalidated by GA changes at I/O (26 days).

**Recommendation**: Do NOT fill the TCI comparison framework from v2.0.0b1 beta. Wait for GA (expected I/O, May 19-20). The framework was designed for this — pre-built columns ready for post-I/O filling. ### Finding 3: Vendor Orchestration Philosophy Divergence (S2 — Paper Input)

**Observation**: ADK v2.0.0b1 crystallizes a philosophical split in multi-agent orchestration:

- **Anthropic approach**: Agent-centric. The agent IS the unit of computation. Orchestration happens through tool calls between agents (Agent tool, Managed Agents `execute()`). The runtime provides infrastructure (sessions, context), but orchestration logic lives in natural language agent instructions.
- **Google approach**: Workflow-centric. The workflow graph IS the unit of computation. Agents are nodes in a graph. Orchestration is structural (edges, conditions, schedulers). The runtime provides orchestration primitives (NodeRunner, ReAct nodes), not just infrastructure.

This isn't a new observation — it was implicit in the v2.0 alpha signals — but v2.0.0b1 makes it concrete with production-quality code.

**S2 paper impact**: This is a citable finding for the heterogeneous multi-agent orchestration paper. The divergence creates a natural experiment: does agent-centric or workflow-centric orchestration perform better for different task types? Our pipeline is currently agent-centric (tool calls between Claude sessions). Phase 5 could test the workflow-centric alternative if ADK v2.0 reaches GA.

## Cross-Pollination Opportunities

### 1. ADK v2.0 NodeRunner Isolation → Phase 5 Subagent Isolation

ADK v2.0's `NodeRunner` provides isolated per-node execution — each node gets its own execution context, preventing state leakage between nodes. Our Phase 5 credential isolation design (`credential-isolation-design.md`) proposes `env -i` + explicit passthrough for subagent isolation. The `NodeRunner` pattern is a higher-level version of the same idea: runtime-managed isolation instead of OS-level isolation.

**Phase 5 design input**: When migrating to Agent SDK (Phase 5.3.3), evaluate whether SDK session isolation provides `NodeRunner`-equivalent containment. If yes, OS-level `env -i` becomes defense-in-depth rather than primary isolation mechanism.

### 2. Explicit ReAct Nodes → Optimizer Branch Control

ADK v2.0's explicit `ReAct` loop nodes make the reasoning-action cycle configurable (number of iterations, exit conditions). Our `autoresearch-optimizer` runs implicit ReAct loops — the model decides when to stop iterating. Explicit ReAct control could:
- Cap optimizer iterations more precisely than the current "≤50 iterations" budget
- Enable branch-specific iteration limits (e.g., conservative branches get fewer cycles)
- Make iteration behavior deterministic for reproducibility

**Not actionable now**: Would require ADK adoption or building equivalent explicit-loop infrastructure. Note for Phase 5 experiment backlog.

### 3. Graph Visualization → Pipeline Observability

ADK v2.0's web UI graph visualization (distinct icons, shapes, active node rendering) provides real-time workflow state visibility. Our pipeline observability is text-based (JSON logs, CLI dashboard). A graph view of our pipeline (GENERATE → VALIDATE → OPTIMIZE → DEPLOY with branching on failure) would improve Jonas's operational visibility.

**Not actionable now**: Phase 7 observability chrome. But the ADK v2.0 web UI provides a design reference.

## Threat Assessment

### Threat 1: ADK v2.0 Graph Model Obsoleting TCI Routing — LOW

**Risk**: If graph-based workflow execution proves categorically superior to threshold-based routing, our Phase 5 TCI design would need fundamental rework.

**Assessment**: LOW. The two approaches solve different problems:
- TCI routing answers: "Should this task run in parallel or sequential?" (resource allocation)
- Graph workflows answer: "What is the execution order of this task's subtasks?" (workflow structure)

These are complementary, not competing. A graph-based workflow could USE TCI scoring to decide whether to parallelize a subgraph. The ROADMAP's decision frame ("Adopt ADK only if concrete benefit") remains correct. The TCI comparison framework will evaluate this properly post-I/O.
