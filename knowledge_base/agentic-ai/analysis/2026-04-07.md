## 1. Gap Analysis: Findings vs. Pipeline State

### 1.1 MCP Cost Amplification Attack — NEW THREAT CATEGORY for Phase 4 (P0)

**Finding**: Adversa AI demonstrates that malicious MCP servers can inflate per-query costs by **658x** with <3% detection rate by steering agents into prolonged tool-calling chains. This is distinct from data exfiltration attacks — it's a **financial denial-of-service**.

**Impact on our pipeline**:
- **Phase 4 (Closed Loop)**: Our `closed_loop.sh` state machine has an OPTIMIZE->VALIDATE retry counter (max 3) that prevents infinite optimization loops. But it has no protection against an MCP tool that creates recursive internal loops within a single step. A compromised MCP server in a generated Skill could:
  - Trigger repeated tool calls that exhaust the agent's step budget
  - Inflate API costs during eval runs (eval runner calls the agent which calls MCP tools)
  - Create a financial DoS against our nightly steward runs
- **Existing mitigation**: `scripts/cmd_chain_monitor.sh` alerts at >30 and blocks at >45 subcommands. This provides partial protection but monitors command-chain length, not MCP tool-call depth or cost accumulation.
- **The 658x figure is staggering**: A $0.10 query inflated to $65.80. Applied to our nightly fleet (8 agent runs, ~$5-15 each), a compromised MCP server could turn a $40-120/night bill into $26,000-$79,000.

**Gap**: No per-step cost ceiling in our pipeline. No MCP tool-call depth monitoring. The `cmd_chain_monitor.sh` monitors shell commands, not MCP interactions.

**Recommendation**: P0 — Implement a cost ceiling mechanism:
1. **Immediate**: Add MCP tool-call depth tracking to `post-tool-use.sh` — alert at >10 MCP calls per agent step, block at >20
2. **Near-term**: Add per-run cost ceiling to steward scripts — abort if estimated cost exceeds 5x the 30-day average for that agent
3. **Phase 4**: Extend `agentic-cicd-gate` to reject Skills that produce abnormal MCP call patterns during eval

### 1.3 Gemini CLI Passive Skill Extraction — ARCHITECTURAL DIFFERENTIATOR We Should Study (P2)

**Finding**: Gemini CLI nightly v0.36.0-nightly.20260406 introduces background memory service for passive skill extraction — the agent automatically learns and extracts reusable patterns from usage without explicit "remember X" commands.

**Impact on our pipeline**:
- **Current approach**: Our skill promotion pipeline is `scripts/skill_logger_hook.sh` → `scripts/promote_cases.py` — a human-triggered, manual process. We log real-world skill usage, then a human decides which cases to promote to the eval set.
- **Gemini CLI's approach**: Implicit pattern → automatic skill → runtime loading. Three-tier: workspace `.gemini/skills/`, user `~/.gemini/skills/`, extension.
- **The gap**: Gemini CLI closes the loop from "observed behavior" to "reusable capability" automatically. Our pipeline requires human intervention at the promotion step.
- **Relevance to Phase 5**: If we adopted passive skill extraction, the `meta-agent-factory` wouldn't only generate skills from explicit requirements — it could also generate skills from observed usage patterns, creating a self-improving skill library.

**Cross-reference with ADK SkillToolset**: ADK's `SkillToolset` runtime expertise generation (agents create new capabilities at runtime) is the complementary mechanism — passive extraction learns from usage, SkillToolset generates on demand. Combined, these represent a fully autonomous skill lifecycle.

**Gap**: Our skill lifecycle is: human requirement → factory → validate → optimize → deploy. Gemini CLI adds: observed usage → automatic extraction → runtime loading. We lack the first half (automatic extraction from usage).

**Recommendation**: P2 — Design an "auto-promotion" extension to `promote_cases.py` for Phase 5 planning. When `skill_logger_hook.sh` detects repeated patterns (same skill triggered >N times with similar context), automatically generate a promotion candidate for human review. This is the minimal viable version of passive skill extraction.

## 2. Cross-Pollination Opportunities

### 2.1 Implicit vs. Explicit Memory — Two Complementary Paradigms (NEW — STRATEGIC)

**What Gemini CLI has**: Background memory service that passively extracts skills from usage patterns. Auto-configure memory. Implicit knowledge capture.

**What Claude Code has**: Explicit memory system (`/memory`, "remember X"). Structured, user-directed. No passive extraction.

**What both approaches teach**: Neither is strictly better:
- **Implicit** (Gemini) risks: noise capture, pattern overfitting, stale skills from one-time behaviors
- **Explicit** (Claude) risks: missing patterns the user doesn't think to save, under-capturing repeated behaviors
- **Optimal**: A hybrid where implicit extraction proposes, and the user (or an evaluator) approves/rejects

**Applicability to our pipeline**: Our skill logger captures usage patterns explicitly. But we don't analyze them for implicit skill proposals. The hybrid approach maps to:
1. `skill_logger_hook.sh` captures all skill invocations (✅ exists)
2. **NEW**: An analysis step identifies recurring patterns across logs
3. **NEW**: Patterns above a threshold become skill proposal candidates
4. `meta-agent-factory` generates a skill from the proposal
5. Normal validate → optimize → deploy pipeline

This extends our pipeline from "factory generates from human requirements" to "factory generates from observed patterns + human requirements." It's Phase 5 material but worth designing now.

### 2.2 MCP Tool Annotations → Our Permission Model (NEW — HIGH VALUE)

**What MCP is building**: Tool annotations with `readOnlyHint`, `destructiveHint`, `idempotentHint` booleans. Five SEPs extending this vocabulary. Clients can auto-approve read-only tools, require confirmation for destructive ones.

**What we have**: `eval/check-permissions.sh` enforces mutually exclusive permissions (review agents can't Write, execution agents can't Task). But this is agent-level, not tool-level.

**Convergence opportunity**: MCP tool annotations provide per-tool risk metadata. If we annotate MCP tools in generated Skills with these hints, our CI/CD gate could:
- Auto-approve Skills that only use `readOnlyHint: true` tools (lower risk)
- Require additional security review for Skills using `destructiveHint: true` tools
- Flag Skills using un-annotated tools as higher risk (unknown behavior)

**Applicability**: Phase 4 security hardening. Extend `mcp_config_validator.sh` to check for tool annotation presence and flag un-annotated MCP tools.

Priority: P2 — implement when MCP tool annotations become widely adopted (currently SEP stage).

### 2.3 Harness Simplification Principle Applied to Our Security Stack (NEW — STRATEGIC)

**What we learned yesterday**: "Every component encodes a model limitation assumption. Stress-test as models improve."

**Application to today's MCP security findings**: Our security stack is accumulating layers:
1. `check-permissions.sh` — static YAML permission validation
2. `mcp_config_validator.sh` — MCP config + content scanning + hash pinning
3. `cmd_chain_monitor.sh` — command-chain length monitoring
4. `pre-deploy.sh` — Bayesian deployment gate
5. (Proposed) MCP tool-call depth monitoring
6. (Proposed) Per-run cost ceiling
7. (Proposed) `mcp-sec-audit` integration

**Each layer encodes an assumption**:
- (1) assumes agents can't self-enforce permission boundaries → probably still valid
- (2) assumes MCP servers can't be trusted → valid, growing more valid
- (3) assumes agents can enter infinite command loops → valid
- (4) assumes generated Skills need statistical quality evidence → valid
- (5) assumes MCP tools can create cost amplification → valid (658x demonstrated)
- (6) assumes cost can spiral without notice → valid
- (7) assumes our pattern-based scanning has gaps → valid (100% MCPTox vs our unknown rate)

**Key insight**: Unlike harness components (which may simplify as models improve), security layers are **cumulative** — they respond to attacker capabilities, not model capabilities. Attackers don't get weaker as models improve. So the simplification principle applies differently to security:
- **Harness components**: Stress-test and simplify as models improve
- **Security components**: Consolidate and standardize, but don't remove

**Recommendation**: Strategic — our security stack should be organized as a layered defense (each layer independent) rather than accumulated patches. Consider consolidating the MCP-related checks (2, 5, 7) into a single `mcp_security_suite.sh` that runs all MCP checks in one pass.

## 4. Strategic Observations

### 4.1 The Security-Cost Tradeoff Is the New Frontier

Three converging signals:
1. **658x cost amplification** — financial attacks via MCP
2. **84.2% tool poisoning success rate** — data attacks via MCP
3. **Zero servers declare tool permissions** — no server-side defense

The implication: MCP security is no longer just about data safety — it's about **financial safety**. A production MCP deployment must now protect against both data exfiltration AND cost amplification. Our pipeline's security stack must evolve from "prevent bad behavior" to "prevent bad behavior AND prevent excessive cost."

**Design principle**: Every MCP integration should have both a security gate (does it do bad things?) and a cost gate (does it cost too much?). These are independent dimensions.

### 4.3 The Agent Platform Landscape Is Bifurcating

Two distinct deployment models are emerging:

**Model A: CLI-Native (Claude Code, Gemini CLI)**
- Agents defined as markdown files in project directories
- Skills loaded from local filesystem
- MCP for tool integration
- Developer-focused, local-first

**Model B: Cloud-Native (Conway, Gemini Enterprise, Vertex AI Agents)**
- Agents deployed as persistent cloud services
- Skills distributed via marketplaces/registries
- A2A for agent-to-agent communication
- Enterprise-focused, cloud-first

**Our pipeline currently targets Model A** (generates SKILL.md files for Claude Code). Phase 7 AaaS envisions something closer to Model B. The emerging question: do we need to support both deployment models?

**Conway's `.cnw.zip` extension format** and **Gemini Enterprise agent registration** are the bridges — mechanisms for taking CLI-native skills and publishing them to cloud-native platforms.

**Recommendation**: Strategic — design Phase 7 with dual deployment in mind. Generated Skills should be packagable both as local SKILL.md files (Model A) and as distributable packages (Model B via Conway extensions or A2A registration).
