## 1. Gap Analysis: Findings vs. Pipeline State

### 1.1 Orchestrator-Worker Pattern — ANTHROPIC CANONICAL VALIDATION (P1 — Architecture Calibration)

**Finding**: Anthropic's engineering blog "How we built our multi-agent research system" is the most detailed public disclosure of their own production multi-agent architecture. Key parameters:
- **Effort scaling heuristics**: 1 agent + 3-10 tool calls for simple fact-finding; 10+ subagents with divided responsibilities for complex research
- **Tool description quality**: improving tool descriptions reduced task completion time by **40%** for future agents
- **Synchronous cohort**: lead waits for all subagents before proceeding (state consistency over throughput)
- **External memory persistence**: research plans + completed phases stored outside conversation context
- **Outcome-focused evaluation**: LLM judges scoring final output quality, not intermediate steps
- **Rainbow deployments**: gradual rollout for long-running processes (not hot-swap during active runs)

**Impact on our pipeline**:

| Pipeline Element | Anthropic Blog Insight | Implication |
|-----------------|----------------------|-------------|
| Phase 5 topology-aware-router | Effort scaling heuristics (1-3 tools = simple, 10+ agents = complex) | TCI threshold calibration: low-coupling tasks < 3 inter-agent dependencies, high-coupling ≥ 10 |
| SKILL.md trigger descriptions | 40% time reduction from improved tool descriptions | Trigger descriptions are performance-critical — every character of precision matters |
| autoresearch-optimizer | Outcome-focused LLM judges | Validates our Bayesian eval approach — we judge final trigger accuracy, not step quality |
| Steward agents (nightly runs) | Rainbow deployments (no hot-swap mid-run) | Coordinate steward restarts at natural phase boundaries, never mid-task |
| Phase 5 session state | External memory for context survival | Phase 5 router MUST store task state externally — in-conversation state is a gap |
| Phase 3 optimizer | Synchronous cohort (consistency over throughput) | Our A/B/C/D parallel branch eval is correct; synchronous comparison prevents race conditions |

**Key actionable insight — 40% from tool descriptions**: The research system blog quantifies what our autoresearch-optimizer already does intuitively. If 40% task completion improvement comes from better descriptions in Anthropic's own system, our optimizer's marginal returns on trigger description precision are higher than we may assume. This validates the G8 Iter 2 investment and suggests continued optimizer iterations have substantial remaining headroom.

**Key actionable insight — effort scaling heuristics for TCI**: The blog provides concrete calibration data for our Phase 5 TCI (Task Coupling Index) router:
- TCI ≤ 2 (1 agent + 3-10 tool calls): parallel Track A (simple, decomposable)
- TCI ≥ 7 (10+ agents, divided responsibilities): single flagship Track B (complex, tightly coupled)
- This is more precise than our current "high/low coupling" binary classification

**Gap**: Phase 5 TCI algorithm needs calibration anchors. Blog provides empirical data from Anthropic's production system. Phase 5 external session state design has not started.

**Recommendation**: P1 — (1) Add effort-scaling heuristics to Phase 5 TCI algorithm design (TCI score should map to agent count via blog's 1/10+ thresholds). (2) Add "external task state store" as Phase 5.3 requirement — Phase 5 router must be resumable after crash without losing task state.

### 1.2 MCP Server Cards — PHASE 5 AGENT AUTODISCOVERY ENABLER (P2 — Strategic)

**Finding**: MCP 2026 roadmap Priority 1 (Transport Evolution) includes **Server Cards** — structured `.well-known` metadata URLs exposing agent capabilities for auto-discovery by browsers, crawlers, and registries. No implementation timeline given; part of Transports Working Group scope. Positioned as the future of service discovery in MCP ecosystems.

**Impact on our pipeline**:

Currently, our Phase 5 topology-aware-router uses hardcoded agent topology (agents are defined in `.claude/agents/*.md` and the router selects from a known list). Server Cards would allow:
1. **Dynamic capability discovery**: Phase 5 router could query each agent's `.well-known` URL to discover what it can handle
2. **Runtime topology updates**: new agents could self-register without updating router configs
3. **Health-aware routing**: Server Cards could include health/capacity metadata for load-aware dispatch
4. **Inter-repo agent discovery**: our multi-repo steward fleet (android, ARM, BSP, LTC) could be discovered from a central registry

**Gap**: Our Phase 5 topology-aware-router assumes static topology. Dynamic agent registration is not in the current Phase 5 design. Server Cards spec is not yet finalized, so premature to implement — but Phase 5 architecture should be designed to accommodate `.well-known` discovery as an upgrade path.

**Recommendation**: P2 — Document Server Cards as a Phase 5 "future-ready hook" in ROADMAP. When Transports WG publishes spec draft, evaluate replacing hardcoded agent registry with `.well-known` discovery.

### 1.6 Gemini CLI Context Compression — STEWARD SESSION QUALITY OPPORTUNITY (P3 — Future)

**Finding**: Gemini CLI v0.38.0 (Apr 14) introduces **context compression services** as a first-class feature — explicit, user-controlled context window management rather than the implicit compaction Claude Code uses. v0.39.0-preview.0 adds context-aware persistent policy approvals that remember prior approvals by context, reducing repeated prompts.

**Impact on our pipeline**:

Our steward agents run hour-long sessions where context quality degrades as the window fills. Claude Code handles this via automatic compaction (PostCompact hook available). The difference:
- **Claude Code implicit compaction**: fires automatically when context fills, summarizes previous turns
- **Gemini CLI explicit compression**: user/agent-controlled, first-class API, explicit checkpoints

**Gap**: Our steward agents do not explicitly checkpoint context state. Long factory-steward or ltc-steward sessions may lose precision in the back half as compaction summaries replace detailed earlier context.

**Possible cross-pollination**: After each major phase within a session (e.g., "complete ROADMAP task X"), trigger an explicit context consolidation (similar to PostCompact) — write a structured summary of completed work + current state to a file, then reference that file in subsequent turns. This preserves semantic precision better than relying on automatic compaction.

**Recommendation**: P3 — Nice-to-have. If steward quality metrics show degradation in multi-hour sessions, implement explicit checkpoint files at phase boundaries. Not urgent given Phase 4 is nearly complete.

## 2. Cross-Pollination Opportunities

### 2.1 Gemini CLI Passive Skill Lifecycle → Our Promote_Cases Pipeline

**Gemini's approach** (v0.38.0/v0.39.0-preview.0):
1. Background memory service passively **observes** interactions
2. Extracts skill candidates automatically
3. Queues to **`/memory inbox`** for user review
4. User approves → skill **activated**

**Our current approach**:
1. `skill_logger_hook.sh` logs skill invocations manually
2. `scripts/promote_cases.py` runs manually to extract positive/negative cases
3. Human review required before adding to eval set
4. Activation = adding test to `eval/prompts/` + `eval/expected/`

**Cross-pollination opportunity**: Implement a lightweight **passive pattern extractor** on top of our existing skill logger data. After each factory-steward session:
1. Parse `logs/factory-YYYY-MM-DD.log` for skill invocation patterns
2. Extract: prompt → skill triggered / not triggered (binary signal)
3. Classify against current test set — flag novel prompts not covered
4. Queue to a `pending_cases/` directory (our equivalent of `/memory inbox`)
5. Human reviews pending cases weekly → promotes to eval set

This would automate what `promote_cases.py` does manually and make the eval set continuously grow from real usage rather than ad hoc promotion. The Gemini signal validates that this passive extraction approach is production-viable.

**Recommendation**: P2 — Write a `scripts/passive_case_extractor.py` proposal. This closes the eval set automation gap in our Phase 4 closed loop.

### 2.2 ADK Parameter Manager → Phase 7 Secrets Architecture

**Google's approach**: ADK agents pull configuration from Google Cloud Secret Manager / Parameter Manager at runtime. Resolves credential leakage bug in Agent Registry.

**Our current approach**: API keys via environment variables in shell scripts. Works for single-machine deployment but is not production-hardened for multi-tenant (Phase 7 AaaS).

**Cross-pollination**: For Phase 7, adopt externalized secrets management:
- Replace `$ANTHROPIC_API_KEY` env vars with runtime secret fetch (AWS Secrets Manager, GCP Secret Manager, or HashiCorp Vault)
- Agent configs reference secret paths, not literal keys
- Credential rotation without redeployment

**Recommendation**: P3 — Note in Phase 7 design. Not urgent until Phase 7 planning begins.

### 2.3 Anthropic Tool Description Quality (40%) → Optimizer ROI Amplification

**Finding**: Anthropic's own production systems achieved 40% task completion time reduction from improved tool/skill descriptions. This is the same axis our autoresearch-optimizer iterates on.

**Cross-pollination insight**: If 40% improvement is achievable on complex task completion from description quality alone, then:
1. Our optimizer's marginal return on each iteration is higher than naive pass-rate improvement suggests
2. Investing more eval budget (more iterations, larger test sets) in description optimization has verified ROI
3. The 40% figure is for task *completion time*, not trigger accuracy — but better descriptions likely improve both

**Actionable implication for our pipeline**: When factory-steward reviews skill performance, treat trigger description quality as the highest-leverage intervention point. Before adding more eval prompts or tweaking thresholds, first ask: "Is this skill's description precise enough to activate only when appropriate?"

**Recommendation**: P1 — Document in AGENTS.md or factory-steward instructions: "Tool description precision has highest measured ROI per engineering hour (40% task performance improvement confirmed by Anthropic engineering blog, 2026-04-16)." This becomes a standing calibration note for the optimizer.

## 4. Comparative Analysis: Anthropic vs. Google Architectural Trajectories

### 4.1 Agent Lifecycle Management Comparison

| Dimension | Anthropic (Claude Code v2.1.90) | Google (Gemini CLI v0.39.0) |
|-----------|--------------------------------|---------------------------|
| Context management | Implicit compaction (PostCompact hook) | Explicit compression service (first-class API) |
| Skill lifecycle | Manual creation → explicit activation | Passive extraction → inbox → approval → activation |
| Session state | In-conversation + auto memory files | In-session + background memory service |
| Hook system | 15+ event types, conditional, HTTP, defer/resume | Less mature, fewer event types |
| Multi-agent | Agent Teams with Git worktrees | ADK multi-agent with 4 languages |
| Discovery | Agent names in typeahead (`@`) | ADK registry + Agent Cards (in progress) |

**Key insight**: Anthropic leads in hook system maturity and multi-agent production readiness. Google leads in automated skill lifecycle (passive extraction) and explicit context management. These represent different philosophies: Anthropic gives developers precise control; Google automates more of the lifecycle.

**For our pipeline**: Anthropic's explicit control model aligns better with our quality-gated pipeline (we don't want automatic skill promotion — we want eval-gated promotion). But Google's passive extraction pattern is worth adopting for the *discovery* phase (finding candidate eval cases), while keeping human review for the *activation* phase.

### 4.2 Protocol Standardization Trajectory

| Protocol | Status | Trajectory |
|----------|--------|-----------|
| MCP (Anthropic-originated) | 10,000+ servers, OpenAI + Google + Microsoft adopting | Becoming universal tool connectivity standard |
| A2A (Google-originated, LF-donated) | 150+ orgs, stable v1.0, I/O v1.1 expected | Becoming inter-agent communication standard |
| MCP + A2A together | Complementary (tool access vs. agent communication) | Converging toward layered standard stack |

**Observation**: The ecosystem is converging on a two-protocol stack: MCP for tool/context access, A2A for agent-to-agent messaging. Our Phase 5 multi-agent topology should plan for both: MCP for how agents access tools, A2A for how agents communicate task state and results.

**Gap**: Our Phase 5 design uses Claude Agent SDK's native subagent mechanism (not A2A). This works but locks our inter-agent communication to Anthropic's stack. If we ever need to orchestrate a mix of Claude and ADK agents (e.g., using ADK-based Gemma 4 agents for cost efficiency), we'd need A2A.
