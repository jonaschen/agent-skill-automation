### 1.2 Token Burn Rate Root Cause Analysis

The token burn rate reports appear to have two compounding factors:
1. **Tokenizer change**: Opus 4.7 uses a new tokenizer producing 1.0-1.35x more tokens for the same content
2. **Default effort raised to `high`**: v2.1.94 (April 7) changed the default from `medium` to `high` for API-key/Bedrock/Vertex users

Combined effect: 1.35x tokenizer × ~1.5-2x effort increase = 2-2.7x token consumption increase. This aligns with user reports of "1% per prompt" (vs. previous ~0.3-0.5%).

**Impact on our pipeline**: Steward sessions currently have no dollar ceiling. A 2-2.7x increase in token consumption directly maps to cost. Our `--max-turns 50` provides iteration limits but not cost limits.

**Recommendation**: Add `--max-budget-usd 10.00` to all steward scripts as a safety net. This is a 1-line change per script.

## 2. OTEL Tracing Architecture Assessment (S1 relevance: automatic agent improvement)

### 2.1 Key Architectural Insight

The most significant finding of this sweep: **OTEL tracing is CLI-native, not SDK-dependent**. The SDK doesn't produce telemetry — it passes env vars to the CLI child process, and the CLI emits traces/metrics/events directly.

This means our existing `claude -p` pipeline can get full distributed tracing by adding 6 environment variables to the launch scripts:

```bash
export CLAUDE_CODE_ENABLE_TELEMETRY=1
export CLAUDE_CODE_ENHANCED_TELEMETRY_BETA=1
export OTEL_TRACES_EXPORTER=otlp
export OTEL_METRICS_EXPORTER=otlp
export OTEL_LOGS_EXPORTER=otlp
export OTEL_EXPORTER_OTLP_ENDPOINT=http://collector:4318
```

### 2.2 What This Enables

With OTEL tracing:
- Every steward run becomes a trace with spans for each tool call, API request, and subagent invocation
- `session.id` attribute correlates multi-turn sessions
- Token/cost counters as metrics
- Structured log events for each prompt and tool result
- Full trace tree: parent → subagent → tool call hierarchy

This directly advances **S1 (Automatic Agent Improvement)** — with traces, we can build automated analysis of agent behavior patterns, identify inefficient tool call sequences, and correlate performance with specific prompt patterns.

### 2.3 Implementation Complexity

| Requirement | Complexity |
|------------|-----------|
| Deploy OTEL collector | 1 day (Docker container) |
| Add env vars to scripts | 30 minutes (6 scripts) |
| Set up Grafana dashboard | 1 day |
| Total | 2-3 days |

This is dramatically lower than the estimated "months" if SDK migration were required.

## 3. Task Budgets CLI Gap Assessment

Task Budgets are **API-only** on Opus 4.7. The CLI provides `--max-budget-usd` (dollar cap) and `--max-turns` (iteration cap) as alternatives. These serve different purposes:

| Feature | Scope | Type | Available on |
|---------|-------|------|-------------|
| Task Budget | Full agentic loop (tokens) | Advisory (soft) | Messages API only |
| `--max-budget-usd` | Full session (dollars) | Hard cap | `claude -p` CLI |
| `--max-turns` | Number of turns | Hard cap | `claude -p` CLI |
| `max_tokens` | Per-request output | Hard cap | All surfaces |

For our pipeline: `--max-budget-usd` + `--max-turns` provides sufficient cost control without Task Budgets. If we later migrate to the Agent SDK's `query()` function, Task Budgets become available.

## 5. v2.1.113 Native Binary Spawn: Architectural Risk Assessment

The shift from "bundled JavaScript" to "native binary spawn" is the most significant architectural change in Claude Code since v2.0.0. Potential impacts:

- **Startup time**: Should improve (native binary vs. Node.js JIT)
- **Memory footprint**: Should decrease (no V8 heap for CLI logic)
- **Platform compatibility**: Per-platform optional dependency — could introduce platform-specific bugs
- **Our concern**: `claude -p` invocation path may differ with native binary. Verify steward scripts work correctly before production use.

**Recommendation**: Run one manual steward session on v2.1.113 and compare output/behavior to v2.1.112 before updating cron fleet.

## 6. Routines vs. Cron: Capability Comparison

| Feature | Our Cron Pipeline | Routines (Research Preview) |
|---------|------------------|---------------------------|
| Minimum interval | Any (we use 1h) | 1 hour |
| Branch push | Any branch | `claude/` prefix only |
| Multi-agent chaining | 3 agents per cycle | Single agent per routine |
| Performance logging | JSON records per run | No custom logging |
| Error handling | Script exit codes + logs | Unknown |
| Trigger types | Cron only | Schedule + API + GitHub |
| Cost | Our compute | Anthropic cloud |
| Daily limits | None | 5/15/25 by plan |

**Verdict**: Routines cannot replace our pipeline. API and GitHub triggers are interesting for Phase 5 HITL workflows (e.g., trigger an agent on PR open), but the core scheduling is inferior to our infrastructure.
