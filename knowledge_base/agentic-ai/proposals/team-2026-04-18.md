# Team Proposal — 2026-04-18

## Overview
This proposal introduces the **"Proposal Threshold" rule** to the research team's workflow to reduce documentation overhead and focus `factory-steward` on implementation rather than spec-reading.

## What to Change and Why
**Change**: Shift the primary instruction medium from standalone `.md` proposal files to the **Discussion verdict** for low-to-medium complexity tasks. Standalone proposals should only be generated for high-complexity architectural shifts or detailed implementation specs.

**Why**: 
- **Efficiency**: The `agentic-ai-researcher` spends significant time generating detailed proposals for items that get DEFERRED.
- **Focus**: `factory-steward` currently has to read 8+ new proposals per day. Many of these contain redundant information already present in the discussion transcripts.
- **Backlog Management**: A large number of small proposals creates a "document graveyard" where P2 items are lost in volume.

## Expected Benefit
- **Reduced Latency**: Discussion-to-implementation cycle time should decrease for P0/P1 items.
- **Improved Signal-to-Noise**: `factory-steward` sessions will focus on 2-3 high-value specs instead of 8+ mixed-value documents.
- **Cost Savings**: Reduced token consumption in researcher, lead, and steward sessions by eliminating redundant document parsing.

## Cost Implication
- **Neutral to Positive**: Fewer Claude sessions spent on redundant document generation and reading.

## Risk Assessment
- **Risk**: Important implementation details might be missed if the discussion verdict is too brief.
- **Mitigation**: The Research Lead must ensure the ADOPT verdict contains sufficient high-level detail, and the Researcher must use their judgment to escalate complex items to standalone proposals.

## Threshold Criteria
**Generate Proposal IF**:
1. Implementation requires >5 distinct file changes across multiple directories.
2. Item introduces a new architectural pattern (e.g., Session Rewind checkpointing).
3. Item involves complex security/permission logic (e.g., MCP hash pinning).

**Use Discussion Verdict ONLY IF**:
1. Item is a ROADMAP design note update.
2. Item is a simple grep/audit task.
3. Item is a known operational ritual (e.g., retirement verification).
4. Item is a Phase 5 requirement update.
