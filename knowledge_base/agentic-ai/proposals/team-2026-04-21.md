## Proposal 1: Fleet Manifest with A2A Agent Card Field Mapping

**Date**: 2026-04-21
**Triggered by**: A2A reaching 150+ org production scale (analysis Finding 2); discussion A1
**Priority**: P2
**Target Phase**: Phase 4 (operational) / Phase 5 (S3 foundation)
**Strategic Priority**: S3 — Platform Generalization

### Rationale

A2A is now a production inter-agent standard with triple-hyperscaler deployment (Azure AI Foundry, AWS Bedrock AgentCore, Google Cloud). Our fleet has no machine-readable catalog. The fleet manifest adopted from the 2026-04-20 discussion should use A2A Agent Card field names as the canonical schema, making it both an internal fleet management tool and a proto-Agent Card for future S3 interop.

### Proposed Specification

- **Name**: fleet-manifest
- **Type**: Data artifact (not a skill)
- **Output**: `fleet_manifest.json` at repo root
- **Field mapping** (from analysis Finding 2):
  - `name` → `name` (A2A native)
  - `description` → `description` (A2A native)
  - `version` → `version` (A2A native)
  - `capabilities` → `skills[]` (A2A native, array)
  - `provider` → `provider.organization` (A2A native, nested)
  - `status` → pipeline extension (no A2A equivalent — `active|suspended|deprecated`)
  - `model` → pipeline extension (no A2A equivalent)
  - `cron_schedule` → pipeline extension
- **Schema validation script**: `scripts/validate_fleet_manifest.sh` — as follow-up session, NOT bundled with initial implementation (per Engineer verdict)

### Implementation Notes

- Factory builds manifest first, validates against A2A schema separately
- Include all 10+ agents in the fleet (active and suspended)
- JSON format, human-readable with comments via `//` or `_comment` fields

### Estimated Impact

- S3: First machine-readable artifact that maps to A2A Agent Card schema
- Operational: Fleet state visible to `agent_review.sh` and future dashboards
- Cost: ~30 min factory time

## Proposal 4: Experiment Log Validator Inline Tests

**Date**: 2026-04-21
**Triggered by**: Validator deployed with zero test coverage on S1 critical path; discussion A4
**Priority**: P2
**Target Phase**: Phase 4 (S1 infrastructure integrity)
**Strategic Priority**: S1 — Automatic Agent/Skill Improvement

### Rationale

`eval/validate_experiment_log.py` guards experiment_log.json integrity — the S1 shadow eval loop depends on this file. The validator was deployed 2026-04-21 but has zero test coverage. A silent validation failure means the shadow eval cron's pre-flight check gives false assurance.

### Proposed Specification

- **Name**: validator-inline-tests
- **Type**: Test addition to existing script
- **Target file**: `eval/validate_experiment_log.py`
- **Test cases** (inline `if __name__ == "__main__"` runner):
  1. Missing required field (`model` absent) → should FAIL
  2. Duplicate entry (same model + timestamp) → should FAIL
  3. Valid entry with all required fields → should PASS
- **Execution**: `python3 eval/validate_experiment_log.py` (no args = run tests)

### Implementation Notes

- Inline tests, not separate file — keeps tests co-located with 10-line validator
- Tests use tempfile for test fixtures, clean up after
- 15 lines of code, 10 minutes of factory time

### Estimated Impact

- S1: Guards shadow eval pre-flight integrity check
- Prevents silent failures in the model migration critical path
- Minimal cost, high confidence return
