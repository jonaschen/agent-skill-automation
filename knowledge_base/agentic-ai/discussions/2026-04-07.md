## Round 1: MCP Cost Amplification Defense (P0 Threat)

### INNOVATOR

**Proposal 1.1 — MCP Tool-Call Depth Monitor in `post-tool-use.sh` (P0)**

Today's analysis identifies a new threat category: MCP cost amplification. Adversa AI demonstrates a 658x cost inflation with <3% detection rate. A malicious MCP server steers agents into prolonged tool-calling chains. Applied to our nightly fleet (8 runs, ~$5-15 each), a compromised MCP server could turn a $40-120/night bill into $26,000-$79,000.

Our `cmd_chain_monitor.sh` monitors shell command-chain length but NOT MCP tool-call depth. The two are orthogonal attack surfaces.

Concrete proposal:
1. Add an MCP tool-call counter to `.claude/hooks/post-tool-use.sh` — increment a per-session counter each time an MCP tool is invoked.
2. Alert threshold: >10 MCP calls per agent step. Block threshold: >20 MCP calls per agent step.
3. Use the same temp-file-per-session pattern as `cmd_chain_monitor.sh` (counter file at `/tmp/mcp_depth_${SESSION_ID}`).
4. On block, emit a structured JSON alert to `logs/security/mcp_depth_alert.jsonl` with agent name, MCP server, tool-call count, and estimated cost.

Files to change: `.claude/hooks/post-tool-use.sh` (add MCP depth tracking), new log directory `logs/security/`.

**Proposal 1.2 — Per-Run Cost Ceiling in Steward Scripts (P0)**

Even without MCP attacks, our nightly fleet has no cost guardrail. A bug, infinite loop, or model behavior change could cause runaway spending. The cost amplification finding makes this urgent.

Concrete proposal:
1. Add a `MAX_COST_MULTIPLIER=5` parameter to each `scripts/daily_*.sh` steward script.
2. Each script records its last 30 days of costs from `logs/performance/*.json` (the `estimated_cost` field exists in some but not all performance records).
3. Before the Claude invocation, compute the 30-day rolling average cost. Set a ceiling at `5x` that average.
4. After the run completes, check actual cost against ceiling. If exceeded, write a warning to `logs/security/cost_alert.jsonl` and email/notify.
5. For the first run (no history), use a hardcoded fallback ceiling of $50.

Files to change: All 6 `scripts/daily_*.sh` files (add cost ceiling logic), or extract to a shared `scripts/lib/cost_ceiling.sh` sourced by each.

**Proposal 1.3 — Extend `agentic-cicd-gate` to Reject Abnormal MCP Call Patterns (P1)**

The CI/CD gate currently evaluates trigger rate and security scans. It should also reject Skills that produce abnormal MCP interaction patterns during eval. If a Skill triggers >20 MCP calls during a single eval test, it's either poorly designed or compromised.

Concrete proposal:
1. During eval runs, count MCP tool calls per test prompt.
2. Add a new gate criterion: `max_mcp_calls_per_test ≤ 20`. If any test exceeds this, the Skill is flagged for review.
3. Add the metric to the eval JSON report alongside trigger_rate and security_score.

Files to change: `eval/run_eval_async.py` (count MCP calls), `.claude/agents/agentic-cicd-gate.md` (add criterion).

### ENGINEER

**On 1.1 (MCP Tool-Call Depth Monitor):**

The threat is real and the 658x figure is alarming. However, let me challenge the implementation: Claude Code's `post-tool-use.sh` hook fires after each tool call — it knows the tool name. But can we reliably distinguish MCP tool calls from regular tool calls (Read, Write, Bash, etc.) in the hook context? The hook receives tool name and parameters. MCP tools show up as `mcp__<server>__<tool>` in Claude Code. If that naming convention holds, we can pattern-match `mcp__*` to identify MCP calls specifically.

The 10/20 thresholds need justification. What's the legitimate maximum MCP call count per step in our current Skills? If we don't know, we risk false positives. But given this is a NEW threat with zero existing protection, I'd argue: ship it with generous thresholds (alert at 15, block at 25) and tighten later based on observed baselines.

**Verdict: ADOPT** — P0 is justified. The implementation cost is low (~30 lines in post-tool-use.sh), the blast radius is minimal (alert-only at first threshold), and the threat is financially catastrophic if realized. Widen the thresholds slightly to reduce false positive risk during rollout.

**On 1.2 (Per-Run Cost Ceiling):**

This is sound in principle but has a practical problem: how do we measure "actual cost" of a Claude run? The performance JSON records `duration` and `exit_code` but not all of them include `estimated_cost`. We'd need to either:
- (a) Add cost estimation to all steward scripts first (parsing Claude's token usage output), or
- (b) Use duration as a cost proxy (cost ≈ duration * rate, where rate is derived from model pricing).

Option (b) is simpler and probably good enough for a ceiling check. Duration-based ceiling is: if a run takes >5x the 30-day average duration, abort. Duration correlates well with cost for the same model.

The shared `scripts/lib/cost_ceiling.sh` approach is right — don't duplicate across 6 scripts. But this is a non-trivial change to all steward scripts. Test it on one steward first (factory, since it runs 3x/day) before rolling to all.

**Verdict: ADOPT** — Use duration-as-proxy for v1. Lower priority than 1.1 because this is a reactive check (post-run), not a preventive one (mid-run abort). Still P0 but second in implementation order.

**On 1.3 (CI/CD Gate MCP Call Patterns):**

Good idea but harder than it looks. Our eval runner (`run_eval_async.py`) invokes Claude via subprocess and checks whether the Skill triggered. It doesn't instrument the agent's internal tool calls — it just checks the final output. To count MCP calls per test, we'd need to either:
- (a) Parse Claude's verbose output for MCP tool invocations, or
- (b) Run eval with `post-tool-use.sh` active and count from the hook's logs.

Option (b) is more robust but couples eval infrastructure to hook infrastructure. Option (a) is brittle (output format can change — L10 taught us this).

This is valuable but the implementation complexity is P1, not P0. And it only protects against Skills we generate — it doesn't protect against MCP servers referenced by existing Skills changing behavior post-deployment (that's what 1.1 covers at runtime).

**Verdict: DEFER to P1** — Good defensive depth, but the runtime monitor (1.1) provides immediate protection. Implement this after 1.1 and 1.2 are stable. When we do implement, option (b) is the right path — use the hook's own data rather than parsing output.

## Round 2: Security Framework Coverage + Passive Skill Extraction

### INNOVATOR

**Proposal 2.1 — Integrate `mcp-sec-audit` into CI/CD Gate (P1)**

Yesterday we shipped `mcp-scan` for hash pinning. Today's analysis reveals `mcp-sec-audit` — a different tool with 100% detection on the MCPTox benchmark. `mcp-scan` does hash-based pinning (prevents rug pulls). `mcp-sec-audit` does static + dynamic analysis (detects active exploitation patterns). These are complementary, not overlapping.

Concrete proposal:
1. Install `mcp-sec-audit` as a dev dependency.
2. Add a `mcp_sec_audit()` function to `eval/mcp_config_validator.sh` that runs `mcp-sec-audit` against each `.mcp.json`.
3. In the `closed_loop.sh` SECURITY_SCAN node, run both `mcp-scan` (hash pinning) AND `mcp-sec-audit` (vulnerability scanning).
4. If either tool flags a concern, the Skill is blocked.

Files to change: `eval/mcp_config_validator.sh` (add mcp-sec-audit call), `scripts/closed_loop.sh` (update SECURITY_SCAN node).

**Proposal 2.2 — Consolidate MCP Security into Unified Suite (P2)**

Our MCP security stack is accumulating layers organically:
- `check-permissions.sh` — agent-level permission validation
- `mcp_config_validator.sh` — config + content scanning + hash pinning
- `cmd_chain_monitor.sh` — command-chain length (partial MCP coverage)
- (Proposed) MCP depth monitor in `post-tool-use.sh`
- (Proposed) `mcp-sec-audit` integration

The analysis correctly observes: security layers should be consolidated, not accumulated as patches. Today's recommendation of a single `mcp_security_suite.sh` is the right architectural call.

Concrete proposal:
1. Create `eval/mcp_security_suite.sh` that orchestrates ALL MCP-related security checks in a single pass.
2. Consolidate: config validation, content scanning, hash pinning, `mcp-sec-audit`, and tool annotation checks (when available).
3. Each check produces a structured sub-report. The suite produces a unified JSON verdict.
4. `closed_loop.sh` and `pre-deploy.sh` call the suite instead of individual scripts.

Files to change: New `eval/mcp_security_suite.sh`, refactor `eval/mcp_config_validator.sh` into callable functions, update `scripts/closed_loop.sh` and `.claude/hooks/pre-deploy.sh`.

**Proposal 2.3 — Auto-Promotion Extension for Skill Logger (P2)**

Gemini CLI's passive skill extraction closes the loop from "observed behavior" to "reusable capability" automatically. Our pipeline requires human intervention at the promotion step (`skill_logger_hook.sh` → manual `promote_cases.py`).

Minimal viable proposal:
1. Extend `scripts/promote_cases.py` with an `--auto-detect` mode.
2. In this mode, it reads the last N days of skill usage logs from `logs/skill_usage/`.
3. It identifies repeated patterns: same skill triggered >5 times with similar context (cosine similarity on prompt embeddings or simple keyword overlap).
4. For each pattern above threshold, generate a "promotion candidate" entry in a new file: `eval/promotion_candidates.json`.
5. A human (or the factory steward) reviews candidates and promotes them to the eval set.

This isn't full passive extraction (Gemini's approach) — it's "semi-automated promotion" that surfaces candidates for human review. It bridges our current manual process and a future fully automated one.

Files to change: `scripts/promote_cases.py` (add `--auto-detect` mode), new `eval/promotion_candidates.json`.

### ENGINEER

**On 2.1 (mcp-sec-audit Integration):**

The 100% MCPTox detection rate is impressive, but I have three concerns:

1. **Availability**: Is `mcp-sec-audit` a stable, installable tool or a research prototype? The analysis mentions it alongside a benchmark (MCPTox) which suggests academic origin. If it's not `pip install`-able or `npx`-able, integration cost jumps significantly.

2. **Overlap with our existing scanning**: Our `mcp_config_validator.sh` already does content scanning with injection pattern detection. What does `mcp-sec-audit` catch that our regex-based scanner doesn't? Without running both against the same test corpus, we don't know the marginal value.

3. **Dynamic analysis concerns**: "Static + dynamic" means it might actually *run* MCP servers to test them. In a CI/CD pipeline, executing untrusted MCP servers during validation is itself a security risk.

**Verdict: DEFER** — Evaluate `mcp-sec-audit` in isolation first. Run it against our existing `.mcp.json` files and compare detection results with our content scanner. If it catches things we miss AND is installable as a CLI tool, adopt it. Don't integrate blind based on a benchmark number. Create a time-boxed evaluation task (2-4 hours).

**On 2.2 (Consolidated MCP Security Suite):**

Architecturally correct but premature. We have 2 MCP security components today (`mcp_config_validator.sh` with hash pinning, `cmd_chain_monitor.sh`). We're proposing to add 2 more (depth monitor, `mcp-sec-audit`). Consolidation makes sense AFTER we have 4+ components, not before.

The refactoring cost is non-trivial: extracting functions, building a unified report schema, updating two calling scripts. And it introduces a coupling risk — a bug in the suite breaks ALL security checks simultaneously.

**Verdict: DEFER to after Proposals 1.1, 1.2, and 2.1 are implemented.** When we have 4+ MCP security components, consolidation earns its complexity. Right now it's premature abstraction — we're consolidating components that don't exist yet.

**On 2.3 (Auto-Promotion):**

I like the direction but challenge the implementation approach. Cosine similarity on prompt embeddings requires either an embedding API call (cost) or a local embedding model (complexity). Simple keyword overlap is too noisy — "create an agent" appearing 5 times might just be the same user doing their job, not a skill gap.

A simpler v1: count exact skill-name + trigger-verb pairs. If the same skill is triggered >N times with the same leading verb ("create", "build", "design"), log it as a candidate. No embeddings needed, no false positives from keyword noise.

More importantly: how much usage data do we actually have? The skill logger was installed recently (per memory). If we have <50 logged invocations, pattern detection is meaningless. This should wait until we have a statistically meaningful sample.

**Verdict: DEFER to Phase 5** — The concept is sound and aligns with Phase 5's skill lifecycle goals. But it requires (a) sufficient usage data (>100 logged invocations), (b) a simpler detection heuristic than embeddings, and (c) Phase 4 stability. Add it to the Phase 5 planning section of ROADMAP.md as a design task, not an implementation task.

## Round 3: A2A Evaluation + Context Efficiency + Security Architecture Principles

### INNOVATOR

**Proposal 3.1 — Lightweight A2A SDK Evaluation (P2, 2-4 hours)**

A2A v1.0 now has formal governance (8-org TSC at Linux Foundation) and 5-language SDK coverage. ROADMAP task 5.3.0 requires evaluating A2A vs. custom message bus for the scrum-team-orchestrator. Today's governance announcement de-risks A2A adoption — governed protocols don't die.

Concrete proposal:
1. Install the Python A2A SDK.
2. Register a minimal test agent with an AgentCard.
3. Send a test message and verify round-trip interop.
4. Document: install experience, API ergonomics, message format compatibility with our 6-type schema, overhead vs. direct function calls.
5. Write findings to `knowledge_base/agentic-ai/evaluations/a2a-sdk-eval.md`.

This is a 2-4 hour time-boxed research task, not an implementation commitment. It produces the data needed for the 5.3.0 decision gate.

Files to change: New `knowledge_base/agentic-ai/evaluations/a2a-sdk-eval.md`.

**Proposal 3.2 — MCP Tool Annotation Awareness in Permission Model (P2)**

MCP is developing tool annotations: `readOnlyHint`, `destructiveHint`, `idempotentHint`. Five SEPs are extending this vocabulary. Today's analysis identifies a convergence opportunity: our permission model is agent-level (`check-permissions.sh`), but MCP annotations provide per-tool risk metadata.

Concrete proposal:
1. Extend `eval/mcp_config_validator.sh` to check for the PRESENCE of MCP tool annotations in referenced MCP servers.
2. If tool annotations are present: log them and factor into risk scoring (read-only tools = lower risk, destructive = higher risk, un-annotated = unknown/flagged).
3. If annotations are absent: log a warning but don't block (most servers don't annotate yet).
4. This is forward-compatible — as more servers adopt annotations, our gate automatically gets better.

Files to change: `eval/mcp_config_validator.sh` (add annotation check).

**Proposal 3.3 — Progressive Context Loading for Factory Agent (P3)**

Google's ADK finding: 50% smaller context files improved agent quality. Our `meta-agent-factory` reads the full CLAUDE.md (~400 lines), the full agent definition, and all references on every invocation. We could pass only the relevant ROADMAP section and task-specific context.

Concrete proposal:
1. Add a `--context-mode` parameter to the factory agent: `full` (current behavior) or `progressive` (loads metadata first, fetches details on demand).
2. In progressive mode, pass only: the agent definition skeleton, the specific Phase section from ROADMAP, and the target skill's existing references. Omit: full CLAUDE.md, other agent definitions, lessons learned.
3. Measure: token count reduction, generation quality (eval pass rate), and cost savings.

Files to change: `.claude/agents/meta-agent-factory.md` (add progressive context instructions).

### ENGINEER

**On 3.1 (A2A SDK Evaluation):**

This is the right size and scope. A 2-4 hour time-boxed evaluation with documented findings is exactly what task 5.3.0 needs. The governance announcement (8-org TSC) genuinely changes the risk calculus — A2A is no longer a "might be abandoned" gamble.

One concern: the evaluation should explicitly test whether A2A's message format can express our 6 message types (TASK_ASSIGNMENT, PARTIAL_OUTPUT, REVIEW_REQUEST, REVIEW_RESULT, ESCALATION, WATCHDOG_HALT). If it can, A2A provides interop for free. If it can't, we need custom extensions, which erodes the benefit.

**Verdict: ADOPT at P2** — Time-box it. Document format compatibility. Don't start implementing A2A — just evaluate whether it fits. This is exactly the kind of low-cost research that prevents expensive wrong decisions later.

**On 3.2 (MCP Tool Annotations):**

Forward-compatible and low cost, but there's a timing problem: "Five SEPs extending this vocabulary" means the annotation schema isn't stable yet. If we code against today's `readOnlyHint`/`destructiveHint` and the SEPs change the field names or semantics, we've built against a moving target.

The proposal's saving grace is that it's check-and-log, not check-and-block. Logging annotation presence is safe even if the schema changes — we just update the field names. And the "absent = warning, not block" policy means zero false positives.

But "zero servers declare tool permissions" today (per the analysis). So this check would fire exactly zero times on real inputs right now. It's defensive coding for a future that may arrive in months.

**Verdict: DEFER** — The annotation spec isn't stable (SEP stage), and zero servers implement it today. When MCP SDK V2 drops (the v1 has been frozen 74 days), annotations will likely be part of it. Implement then, when there's actual data to check against. Add a tracking note to ROADMAP.md under Phase 4 security.

**On 3.3 (Progressive Context Loading):**

The Google finding is interesting but our pipeline is different from ADK's "vibe building" use case. ADK's 50% reduction improved *conversational* prototyping — an interactive, iterative flow. Our factory generates Skills in a single shot — it reads context once and produces output once. The benefit of progressive loading is mainly in interactive sessions where irrelevant context accumulates over turns.

For our single-shot factory, the more likely improvement is *pruning* irrelevant context, not *progressive loading*. And we'd need to measure first: is the factory actually suffering from context noise? The current T=0.895 suggests the description quality is fine. If it ain't broke...

Also, modifying the factory agent definition to support a `--context-mode` adds complexity to an agent that's already been carefully tuned (G8 Iter 2 pushed V to 0.900). Changes to the factory definition risk regression — L7 taught us this.

**Verdict: REJECT** — The cost-benefit doesn't justify the risk. Current factory performance is adequate. The 50% context reduction finding applies to interactive sessions, not single-shot generation. If factory quality degrades in the future, revisit this as a potential remedy. Don't preemptively optimize.
