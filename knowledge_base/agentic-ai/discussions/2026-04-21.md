**Proposal 1.3: Haiku 3 Guard Pattern — Generalized Model Lifecycle Event System (S1)**

Today's analysis validates our Haiku 3 guard end-to-end — our automation detected the retirement before the vendor's own page updated. This is genuinely an S1 milestone: the pipeline detected a platform change faster than the platform's documentation reflected it.

But the guard is currently hardcoded for model deprecation events only. The same pattern applies to other lifecycle events: SDK deprecations, API version sunsets, MCP spec changes, protocol version bumps. Proposal: generalize the `deprecated_models.json` + `model_deprecation_check.sh` pattern into a `lifecycle_events.json` + `lifecycle_check.sh` that handles any dated lifecycle event (not just model retirement).

Schema extension:
```json

  "event_type": "model_retirement|sdk_sunset|api_deprecation|protocol_version",
  "identifier": "claude-3-haiku-20240307",

  "replacement": "claude-haiku-4-5-20251001",
  "guard_action": "warn|block",
  "verified_clean": false
}
```

**On 1.3 (Generalized Lifecycle Event System):**

This is a good idea at the conceptual level but premature at the implementation level. The Haiku 3 guard works because model deprecation events have a specific, predictable shape: known model ID → known retirement date → known replacement → grep for references. SDK deprecations, API sunsets, and protocol changes have fundamentally different shapes — an SDK sunset might mean "this function signature changes," an API deprecation might mean "this endpoint returns 410," a protocol version bump might mean "the wire format changed."

Trying to unify these under one schema forces false uniformity. The `guard_action: "warn|block"` field, for example — what does "block" mean for a protocol version bump? Block deployment? Block what deployment? The model deprecation check blocks deploys that reference deprecated models. An SDK sunset check would need to block deploys that import deprecated functions. These are different grep patterns, different check logic, different remediation paths.

What IS generalizable: the registry pattern (JSON file of dated events) and the scheduled check pattern (pre-deploy + post-retirement audit). But the check logic needs to be event-type-specific.

**Verdict: DEFER to Phase 5.** The pattern is proven (Haiku 3) and reusable (Opus 4/Sonnet 4 June 15 retirement uses the same code unchanged). Generalization makes sense when we have 3+ different event types to generalize across. Right now we have 1. Build the second instance (SDK deprecation check, when one actually occurs) and THEN extract the pattern.
