## Round 2: Eval Set Growth — Passive Extraction vs. Manual Promotion, and MCP False-Positives

### INNOVATOR

**Proposal 2.1: Passive Case Extractor — Automate the Promote_Cases Pipeline**

Our eval set grows by manual promotion: `promote_cases.py` runs occasionally, a human reviews candidate prompts, and new tests get added. The analysis identifies a concrete cross-pollination opportunity from Gemini CLI v0.39.0's passive skill lifecycle: background observation → candidate queue → human review → activation.

We already have the raw material: `skill_logger_hook.sh` logs every skill invocation in two instrumented projects (49 entries already, source of test_55-59). The missing piece is automatic candidate *extraction* — today a human parses log entries to decide which are novel enough to promote.

Concrete proposal: write `scripts/passive_case_extractor.py` that:
1. Reads `logs/skill_usage.jsonl` (the logger hook output)
2. For each logged prompt, computes cosine similarity against existing `eval/prompts/test_*.txt` content using sentence embeddings (or a simple TF-IDF bag-of-words approximation if we want zero new deps)
3. Flags prompts where max similarity < 0.7 (no existing test covers this semantic territory)
4. Writes flagged prompts to `pending_cases/YYYY-MM-DD/` with the logged skill-triggered label
5. Human reviews `pending_cases/` weekly and runs `promote_cases.py` on approved entries

This keeps human judgment in the loop (avoids auto-promotion quality collapse) while removing the tedious "which of these 49 entries are novel?" triage step. Expected effect: eval set grows from 59 to 70-80 tests within 60 days from natural usage alone.

**Proposal 2.2: MCP Ecosystem False-Positive Tests**

10,000+ registered MCP servers means MCP vocabulary is now mainstream language. The analysis flags a growing risk: "configure/debug/update my MCP server" prompts shouldn't trigger meta-agent-factory, but they increasingly share vocabulary with legitimate triggers ("create/add/build an MCP integration"). Our eval set has test_40-44 for cross-domain conflicts, but no MCP-specific false-positive coverage.

Concrete change:
- Add 5 new test cases to eval set (test_60-64):
  - "Add MCP authentication to my existing server" → NOT meta-agent-factory
  - "Debug why my MCP server's tool calls are failing" → NOT meta-agent-factory
  - "Update the tool list in my MCP server config" → NOT meta-agent-factory
  - "Configure MCP server rate limiting" → NOT meta-agent-factory
  - "I need an MCP integration for my project's GitHub workflow" → AMBIGUOUS, likely NOT (modifying existing, not creating new agent)
- Update `eval/splits.json` to add these to Training set T (T=44, V=20)
- Run regression eval to confirm existing descriptions don't suddenly fail on the new negatives

This closes a real gap before the MCP ecosystem vocabulary expands further.

**Proposal 2.3: Structured Pending_Cases Review in Factory-Steward Cadence**

The passive extractor creates a queue. But a queue that nobody checks is just log files. The factory-steward already runs 5+ times daily. Add a lightweight "check pending_cases/ and summarize" step to the evening factory-steward run — no promotion decision required, just a one-line summary of how many candidates are waiting and when the oldest was logged. This creates visibility without adding a new ceremony.

### ENGINEER

**Response to 2.1 (Passive Case Extractor):**

The proposal is sound in principle but I want to challenge the similarity threshold. TF-IDF bag-of-words at 0.7 cosine similarity on short prompts (our test prompts are typically 10-30 words) will have poor discriminative power. Two prompts sharing "create" and "agent" and "skill" will score >0.7 even if they represent genuinely different trigger scenarios. Sentence embeddings are more accurate but add a non-trivial dependency (sentence-transformers is 400MB+).

My suggested alternative: use the eval runner itself as the similarity oracle. A novel prompt is one that, when run through `run_eval_async.py` with the current SKILL.md, produces a different outcome than expected from its label. This is already our ground truth — instead of approximate similarity, we get *exact behavioral novelty*. The extractor becomes:
1. Read logged prompts with their observed trigger/no-trigger outcome
2. Run each through `run_eval_async.py` with `--single-test`
3. If the model's response disagrees with the observed outcome, flag as "behavioral novelty" candidate
4. Queue for human review

This has higher implementation cost (it burns API tokens per candidate) but generates candidates that are definitionally useful for the eval set — they represent *cases where the model disagrees with logged behavior*.

Also: the `pending_cases/` directory needs a format spec before we implement the extractor. What fields does each entry have? Prompt text, logged label, similarity score (or behavioral outcome), timestamp, source project. This should be in a `pending_cases/README.md` before code is written.

**Verdict on 2.1**: ✅ Adopt, but replace TF-IDF similarity with eval-runner behavioral novelty detection. Spec the `pending_cases/` format first. This is a P2 item — post-Phase-4-close.

**Response to 2.2 (MCP False-Positive Tests):**

This one is clean. Five targeted test cases, all T-set additions, no architecture change required. The "I need an MCP integration" test case is the most important because it's the exact near-miss that's hardest to classify: "I need an X" phrasing usually triggers meta-agent-factory, but "MCP integration for GitHub workflow" suggests modifying an existing system, not creating a new agent.

One process note: adding test_60-64 without updating the regression baseline (`eval/regression_baseline.json`) would flag a false regression on the next steward run (the new tests won't be in the baseline). The implementation sequence matters:
1. Add test_60-64 to `eval/prompts/` and `eval/expected/`
2. Update `eval/splits.json` (T=44, V=20)
3. Run regression_test.sh with `--update-baseline` to absorb the new tests
4. Verify new negatives pass with current description before committing

**Verdict on 2.2**: ✅ Adopt (P2). Important to run `--update-baseline` before committing.

**Response to 2.3 (Pending Cases in Factory-Steward Cadence):**

I like the intent — visibility without ceremony — but not the mechanism. Adding logic to `daily_factory_steward.sh` to check a directory that doesn't exist yet creates dead code until 2.1 ships. And the factory-steward's session prompt is already long; appending a conditional directory check adds fragility.

Simpler alternative: add pending cases to the existing `health_dashboard.py` output. The dashboard already summarizes agent states, performance, and alerts. A `pending_cases/` count line costs ~5 lines of Python and surfaces the queue alongside everything else the dashboard already shows.

**Verdict on 2.3**: ✅ Adopt in reduced form — add pending_cases count to `health_dashboard.py`, not to factory-steward session prompt. Deferred until 2.1 exists.

## Round 3: Architecture Horizons — Server Cards, A2A/ADK Lockout Risk, and Mythos Readiness

### INNOVATOR

**Proposal 3.1: Phase 5 Forward-Compatibility Hook for MCP Server Cards**

MCP Server Cards (structured `.well-known` agent capability metadata) are in the Transports Working Group scope but have no implementation timeline. Today's analysis recommends documenting this as a forward-compatibility note rather than implementing anything. I want to go slightly further.

Our Phase 5 topology-aware-router will be built around a static agent registry — a hardcoded list of agents defined in `.claude/agents/*.md`. If we build Phase 5 with this assumption baked into the router *algorithm* (not just the *configuration*), retrofitting Server Cards later requires refactoring the core routing logic.

A one-day investment now: design the Phase 5 router to read agent capabilities from a structured config file (`agents_registry.json` or similar) rather than directly parsing `.md` frontmatter. The config file today would be manually maintained (no `.well-known` involved), but its schema would be designed to mirror what Server Cards will eventually expose (name, capabilities, tool list, health endpoint). When Server Cards stabilize, we replace the manual config with a discovery query — zero router logic changes needed.

This is the "graceful accommodation" principle applied to a known future protocol.

**Proposal 3.2: Post-I/O ADK Decision Gate as an Explicit ROADMAP Item**

The analysis recommends no new ADK/A2A integrations until after Google I/O (May 19-20). But "don't do ADK until after I/O" is currently undocumented — it lives in today's analysis and the researcher's informal recommendations. If the factory-steward or a future collaborator reads the Phase 5 section in isolation, nothing blocks them from starting an A2A integration that may need full rewrite in 34 days.

Concrete change:
- Add an explicit **HOLD: No ADK/A2A integration until post-I/O (after 2026-05-20)** note at the top of the Phase 5 section in ROADMAP.md
- After I/O, the factory-steward's post-I/O sweep (already in `daily_research_sweep.sh` event queries) triggers a decision point: evaluate A2A v1.1 + ADK v2.0 for Phase 5 inter-agent communication vs. staying on native Claude Agent SDK subagent calls
- This converts an informal researcher recommendation into a durable architectural gate

**Proposal 3.3: Mythos Eval Harness Readiness — One-Line Config Change**

When Mythos (`claude-mythos-preview-*`) becomes publicly available, we want to benchmark it against our meta-agent-factory eval immediately — not spend a sprint wiring it up. Currently, `eval/run_eval_async.py` has the model ID hardcoded or read from a config. If we add a `--model` CLI flag (if it doesn't already have one) and ensure the eval can run against any model ID, the Mythos benchmark is a single command change when the API opens.

Concrete change: verify `eval/run_eval_async.py` accepts `--model <model-id>` as a flag. If it doesn't, add it (10 lines of argparse). Document the Mythos benchmark command in ROADMAP.md Phase 6 section: `python eval/run_eval_async.py --model claude-mythos-preview-XXXX --skill .claude/agents/meta-agent-factory.md`.

This is a 30-minute preparatory action that eliminates a future friction point with zero blast radius.

### ENGINEER

**Response to 3.1 (Server Cards Forward-Compatibility Hook):**

The principle is sound — designing for known future protocols costs less now than refactoring later. But I want to pressure-test the proposal on blast radius.

The "structured config file instead of parsing .md frontmatter" approach introduces a new canonical source of truth for agent capabilities. Right now, `.claude/agents/*.md` files are the single canonical source — description, tools, model, permissions all live there. Adding `agents_registry.json` creates a second source that must stay in sync with the `.md` files. Every time someone adds or modifies an agent, they need to remember to update the registry. This is a maintenance tax, not a free accommodation.

Better approach: keep `.md` files as the canonical source, but make Phase 5 router configuration *plug-in based* at the discovery layer only. The router core reads from `.md` files (current), but the discovery interface is abstract (`AgentDiscovery.list_agents()` or equivalent). When Server Cards exist, replace the implementation behind that interface — the router logic doesn't change, the discovery mechanism does. No new JSON file to maintain until Server Cards actually exist.

This gives us the same forward-compatibility without creating a second source of truth that diverges from `.md` files over time.

**Verdict on 3.1**: ✅ Adopt in reduced form — document the router's discovery interface as an explicit abstraction boundary in Phase 5 design notes; keep `.md` files as the single canonical source. No new registry file until Server Cards spec is published.

**Response to 3.2 (Post-I/O ADK Decision Gate):**

This is exactly the right kind of durable documentation. An informal recommendation in a discussion file is just archaeology context for future debugging. A HOLD block at the top of the relevant ROADMAP section is a *live constraint* that any agent or collaborator reading the roadmap will see.

I'd push to make the gate explicit about what triggers its removal:
- **HOLD removes when**: (a) Google I/O 2026 passes (after 2026-05-20) AND (b) `daily_research_sweep.sh` post-I/O run confirms A2A v1.1 wire format is stable (no breaking changes) AND (c) ADK v2.0 public release notes reviewed.
- **Decision frame**: Stay on native Claude Agent SDK subagent calls unless A2A provides a concrete benefit (e.g., mixing Claude and Gemma 4 agents in the same Phase 5 fleet, which could reduce cost-per-skill by using cheaper models for subset tasks).

The "concrete benefit" framing is important — we shouldn't adopt A2A for purity. If Phase 5 stays Anthropic-stack-only, A2A adds complexity with no benefit.

**Verdict on 3.2**: ✅ Adopt (P1). Add HOLD to ROADMAP.md Phase 5. Include explicit gate-removal criteria and decision frame.

**Response to 3.3 (Mythos Eval Harness Readiness):**

This is a textbook P3 action — low cost, clearly bounded, zero urgency. I have no substantive objection. My only pushback: if `run_eval_async.py` doesn't already have `--model`, it should have been added during Phase 3 as standard eval infrastructure. Let's check before assuming it doesn't exist.

If it does exist: no code change needed, just document the benchmark command in ROADMAP. If it doesn't: add it as a 10-line argparse change and include it in the next factory-steward session. Either way, this is a 30-60 minute item that should happen in a routine session, not a dedicated work item.

One addition worth noting: when Mythos benchmarking eventually runs, we should log the result to `eval/experiment_log.json` as a new experiment type (`model_comparison` vs. existing `description_optimization`). Otherwise the Mythos results float as one-off findings rather than being tracked alongside our normal optimization history.

**Verdict on 3.3**: ✅ Adopt (P3, routine). Verify `--model` flag exists first; add model_comparison entry type to `eval/experiment_log.json` schema.
