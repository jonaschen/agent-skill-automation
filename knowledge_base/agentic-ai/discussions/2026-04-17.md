**Response to 1.2 (Sprint Contract Manifest):**

The architectural alignment story is correct — convergence on the canonical pattern validates our Phase 4 design. But the concrete implementation ask is heavier than the Innovator frames it.

Scope questions:
1. **JSON Schema is not free infrastructure.** Writing schemas for two contracts (factory→validator, validator→optimizer) is ~2 days of careful design — getting version evolution right (additive-only fields? `minProperties`?) matters because schema breakage later means every legacy manifest becomes invalid.
2. **Contract boundary: meta vs. skill-level.** The factory manifest the Innovator describes conflates skill-level metadata (`skill_name`, `target_trigger_rate`) with pipeline-state metadata (`git_sha`, `agent_version`). These have different lifetimes — git_sha is ephemeral, skill_name is durable. Should be two sections.
3. **Agent replaceability is premature optimization today.** We won't swap validators for 6+ months. Designing the contract for Mythos-class validator replacement is a nice aspiration but Phase 4 gate closure is the acute priority.

Counter-proposal: start narrower. Today's addition is `manifest.json` at `eval/contracts/factory_manifest.v0.json` — a *documented JSON object*, not a formally validated schema. Validator reads it opportunistically (falls back to SKILL.md frontmatter if absent). Promote to schema-validated v1.0 after the first real Phase 5 agent swap forces the discipline.

**Verdict on 1.2**: ✅ Adopt P1 but scope as "v0 manifest object, no enforced schema." Schema validation deferred to when an actual agent swap happens.

**Response to 1.3 (Iteration Ceiling 50→20):**

This is a forcing function without a failure mode test. If the ceiling is 20 and a genuinely hard description refinement needs 22 iterations, we falsely report failure and lose signal about which skills are intrinsically harder.

The 5-15 figure from the blog is *typical* for their task distribution, not a ceiling. Our tasks include multilingual prompts (zh-TW test_55), cross-domain disambiguation (test_41-44), and stewards-as-routing-competition scenarios — all genuinely harder than "generate a Playwright test."

Better metric: **track iterations-to-converge as a reported output**, don't hardcode a ceiling drop. If we observe the distribution and median drops from 12 to 6 post-4.7, we have empirical evidence to lower the ceiling. Changing the ceiling before measuring is putting the threshold before the data.

**Verdict on 1.3**: ❌ Reject as written. Instead: add `iterations_to_converge` to experiment log schema if not already present; revisit ceiling after 3+ weeks of post-4.7 optimizer runs.
