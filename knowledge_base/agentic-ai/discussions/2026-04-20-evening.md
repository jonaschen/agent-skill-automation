## Round 1

### INNOVATOR

**Proposal 1.1: Dollar Ceiling Post-Migration Monitoring Plan**

The dollar ceiling ($10 max-budget-usd) was deployed today. Opus 4.7 has a confirmed ~37% token cost increase. If we migrate, some sessions may hit the ceiling prematurely. We don't know which session types are most affected because the increase depends on workload: reasoning-heavy tasks (factory code writing) will burn more adaptive thinking tokens than search-heavy tasks (researcher L1 sweeps).

I propose: after the shadow eval passes go/no-go and migration begins, add a 3-day monitoring phase. During this phase, `agent_review.sh` checks for `"exit_code": 1` entries that correlate with ceiling hits (the `--max-budget-usd` flag causes a non-zero exit when exceeded). If >2 ceiling hits occur in 3 days, raise the ceiling to $15 for affected session types. Log the decision in ROADMAP.md as a post-migration tuning item.

This is zero-cost until migration actually happens. It's a conditional plan, not infrastructure.

**Proposal 1.2: Fleet Manifest with A2A-Compatible Field Names**

Morning discussion adopted A5 (fleet manifest) but deferred full Agent Cards to Phase 5. The evening analysis notes A2A at 150+ organizations. If we use custom field names in fleet_manifest.json, we'll need to rename them later. If we use A2A Agent Card field names from the start (even for the subset we need), the Phase 5 upgrade is additive.

I propose: when the factory steward builds fleet_manifest.json, use A2A Agent Card field names for the overlapping fields: `name`, `description`, `version`, `capabilities` (mapped to our tools list), `provider` (mapped to model). Leave A2A-specific fields (`url`, `authentication`, `defaultInputModes`) absent — they'll be added in Phase 5. This costs nothing extra (same JSON, different key names) but aligns the upgrade path.

**Proposal 1.3: Monday Unfreeze Preparation Checklist**

The 96h+ Anthropic freeze and 3-5 day Google freeze are unusual. Monday is the most likely unfreeze day. When multiple releases land simultaneously, the researcher needs to prioritize efficiently. The current approach: scan all repos, note all changes, update all KB files. This works for 1-2 releases but becomes order-dependent with 5+.

I propose: the Monday morning sweep begins with a 5-minute pre-flight:
1. `gh api repos/anthropics/claude-code/releases` — count new releases since v2.1.114
2. `gh api repos/anthropics/claude-agent-sdk-python/releases` — count since v0.1.63

If total new releases > 3, set `FORCE_SWEEP=1`. This is not burst mode (rejected this morning) — it's using the existing mechanism with a pre-flight trigger.

## Round 2

### INNOVATOR

**Proposal 2.1: CheckPoint Hooks CVEs — Phase 5 Security Requirements Update**

The CheckPoint CVEs (CVE-2025-59536, CVE-2026-21852) are a distinct vulnerability class from the STDIO family. They exploit Claude Code project hooks — specifically, malicious `.claude/hooks/` files in cloned repositories can achieve RCE and API token exfiltration. Our exposure is low (trusted codebase, no untrusted repos), but Phase 5 introduces agents operating on external codebases.

I propose: add a line to the Phase 5 security requirements section in ROADMAP.md: "Hooks sanitization for external repositories — scan `.claude/` directory in any cloned/forked repo before agent execution (ref: CVE-2025-59536, CVE-2026-21852)." This is a 1-line note — not implementation, just requirements capture.

**Proposal 2.2: Experiment Log Health as Factory Pre-Flight**

Morning discussion adopted A3 (experiment log validator). The evening analysis notes a parse concern with experiment_log.json. The validator should run as part of the shadow eval script's pre-flight (already specified in morning discussion) AND as part of factory's gate-first check. If the log is corrupt, gate-first would silently find "no entries" and re-trigger the eval unnecessarily.

I propose: when the factory steward implements A3, wire it into both:
1. `daily_shadow_eval.sh` pre-flight (already planned)
2. `daily_factory_steward.sh` gate-first preamble (new — adds 1 line)

The gate-first check currently does: `grep "claude-opus-4-7" experiment_log.json`. If the JSON is malformed, grep finds nothing, concluding "no results" → triggers eval. With the validator, it would be: validate log first → if corrupt, alert and skip eval trigger → prevents cascading failures.

### ENGINEER

**On 2.1 (Hooks CVEs → Phase 5 Requirements):**

Correct. This is requirements capture, not implementation. One line in ROADMAP.md Phase 5 security section. Low cost, high value — we won't forget this when Phase 5 design begins.

**Verdict: ADOPT (P3).** Factory-steward adds one line to ROADMAP.md Phase 5.

**On 2.2 (Validator Wiring):**

The wiring into shadow_eval.sh was already specified in morning A3. Adding it to gate-first is the correct extension — the same data integrity concern applies to both consumers.

One concern: the gate-first preamble is a shell script snippet embedded in factory steward's session preamble. Adding a Python validator call there introduces a dependency. Simpler: make the Python validator exit 0/1, and gate-first calls `python3 eval/validate_experiment_log.py && grep "claude-opus-4-7" experiment_log.json`. If validation fails, the `&&` short-circuits and gate-first doesn't proceed.

**Verdict: ADOPT (P2) — amend A3 to wire into both shadow eval and gate-first.** Clean `&&` chaining.
