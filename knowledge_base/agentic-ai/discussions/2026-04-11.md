## Round 1

### INNOVATOR

**Proposal 1.1: Advisor Tool as Fourth Topology Option for Phase 5**

The Advisor Tool is the most significant cost-quality primitive since Agent Skills. The analysis shows Sonnet+Opus advisor achieves +2.7pp SWE-bench at -11.9% cost per agentic task. Today our Phase 5 topology-aware-router (`topology-aware-router.md`) routes to three tracks: Track A (parallel), Track B (sequential), Changeling (identity switch). The advisor pattern fills a gap in the medium-coupling band (TCI 0.35-0.65) where Track B is overkill and Changeling lacks the quality ceiling.

Concrete change:
- Update `.claude/agents/topology-aware-router.md` to add a fourth routing option: "Advisor Pattern" for TCI 0.35-0.65 where a Sonnet executor consults Opus at 2-3 key decision points (approach planning, stuck recovery, completion verification)
- Add a row to the routing decision table in the agent definition
- Add the advisor-pattern topology to `eval/tci_compute.py` output recommendations

This doesn't require implementation now — it's a design addition to a Phase 5 agent definition that already exists as a stub.

**Proposal 1.2: Pre-Execution Bash Metacharacter Blocking (ADK-Inspired Security Hardening)**

Our `post-tool-use.sh` monitors Bash commands *after* execution — it counts command chain length and MCP call depth, but a malicious command has already run by the time the hook fires. ADK v1.29.0's BashTool blocks shell metacharacters *before* execution. Our Phase 5.5 plans a PreToolUse hook, but we can act now by adding a regex check to `post-tool-use.sh` that logs metacharacter patterns as a detection layer (`;`, `|`, `&&`, `||`, backtick substitution, `$()`) — not blocking today, but building the detection baseline for the PreToolUse migration.

Concrete change:
- Add metacharacter pattern detection to `scripts/cmd_chain_monitor.sh` — new function `detect_metachar_patterns()` that logs structured alerts to `logs/security/metachar_alert.jsonl`
- Mode: **detect-only** (advisory), not blocking — avoids breaking legitimate piped commands (`grep | head`)
- Build a 30-day baseline of metacharacter usage across all 8 daily agents to calibrate the Phase 5.5 PreToolUse allowlist

**Proposal 1.3: Fleet Version Bump to v2.1.98 + Minimum Version Update**

The analysis elevates the fleet version gap from P1 WARNING to P0 ACTIVE SECURITY RISK — 11 versions behind, missing PID namespace isolation, MCP memory leak fix, permission bypass corrections, compound command hardening. The fleet is on v2.1.87.

Concrete change:
- Human action: `npm update -g @anthropic-ai/claude-code` on the fleet host to get >=v2.1.98
- Factory steward action: Update `scripts/lib/fleet_min_version.txt` from `>=2.1.97` to `>=2.1.98`
- Verify PID namespace isolation is active in the next nightly run by checking `/proc/self/ns/pid` in a test Bash call

### ENGINEER

**Response to 1.1 (Advisor Tool as Fourth Topology):**

The topology gap analysis is solid — the medium-coupling band is genuinely underserved. But let me stress-test the proposal:

1. **The advisor tool is public beta** (`advisor_20260301`). Beta APIs can change. Recording it as a design option is fine; hardcoding routing decisions around it is premature.
2. **TCI band overlap**: The current Track B (TCI > 0.65) and Changeling (single-domain) are clean separations. Adding "advisor for 0.35-0.65" creates a three-way overlap since Track B already covers everything above 0.35 as a conservative default. We'd need to tighten Track B to > 0.65 only, which changes existing routing behavior.
3. **Cost savings are marginal for our fleet**: 11.9% per iteration on a pipeline that runs 8 times/day is pocket change vs. the complexity of a fourth routing path.

That said, recording it as a design note in the Phase 5.2 section costs nothing and ensures we don't forget it when Phase 5 implementation starts. The advisor pattern's real value isn't cost savings — it's quality amplification for medium-complexity tasks without the overhead of a full planner session.

**Verdict: ADOPT (P2)** — Add advisor-pattern as a fourth topology option in the Phase 5.2 design notes in ROADMAP.md. Note it as beta, don't hardcode routing ranges yet. Re-evaluate when advisor reaches GA and Phase 5 implementation begins.

**Response to 1.2 (Metacharacter Detection):**

I like the detect-only approach — it avoids the blast radius of accidentally blocking legitimate pipes while building the data we need for Phase 5.5. But:

1. **False positive flood risk**: Our daily agents use pipes extensively (`grep | head`, `git log --format | sort`, `jq | wc -l`). A naive metacharacter regex will fire constantly, drowning real alerts in noise.
2. **The right data to collect isn't "all metacharacter usage"** — it's "metacharacter patterns that don't match known-safe commands." The allowlist should be built into the detector from day one, seeded from a snapshot of the current scripts' piped commands.
3. **Effort**: This is ~2 hours of work for a detection layer that produces data we won't act on until Phase 5.5 (months away). The existing `cmd_chain_monitor.sh` already catches the most dangerous pattern (long command chains as denial-of-service).

The 30-day baseline idea is good. But start with a more focused detector: log only metacharacters in *user-provided* arguments (tool inputs from Claude), not in our own scripts. The daily scripts control their own commands; the threat model is Claude generating dangerous shell metacharacters in tool calls.

**Verdict: ADOPT (P2)** — Add metacharacter pattern detection to `cmd_chain_monitor.sh`, detect-only mode, with an initial allowlist seeded from known-safe patterns. Scope: Claude-generated Bash tool inputs only (not our scripts). Start the 30-day baseline.

**Response to 1.3 (Fleet Version Bump):**

No debate here. This is a P0 security action. The fleet is 11 versions behind with critical fixes (PID namespace isolation, MCP memory leak at 50 MB/hr that could OOM a long research sweep, permission bypass vectors). The `check_fleet_version.sh` infrastructure already exists — just needs the version file updated after the human performs the upgrade.

One nuance: verify the upgrade doesn't break any agent behavior. Run `scripts/agent_review.sh 1` on the first post-upgrade nightly cycle and compare durations/exit codes against the 7-day baseline.

**Verdict: ADOPT (P0)** — Human upgrades fleet to >=v2.1.98. Factory steward updates `fleet_min_version.txt`. Post-upgrade: monitor first nightly cycle via `agent_review.sh`.

**Proposal 2.1: Advisor Pattern for Phase 3.5 Distillation (Cost-Quality Stepping Stone)**

Phase 3.5 (heterogeneous model distillation) plans to go from "all Opus" → "all Haiku" for optimization iterations. The advisor tool creates a *middle step*: Sonnet+Opus advisor. Instead of a cliff from Opus to Haiku, we get a graceful degradation ladder:

```
Opus (current) → Sonnet + Opus advisor (-11.9% cost, ~equal quality)
              → Haiku + Opus advisor (-85% cost, 2x BrowseComp)
              → Haiku standalone (target: ≥90% of Opus baseline)
```

Each step is independently measurable with our existing Bayesian eval infrastructure. We can stop at whichever tier meets the 90% quality threshold at the lowest cost.

Concrete change:
- Add a note to ROADMAP Phase 3.5 describing the three-tier distillation ladder
- When Phase 3.5 begins, the first experiment should be Sonnet+advisor vs. pure Opus on the existing 36-prompt Training set

**Proposal 2.2: Thinking Configuration Audit for Fleet Baseline Integrity**

The analysis flags a silent bug in Agent SDK where `thinking: {type: "adaptive"}` mapped to `--max-thinking-tokens 32000` instead of `--thinking adaptive`. While our agents run via CLI, not SDK, the concern is valid: if Claude Code CLI had the same mapping issue, our effort monitoring baseline (started Apr 9) could be invalid.

Concrete change:
- Add a diagnostic step to `scripts/lib/check_fleet_version.sh`: after the version check, run `claude --version` and `claude config list 2>/dev/null | grep thinking` (or equivalent) to log the active thinking configuration
- Record the thinking config in each performance JSON as `thinking_mode` field alongside `effort_level`
- If the CLI thinking behavior changed between v2.1.87 and v2.1.98, flag it in the post-upgrade `agent_review.sh` output

**Proposal 2.3: Workflow Graph Visualization Reference for Phase 5 Observability**

ADK v2.0.0-alpha.3 introduces workflow graph visualization with active node rendering and lazy scan dedup on resume. Our Phase 5 has `health_dashboard.py` for agent-level health but nothing for task-level workflow state. The ADK web UI pattern (and Vercel WDK's deterministic replay model) provide concrete references.

Concrete change:

- Record the three-way convergence pattern (ADK lazy scan dedup + WDK deterministic replay + our state machine skip) as a knowledge base note in `knowledge_base/agentic-ai/evaluations/`

### ENGINEER

**Response to 2.1 (Distillation Ladder):**

The three-tier ladder is elegant and maps perfectly to our eval infrastructure — each tier is just a model config change, not an architecture change. The Bayesian eval can compare Sonnet+advisor vs Opus exactly the same way it compares any two runs.

However:
1. **Phase 3.5 isn't started** and has no timeline. Adding detailed design notes for an unscheduled phase creates documentation debt that goes stale.
2. **The advisor API is beta**. If Anthropic changes the billing model or quality profile before Phase 3.5 starts, the ladder needs redesigning.
3. **The simpler first experiment** is pure Sonnet vs. Opus on the 36-prompt Training set. If Sonnet alone matches (it might — our prompts aren't that complex), the advisor is unnecessary overhead.

The design note approach is fine — one line in Phase 3.5 saying "evaluate advisor-pattern as intermediate tier between Opus and Haiku." Don't over-specify the ladder until Phase 3.5 actually starts.

**Verdict: ADOPT (P3)** — Add one-line note to ROADMAP Phase 3.5: "Evaluate Sonnet+Opus advisor as intermediate distillation tier." No detailed ladder design until Phase 3.5 is scheduled.

**Response to 2.2 (Thinking Configuration Audit):**

This is good operational hygiene but the blast radius of the bug is limited:

1. **Claude Code CLI uses `--thinking` flag directly**, not the SDK's `thinking: {type: "adaptive"}` configuration object. The SDK mapping bug wouldn't affect CLI behavior.
2. **Our agents don't specify thinking config** — they inherit whatever Claude Code's default is. The default hasn't changed across the v2.1.87→v2.1.98 range (it's been "enabled" since 2.x).
3. **Adding `thinking_mode` to perf JSON** is ~5 lines per daily script. Low effort, useful for debugging future effort/duration anomalies.

The version check enhancement is premature — `claude config list` doesn't exist as a CLI command. Just capture `claude --version` (already done by `check_fleet_version.sh`) and add `thinking_mode: "default"` to perf JSONs.

**Verdict: ADOPT (P2)** — Add `thinking_mode` field to performance JSONs in all 6 daily scripts. Record "default" for now. If we ever set explicit thinking config, this field captures the change. Skip the CLI config inspection — it's not a real command.

**Response to 2.3 (Workflow Visualization):**

The three-way convergence (ADK/WDK/our state machine) is a genuine pattern worth documenting. But the implementation proposal needs scoping:

1. **A real-time graph UI is a substantial frontend project** — React component, WebSocket updates, layout engine. This is months of work if built properly, and our pipeline is CLI-native with no web UI.
2. **The right Phase 5 deliverable isn't a visual UI** — it's structured JSON state that *could* be visualized. The `health_dashboard.py` already outputs JSON. Extending it with task-level step state (pending/running/completed/failed) is the right abstraction. A web UI is Phase 5.5 or Phase 7 chrome.
3. **The knowledge base note** about the convergence pattern is valuable regardless. Write it.

**Verdict: ADOPT (P2)** — Add "5.3.2 Task-level workflow state tracking (JSON)" to ROADMAP Phase 5.3. Note ADK and WDK as references. Write the convergence pattern to knowledge base. Explicitly exclude a web UI from Phase 5 scope — that's Phase 7 observability.

## Round 3

### INNOVATOR

**Proposal 3.1: Cross-User Prompt Caching Architecture Pattern for Phase 5**

Agent SDK v0.1.57 introduces `exclude_dynamic_sections` on `SystemPromptPreset` — moves per-user context out of system prompt for cross-user cache hits. Our fleet shares CLAUDE.md (~4K tokens) across all 6 steward agents. When we migrate to Agent SDK in Phase 5, separating static instructions from dynamic context (git status, memory, date) would yield cache hits across agents.

The savings for 8 sessions/day are small (~$0.05-0.10/day), but the *architectural pattern* — separating static system prompt from dynamic context — is a clean design principle that improves Phase 5 multi-agent sessions where multiple agents share base instructions within the same task.

Concrete change:
- Add a design note to Phase 5 ROADMAP: "When adopting Agent SDK, separate CLAUDE.md into static base (cacheable) and dynamic session context (per-agent). Reference: Agent SDK `exclude_dynamic_sections` pattern."

**Proposal 3.2: `usage.iterations[]` Billing Model as Phase 7 Cost Attribution Reference**

The advisor tool response includes `usage.iterations[]` — per-component billing with separate token counts for executor and advisor. This is exactly the decomposition our Phase 7 outcome-billing-engine needs: showing customers "planning: $0.04, generation: $0.12, review: $0.02" instead of a flat token total.

Concrete change:
- Record the `usage.iterations[]` schema in `knowledge_base/agentic-ai/anthropic/tool-use-and-function-calling.md` under a new "Advisor Tool Billing" section
- Add a note to ROADMAP Phase 7.1: "Reference Anthropic's `usage.iterations[]` billing decomposition for per-component cost attribution"

**Proposal 3.3: 1M Context Beta Sunset Contingency (19 Days)**

### ENGINEER

**Response to 3.1 (Prompt Caching Architecture):**

This is a pure design note — no implementation, no blast radius. The pattern is sound: separating static from dynamic context is good software design regardless of caching benefits. And the analysis correctly identifies that the real payoff is intra-task cache hits in Phase 5 multi-agent sessions, not our current 8-session fleet.

One concern: our CLAUDE.md isn't purely static — it references current phase status, completion percentages, and version numbers that change weekly. A true "static base" would need a structural refactor of CLAUDE.md itself (split into `CLAUDE_BASE.md` + `CLAUDE_STATUS.md`). That's not needed for a design note, but flag it as an implementation consideration.

**Verdict: ADOPT (P3)** — Add one-line Phase 5 design note about `exclude_dynamic_sections` pattern. No CLAUDE.md refactoring now.

**Response to 3.2 (Billing Model Reference):**

Straightforward knowledge capture. The `usage.iterations[]` schema is the first real-world example of per-component billing decomposition in a major API. Recording it now means Phase 7 doesn't start from scratch.

Two caveats:
1. Don't over-engineer the knowledge base entry — the advisor API is beta and the billing schema could change. Record the current schema and mark it as "beta, verify at Phase 7 start."
2. The ROADMAP Phase 7.1 note should reference the knowledge base file, not duplicate the schema inline.

**Verdict: ADOPT (P3)** — Record `usage.iterations[]` in knowledge base. Add one-line ROADMAP reference.

**Response to 3.3 (1M Context Sunset):**

The contingency planning is sensible but the threat is low:

1. **Our agents don't depend on 1M context.** 200K is the normal ceiling. The research sweep processes 14 files × ~1KB each = ~14KB of topic content. Even with CLAUDE.md + agent definition + analysis template, that's well under 200K.
2. **Anthropic is unlikely to reduce context limits** — they're a competitive differentiator. The beta sunset likely means pricing changes (1M becomes a paid tier), not removal.
3. **The `max_context_tokens` field** is useful for operational visibility regardless of the sunset. But it requires instrumenting the performance JSON collection, which currently doesn't have access to context window usage (Claude Code doesn't expose this metric).

The sweep segmentation contingency (7 topics per session) is the right backup plan, but implementing it pre-emptively wastes effort. Wait until we see the post-beta terms.

**Verdict: DEFER** — Monitor Anthropic's post-beta announcement (expected late April). Don't implement segmentation or add perf JSON fields until we know the actual impact. If context_tokens becomes available via Claude Code API, add it then.
