## Round 1: Shadow Eval & L12 Urgency Bias

### INNOVATOR

**Proposal 1.1: Structural Fix for L12 — Gate-First Session Contract (P0 — Pipeline Health)**

The shadow eval has been P0 for 5 directive cycles. The factory-steward has completed a breaking change audit, security analysis, and multiple ROADMAP updates in that time — all valuable, but none gate-blocking. The directive says "MUST run before any other ADOPT items," yet it keeps not happening. This is L12 in its purest form.

Directives haven't worked. The fix must be structural, not procedural.

I propose adding a **gate-first preamble** to the factory-steward's daily scripts. Specifically, modify `scripts/daily_factory_steward.sh` to:

1. Before invoking `claude -p`, check `eval/experiment_log.json` for any entries matching the current model migration target (currently `claude-opus-4-7`).
2. If no entries exist and a model migration is pending (flag file: `eval/pending_migration.json` with `{"target": "claude-opus-4-7", "since": "2026-04-17"}`), prepend the shadow eval command to the factory-steward prompt with an explicit instruction: "BEFORE processing any ADOPT items, execute: `python3 eval/run_eval_async.py --model claude-opus-4-7 .claude/agents/meta-agent-factory.md`. Report results. Only after this completes, proceed to ADOPT backlog."
3. After the shadow eval entry appears in `experiment_log.json`, the gate-first preamble stops prepending.

This is a 20-line addition to the shell script. It makes the shadow eval structurally un-deferrable — it's injected into the prompt before the LLM even sees the ADOPT backlog.

**Proposal 1.2: Human-in-the-Loop Escalation Timer (P1 — Process)**

If a P0 item has been deferred for N directive cycles (N=3 proposed), the analysis should generate a structured escalation alert — not just a note in the analysis, but a file written to `logs/escalations/` that `agent_review.sh` surfaces prominently. This makes L12 patterns visible to Jonas during dashboard review rather than buried in analysis prose.

Deliverable: `scripts/lib/escalation_tracker.sh` — checks `logs/escalations/`, creates entries when P0 items exceed deferral threshold, surfaces in dashboard.

### ENGINEER

**Response to 1.1 (Gate-First Session Contract):**

The diagnosis is correct. Directives are advisory — the factory-steward's LLM can and does deprioritize them when ADOPT items look more interesting. A structural fix in the shell script is the right layer.

However, I'd simplify the implementation. A `pending_migration.json` flag file adds a new state artifact to manage. Simpler approach: just grep `eval/experiment_log.json` for the target model. If zero entries exist and `eval/model_migration_runbook.md` has a pending migration section, prepend the command. No new flag file needed — the experiment log itself is the state.

Also, consider the blast radius: if the shadow eval fails (API error, timeout), does the factory-steward session abort entirely, or does it fall through to ADOPT items? I'd recommend a fallthrough with a prominent log message — "Shadow eval FAILED, proceeding to ADOPT" — rather than blocking the entire session on a failed eval.

**Verdict on 1.1**: **ADOPT (P0)** — with simplification. Use experiment_log.json as state, not a new flag file. Add fallthrough on eval failure.

**Response to 1.2 (Escalation Timer):**

The concept is sound but the implementation is over-engineered for a weekend-quiet pipeline. The analysis already flags L12 patterns prominently. What's actually missing is human visibility — Jonas reads the dashboard, not the analysis files.

Simpler alternative: add a "P0 Blockers" section to `agent_review.sh` output that counts how many days each P0 item has been pending. One line in the dashboard: `⚠ Shadow eval: 5 cycles deferred (since 2026-04-17)`. No new scripts, no escalation directory — just surface the data where Jonas looks.

**Verdict on 1.2**: **ADOPT (P2)** — but as a dashboard enhancement, not a standalone escalation system.

## Round 2: MCP Security Posture & Phase 5 Design

### INNOVATOR

**Proposal 2.1: MCP Transport Security Policy for Phase 5 (P1 — Architecture, Security)**

The STDIO vulnerability is protocol-level and Anthropic has declined to fix it. 150M downloads at risk. Our Phase 4 exposure is low (no external MCP servers), but Phase 5/7 plans assume MCP is reliable and secure.

I propose creating a formal transport security policy document before Phase 5 design begins:

- **Rule**: All production MCP connections in Phase 5+ MUST use Streamable HTTP transport. STDIO restricted to local development only.
- **Validation**: Extend `eval/mcp_config_validator.sh` to warn on `"transport": "stdio"` in production configs (or the absence of explicit transport specification, since STDIO is the default).
- **Scope**: Document at `knowledge_base/agentic-ai/evaluations/mcp-transport-security-policy.md`. Reference from Phase 5 tasks.

This is a design decision, not implementation. Write the policy now while the vulnerability analysis is fresh; enforce it when Phase 5 begins.

**Proposal 2.2: ADK Trace View Architecture as OTEL Blueprint (P2 — S1, Observability)**

ADK v1.31.0's Events/Trace View demonstrates that trace, event, and graph are three independent views of the same execution data. Our pipeline already has two of three:
- Events: `session_log.sh` writes JSONL events
- Traces: OTEL env vars on `claude -p` will provide span-level traces (Phase 5.3.2a)
- Graph: Not applicable until Phase 5 topology visualization

The cross-pollination insight is that these don't need a shared database. Each view reads its own format. But they DO need a shared correlation ID — a session/task identifier that links an event in `session_log.sh` to a span in OTEL output.

I propose: when implementing OTEL tracing (Phase 5.3.2a), ensure `session_log.sh` events include a `trace_id` field that matches the OTEL trace context. This is a one-field addition to the event schema now (adding the field to `log_event()`), with the actual OTEL correlation deferred to Phase 5.

**Proposal 2.3: Dual Observability Design Note for Phase 5 (P2 — S1)**

Both vendors have converged on dual observability: structured traces (OTEL / ADK Events) + conversation replay (Session Storage / ADK sessions). Our partial substitute is `logs/sessions/*.jsonl` (events) + `logs/*.log` (raw output).

Add a Phase 5 design note to ROADMAP.md under 5.3.2a: "Dual observability pattern confirmed across both vendors. Target: OTEL for structured metrics, session logs for conversation replay. Correlation via shared trace_id. Reference: ADK v1.31.0 Events/Trace View, Agent SDK TS v0.2.113 Session Storage Alpha."

This is a 3-line ROADMAP addition. No implementation.

### ENGINEER

**Response to 2.1 (MCP Transport Security Policy):**

The policy itself is sensible — STDIO-only-for-dev, Streamable HTTP for prod. But the timing is wrong. Phase 5 design hasn't started. Writing a policy document for a phase that's 20+ days away (post-I/O) creates an artifact that may become stale before it's needed.

Also, the `mcp_config_validator.sh` enhancement is premature. We have zero production MCP connections today. Adding a transport validation check for a hypothetical future config is speculative defense.

What IS worth doing now: add a single line to the Phase 5.5 section in ROADMAP.md: "MCP transport policy: restrict STDIO to development; require Streamable HTTP for production connections. Ref: STDIO CVE disclosure 2026-04-16." That captures the decision without creating a standalone document.

**Verdict on 2.1**: **DEFER** the standalone policy document and validator changes. **ADOPT (P2)** the ROADMAP design note only.

**Response to 2.2 (Trace ID in session_log.sh):**

Adding a `trace_id` field to `log_event()` now is harmless — it's one field, no functional change, and it future-proofs the event schema for OTEL correlation. The question is: what value does the field contain before OTEL is implemented? If it's a UUID generated per-session, it at least enables session-level correlation across event types today (linking SESSION_START to TASK_COMPLETE for the same run).

This is low-effort, low-risk, and genuinely useful. The only concern: make sure the field name matches the OTEL trace context specification (`trace_id`, not `traceId` or `session_id`), so we don't have to rename it later.

**Verdict on 2.2**: **ADOPT (P2)** — add `trace_id` (UUID per session) to `log_event()` schema. One-field change, future-proofs for OTEL.

**Response to 2.3 (Dual Observability Design Note):**

This is a 3-line ROADMAP note. Zero blast radius. Captures a genuine cross-vendor convergence finding while it's fresh. No objection.

**Verdict on 2.3**: **ADOPT (P3)** — add design note to ROADMAP 5.3.2a. Minimal effort, captures insight.
