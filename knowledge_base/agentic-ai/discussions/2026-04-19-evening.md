## Round 2: MCP Security Escalation & cmd_chain_monitor Gap

### INNOVATOR

**Proposal 2.1: Document cmd_chain_monitor.sh Argument-Blind Gap for Phase 5 (P1 — Security)**

CVE-2026-40933 (Flowise) reveals a pattern directly relevant to our security stack: command allowlists that check binary names but not arguments can be bypassed trivially. The Flowise attack used `npx -c "malicious code"` — `npx` is an allowed binary, but `-c` enables arbitrary code execution.

Our `cmd_chain_monitor.sh` uses a 60+ binary allowlist in detect-only mode. Today, this is fine — it's informational, not blocking. But Phase 5.5 plans to enable blocking mode (`METACHAR_MODE=block`). If we enable blocking with the current argument-blind allowlist, we'd have the exact same vulnerability class as Flowise.

I propose adding a design note to the Phase 5.5 section of ROADMAP.md:

> **Design note (2026-04-19)**: CVE-2026-40933 demonstrates that binary-name-only allowlists are insufficient — argument injection (`npx -c`, `node -e`, `python3 -c`) bypasses the allowlist while executing arbitrary code. Phase 5.5 `PreToolUse` blocking mode MUST include argument-pattern validation alongside binary allowlisting. Reference: Flowise bypass of `validateCommandInjection`.

This is a ROADMAP note — 3 lines. No code changes. It ensures the Phase 5 designer doesn't repeat Flowise's mistake.
