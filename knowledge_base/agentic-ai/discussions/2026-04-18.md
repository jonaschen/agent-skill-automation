## Round 1: Opus 4.7 Migration Hardening and Task Budget Controls

### INNOVATOR

**Proposal 1.1: Three-Step Opus 4.7 Breaking Change Audit + Shadow Eval (P0)**

Yesterday's discussion established the shadow eval plan; today's analysis crystallizes the full breaking change surface. The analysis identifies five specific breaking changes, three of which have non-zero pipeline exposure. We should execute the audit *before* the shadow eval — a 400 error from a stray `budget_tokens` parameter would corrupt eval results and waste the run.

Concrete sequence:
1. **Grep audit** (10 min): Search all scripts, agent configs, and eval infrastructure for `temperature`, `top_p`, `top_k`, `budget_tokens`. The analysis rates our exposure as Low/Medium, but "grep needed to confirm" means nobody has actually checked. Run `grep -rn 'temperature\|top_p\|top_k\|budget_tokens' scripts/ eval/ .claude/` and fix any hits.
2. **Thinking mode migration**: Any agent config using `{type: "enabled", budget_tokens: N}` must change to `{type: "adaptive"}`. Our `thinking_mode` field in perf JSONs records `"default"` — but verify no daily script passes explicit thinking configuration to `claude -p`.
3. **Shadow eval on T=39** (15 min): `python3 eval/run_eval_async.py --model claude-opus-4-7 .claude/agents/meta-agent-factory.md`. Compare via `bayesian_eval.py --compare` against T=0.829 baseline (re-baselined on T=39 per ROADMAP §4.2a).
4. **Cost baseline marker**: Before fleet rollout, snapshot current 7-day token costs as the pre-4.7 baseline. After rollout, the new tokenizer's 1.0-1.35× inflation will silently drift the 30-day rolling average in `cost_ceiling.sh`. Without a pre-4.7 marker, we can't distinguish tokenizer inflation from actual cost increase.

Files to change: None for audit (grep only). For cost baseline: add a `pre_47_cost_snapshot` field to `logs/performance/` JSON schema.

**Proposal 1.2: Task Budgets for Steward Cost Control (P1)**

Task Budgets (`task-budgets-2026-03-13`, public beta, Opus 4.7 only) provide model-level token ceilings — the model sees a running countdown and self-regulates reasoning depth. Our current `cost_ceiling.sh` is duration-based (wall-clock × 5x rolling avg), which is a blunt proxy for actual token consumption. A fast run that generates excessive tokens (e.g., steward spawns unnecessary subagents) isn't caught.

Concrete implementation:
1. **Verify CLI availability**: Check if `claude -p --task-budget <tokens>` is supported (the analysis notes this is uncertain — may be API-only).
2. **If CLI-available**: Add `--task-budget 400000` to `daily_factory_steward.sh` (factory is most predictable: median ~180K tokens based on session durations). Set at 2.2× observed median as safety margin.
3. **Layer with existing ceiling**: Task Budget (token-level) + duration ceiling (wall-clock) = defense-in-depth. One catches token-heavy fast runs; the other catches wall-clock-heavy slow runs.
4. **Monitor for truncation artifacts**: Add a `task_budget_exhausted` boolean field to perf JSONs. If the steward ends mid-task because budget depleted, that's worse than no budget at all.

Files to change: `scripts/daily_factory_steward.sh`, performance JSON schema.

**Proposal 1.3: Delegation Regression Monitor for 4.7 Rollout (P2)**

The analysis flags Opus 4.7's "fewer subagents spawned by default" as a HIGH exposure risk for our steward agents. Our L13 lesson already mandates explicit agent naming ("Use the meta-agent-factory agent to..."), which partially mitigates. But we have no empirical data on 4.7's delegation behavior with our prompts.

Concrete implementation:
1. **Baseline delegation count**: Before 4.7 rollout, parse the last 5 factory-steward session logs (`logs/sessions/factory-*.jsonl`) and count `TASK_START` events where the task description contains "meta-agent-factory", "skill-quality-validator", or "autoresearch-optimizer". This is the 4.6 delegation baseline.
2. **Post-rollout comparison**: After 3 factory-steward runs on 4.7, compare delegation counts. Alert if delegation drops >30%.
3. **Automated check**: Add a `delegation_count` field to factory-steward's perf JSON. `agent_review.sh` already shows duration trends — add delegation trends.

Files to change: `scripts/daily_factory_steward.sh` (count extraction), `scripts/agent_review.sh` (display).

### ENGINEER

**Response to 1.1 (Breaking Change Audit + Shadow Eval):**

Fully agree on sequencing the grep audit before the shadow eval — a 400 error mid-eval would produce FAIL results that look like routing regression, wasting hours of root-cause diagnosis (echoing L10's lesson about measurement failures masquerading as routing failures).

The audit itself is trivial and should have been done yesterday. No objection.

On the cost baseline marker: adding a field to the perf JSON schema is low-cost but I question whether it earns its complexity. The 30-day rolling average in `cost_ceiling.sh` uses a 5× multiplier — even at 1.35× tokenizer inflation, the ceiling rises from 5× to 6.75×, still well within the existing safety margin. The real risk isn't false cost alerts; it's that the cost ceiling becomes permanently inflated and loses its signal value over 30+ days. A simpler approach: after 4.7 adoption, reset the rolling average window to 7 days for the first month, then expand back to 30. This handles tokenizer drift without schema changes.

**Verdict on 1.1**: ADOPT with Engineer's rolling-average-reset alternative instead of the perf JSON schema change. The grep audit and shadow eval sequence are non-negotiable P0.

**Response to 1.2 (Task Budgets):**

The concept is sound but the implementation is speculative on a critical unknown: CLI availability. The analysis explicitly flags this: "If available via `claude -p` CLI (verify)." If it's API-only, the entire proposal is moot for Phase 4.

Even if CLI-available, I have concerns about the 400K budget. The analysis recommends 2× observed median — you're proposing 2.2× at 400K. Where does the 180K median estimate come from? Our perf JSONs track `duration_seconds`, not tokens. We're estimating tokens from duration, which is exactly the kind of imprecise proxy we're trying to replace. The first implementation should be observational: set the budget high enough that it never triggers (e.g., 1M), capture the actual token consumption in perf JSONs, *then* set a meaningful ceiling after 2 weeks of data.

Also: Task Budgets are Opus 4.7-only. This proposal depends on Proposal 1.1 completing first.

**Verdict on 1.2**: DEFER until (a) CLI availability confirmed, (b) Opus 4.7 fleet rollout complete from 1.1. If CLI-available, start with observational mode (high ceiling, capture actual consumption) before setting a meaningful budget.

**Response to 1.3 (Delegation Regression Monitor):**

The risk is real but the proposed monitoring is manual and fragile. Parsing session JSONL logs for task descriptions containing agent names is a string-matching heuristic — if the steward phrases the delegation differently ("invoke meta-agent-factory" vs. "use the meta-agent-factory agent"), the count changes without the behavior changing.

A simpler proxy: compare session duration and commit count. If 4.7's steward takes 2× longer with the same number of commits, it's probably doing work itself instead of delegating. This is already tracked in perf JSONs — no new fields needed.

If delegation really drops >30%, the fix is already documented in the analysis: "add stronger delegation prompting to steward skill." That's a one-line description change, not an infrastructure project.

**Verdict on 1.3**: ADOPT in simplified form — use existing perf JSON fields (duration, commits) as delegation proxy during 4.7 rollout. No new fields or scripts needed. Escalate to explicit delegation counting only if the proxy signals a problem.

## Round 2: Phase 5 Architecture Inputs and MCP Security Posture

### INNOVATOR

**Proposal 2.1: Add OTEL Tracing to Phase 5 Requirements as First-Class Architecture Decision (P1)**

Agent SDK v0.1.60's W3C distributed tracing (`claude-agent-sdk[otel]`) is exactly the observability foundation Phase 5 needs. Our current file-based logging (perf JSONs, session JSONL, health dashboard) works for Phase 4's sequential single-agent execution. Phase 5's parallel multi-agent topology will produce interleaved traces that file-based logging cannot correlate.

This isn't a "nice to have" — it's an architectural decision that shapes how we design Phase 5's topology-aware-router. If we commit to OTEL, the router emits trace context that propagates to all spawned agents. If we don't, we build another bespoke logging layer that we'll replace later.

Concrete changes:
1. Add `claude-agent-sdk[otel]` to ROADMAP §5.3.2 requirements.
2. Add a new Phase 5 task: "Implement OTEL collector for multi-agent pipeline traces (stdout JSON format initially, Jaeger/Tempo optional)."
3. Design principle: Phase 5 observability must be OTEL-native (not vendor-specific), so it works with both Agent SDK traces and any future ADK OTEL support. This is a cross-pollination opportunity from today's analysis §2.2.

Files to change: `ROADMAP.md` §5.3.2.

**Proposal 2.2: Four-Topology TCI Dispatch (Align with Anthropic's Official Taxonomy) (P1)**

Anthropic now officially documents four multi-agent patterns: Teams, Subagents, Three-Agent Harness, Orchestrator-Worker. Our Phase 5 TCI router currently routes to two tracks (parallel Track A, flagship Track B). Today's analysis §2.5 maps these to four dispatch targets based on TCI score.

The proposal: update `topology-aware-router.md` and ROADMAP §5.2 to reference the four-topology model as the Phase 5 design target. Not implement it now — just align the design vocabulary so Phase 5 starts from the right architecture.

| TCI Score | Anthropic Pattern | Our Dispatch |
|-----------|------------------|-------------|
| ≤ 2 | Subagents | Single agent with inline delegation |
| 3-4 | Teams | Parallel Track A (independent specialists) |
| 5-6 | Three-Agent Harness | Sequential pipeline (plan→generate→evaluate) |
| ≥ 7 | Orchestrator-Worker | Track B flagship with synchronous cohort |

Concrete changes:
1. Add a design note to ROADMAP §5.2 referencing the four-topology model.
2. Update `topology-aware-router.md` description to mention four dispatch targets.
3. Keep TCI thresholds provisional — calibrate against actual task distribution in Phase 5.

**Proposal 2.3: Elevate MCP Security Audit from "Deferred" to "Pre-Phase-5" (P2)**

MCP ecosystem doubled to 10K+ servers in ~2 weeks. Our `mcp_config_validator.sh` does static pinning and content scanning for known servers. Phase 5's multi-agent topology will likely need to discover and validate MCP servers dynamically — especially if we adopt Server Cards (MCP v2.1 spec).

The deferred `mcp-sec-audit` evaluation (ROADMAP §4.4, P2) should become a pre-Phase-5 gate. Not blocking Phase 4 — but completed before Phase 5 design sprint begins.

Concrete change: Move the `mcp-sec-audit` task from "deferred P2" to "pre-Phase-5 gate" in ROADMAP §4.4.

I'd support elevating from "deferred" to "Phase 5 planning period" (do it during the Phase 5 design sprint, not as a gate for starting Phase 5). This preserves the urgency signal without creating a blocking dependency.

When migrating to Agent SDK sessions, evaluate whether `get_subagent_messages()` + session persistence enables rewind-to-checkpoint behavior for the optimizer's parallel branch search."
