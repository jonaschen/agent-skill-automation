## Round 1

### INNOVATOR

**Proposal 1.1: External Session State for Steward Agents (Brain-Hands Lite)**

Today's Managed Agents deep dive reveals the most impactful pattern for our pipeline: stateless harnesses that read state from an append-only session log. Our steward agents are the opposite — they dump everything into conversation context, rely on Claude Code's compaction, and lose all state on crash. The analysis notes this gap explicitly (Section 1.1).

We don't need the full brain-hands architecture yet, but we can adopt the session log pattern *today* for steward agents. Each steward already writes performance JSON — extend this to a structured event log that captures task progress, not just metrics.

Concrete change:
- Create `scripts/lib/session_log.sh` — shared library with `log_event(agent, event_type, payload)` writing append-only JSONL to `logs/sessions/{agent}-{date}.jsonl`
- Event types: `TASK_START`, `TASK_COMPLETE`, `TASK_SKIP`, `ERROR`, `CHECKPOINT`
- Modify each daily steward script to call `log_event` at key milestones
- On session start, steward agents read the latest session log to detect incomplete tasks from prior runs (crash recovery)
- This is the foundation for Phase 5.3.2's "task-level workflow state tracking" requirement

**Proposal 1.2: Lazy Provisioning for Nightly Agents**

The Managed Agents architecture achieves 60% median TTFT reduction from lazy container provisioning. Our nightly stewards spin up full Claude sessions regardless of available work. The researcher checks for news every night even when no releases happened. The stewards check for phase work even when no tasks are pending.

Concrete change:
- Add a pre-flight "work check" to each daily script — a lightweight function (pure bash, no Claude session) that checks whether there's actually work to do
- `daily_research_sweep.sh`: curl the GitHub releases API for Anthropic/Google repos; skip full session if no new releases since last sweep
- `daily_android_sw_steward.sh` / `daily_arm_mrs_steward.sh` / `daily_bsp_knowledge_steward.sh`: check git log for new commits in target repo since last run; check ROADMAP for unchecked tasks
- If pre-flight finds no work, write a "SKIP" performance JSON and exit — saves the full session cost
- Expected savings: 20-40% of daily compute on quiet days (weekends, freeze periods like current Google I/O prep)

**Proposal 1.3: Credential Isolation Design Document for Phase 5/7**

The analysis identifies that our entire fleet runs with Jonas's env vars — a compromised agent session exposes everything. Managed Agents' vault-based MCP proxy pattern is the reference architecture. This won't be implemented until Phase 5, but the design document should be written now while the Managed Agents architecture is fresh.

Concrete change:
- Write `knowledge_base/agentic-ai/evaluations/credential-isolation-design.md`
- Map current credential surface: which env vars each agent actually needs (most need only `ANTHROPIC_API_KEY`)
- Design a minimal isolation model: per-agent env var subsets, MCP proxy for shared credentials
- Reference Managed Agents' two patterns: resource-bundled auth (git) and vault-based proxy (OAuth)
- This becomes the Phase 5.3 security requirements input

### ENGINEER

**Response to 1.1 (External Session State):**

The pattern is sound and the gap is real. But let me push back on scope:

1. **We already have partial session state** — performance JSONs capture duration, exit code, commits, files changed. The `closed_loop.sh` state machine tracks pipeline stage in a JSON file. Adding a JSONL event log is incremental, not revolutionary.

2. **Crash recovery complexity**: "Read the latest session log to detect incomplete tasks" sounds simple, but it requires steward agents to understand their own event logs and make resume decisions. Today they're stateless by design — they read ROADMAP.md and decide what to work on. That's simpler and more robust than parsing a recovery log.

3. **What does a steward *do* with a `TASK_START` without `TASK_COMPLETE`?** Retry? Skip? The answer depends on whether the incomplete task left partial artifacts. This is the hard part, and the proposal doesn't address it.

That said, the event log *without* crash recovery is already valuable — it provides the observability foundation for Phase 5.3.2 and gives the project-reviewer structured data instead of git-log mining.

**Verdict: ADOPT (P2)** — Implement `session_log.sh` with structured event logging. Integrate into steward scripts for observability. **Defer** the crash-recovery-on-resume logic to Phase 5 where external session state is a first-class requirement. Keep the scope to logging, not recovery.

**Response to 1.2 (Lazy Provisioning):**

This is a solid cost optimization, but I want to verify the savings model:

1. **Research sweep is the strongest candidate** — it curls public APIs and genuinely has nothing to do when no releases happened. A GitHub API pre-flight is ~2 seconds vs. a 30-60 minute Claude session.

2. **Steward agents are weaker candidates** — they don't just react to external events; they advance ROADMAP tasks autonomously. Even with zero new commits in the target repo, the steward might need to write new tests, expand documentation, or run eval suites. A "no new commits" check would incorrectly skip these sessions.

3. **The factory steward runs 3x/day (weekday) and 2x/day (weekend)** — it always has work (proposals, performance tuning, eval refinement). Lazy provisioning doesn't apply.

4. **Risk of stale detection**: If the pre-flight check has a bug (e.g., wrong git remote, cached API response), agents silently stop running. We'd need monitoring for "too many consecutive SKIPs."

**Verdict: ADOPT (P2)** — Implement lazy provisioning for `daily_research_sweep.sh` only (strongest ROI, clearest skip criteria). Add a "consecutive skip" counter to performance JSONs — alert if any agent skips >3 consecutive runs. **Defer** steward lazy provisioning until we have better work-detection heuristics.

**Response to 1.3 (Credential Isolation Design Doc):**

No objections. A design document has near-zero cost and high future value. The current threat model (single user, single machine) means this is purely a design exercise — which is exactly what it should be at Phase 4.

Two additions:
1. Include the Agent SDK `exclude_dynamic_sections` pattern — it's relevant to credential isolation because it determines what system prompt content (potentially including credential references) is shared across cached sessions.
2. Map the credential flow for the daily scripts specifically: `crontab → bash script → env vars → Claude session → tool calls`. Each hop is an isolation boundary candidate.

**Verdict: ADOPT (P2)** — Write the design document. Include current credential surface mapping and Managed Agents reference patterns. No implementation until Phase 5.

**Proposal 3.1: CLAUDE.md Structural Split for Agent SDK Compatibility**

The ROADMAP already has a design note (Phase 5.2): "Separate CLAUDE.md into static base (cacheable across agents) and dynamic session context (per-agent)." The Agent SDK's `exclude_dynamic_sections` feature enables 40-60% input token savings through cross-user prompt caching — but only if static content is separated from dynamic content.

Our CLAUDE.md mixes static architecture descriptions with dynamic status (version numbers, countdown timers, completion percentages). When the factory steward updates the ROADMAP status line daily, it invalidates the entire CLAUDE.md cache for all agents.

Concrete change:

- All daily scripts source both files but only `CLAUDE_STATUS.md` gets daily updates
- Static `CLAUDE.md` becomes cacheable across all agent sessions — estimated 40-60% input token savings on the ~4000-token CLAUDE.md preamble
- This is a prerequisite for Agent SDK adoption in Phase 5

### ENGINEER

**Response to 3.1 (CLAUDE.md Structural Split):**

The cache invalidation analysis is compelling — daily status updates invalidating the entire CLAUDE.md cache is genuine waste. But execution concerns:

1. **Two-file complexity**: Every agent definition currently says "read CLAUDE.md." Splitting requires updating all 12+ agent definitions to read both files, or using an include mechanism. The maintenance burden increases.

2. **What counts as "static"?** The architecture section references specific file paths (`eval/run_eval_async.py`) and agent names — these change occasionally. The "static" file would still get occasional invalidations, just less frequently. The savings depend on how strictly we separate.

3. **Agent SDK adoption timeline**: We're not adopting Agent SDK until Phase 5. The cache savings are meaningless until then — our cron agents don't share sessions, so there's nothing to cache across.

4. **Simpler alternative**: Instead of splitting files, we could just move the dynamic status line from CLAUDE.md to ROADMAP.md (where it arguably belongs anyway). CLAUDE.md rarely needs the daily countdown updates — those are ROADMAP concerns.

**Verdict: DEFER** — The split is correct for Phase 5 Agent SDK adoption but premature now. Our agents don't share session caches. Moving the dynamic status line to ROADMAP.md header (where it's already duplicated) is the simpler immediate fix. Revisit when Phase 5 implementation begins and Agent SDK `exclude_dynamic_sections` becomes relevant.

**Response to 3.2 (Workflow Convergence Note Extension):**

No objections. The Managed Agents session log pattern is the missing piece in yesterday's comparison. Three points:

1. **The four-way comparison table is the real value** — it makes Phase 5.3.2 design decisions concrete: which pattern do we adopt? The answer is probably "Managed Agents event log" since it's closest to our existing JSON approach but adds crash safety.

2. **Keep it concise** — this is a design reference, not a research paper. One comparison table + one paragraph of recommendation is sufficient.

3. **Link it from the ROADMAP Phase 5.3.2 task** so future implementers find it.

**Verdict: ADOPT (P3)** — Update the convergence note with Managed Agents pattern. Add the four-way comparison table. Keep it brief. Link from ROADMAP Phase 5.3.2.

**Response to 3.3 (Sonnet 4.6 Knowledge Freshness Eval):**

Interesting idea but I think the premise is flawed:

1. **Our researcher doesn't rely on training data for current events** — it uses WebSearch and WebFetch to get live information. The training cutoff is irrelevant for "What's the latest Claude Code version?" because the agent reads the GitHub releases page directly.

2. **Where training data matters is in *interpreting* what it finds** — understanding architectural implications, comparing against prior art, generating strategic recommendations. That's where Opus's reasoning advantage dominates, and knowledge freshness is irrelevant.

3. **The experiment conflates two things**: knowledge recall (training data) and research quality (web search + analysis). We care about the latter, and Opus wins there regardless of cutoff date.

4. **Mixed-model complexity**: Running the researcher with Sonnet for planning and Opus for analysis means two model invocations per step, increasing latency and cost — the opposite of the savings goal.

**Verdict: REJECT** — The researcher's value comes from reasoning quality, not training data recency. Web search compensates for cutoff gaps. Mixed-model execution adds complexity without clear benefit. If we want to evaluate Sonnet for cost savings, do it on a simpler agent (e.g., project-reviewer) where reasoning depth matters less.
