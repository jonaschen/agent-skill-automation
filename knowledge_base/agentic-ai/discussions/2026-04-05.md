## Round 1: Security & Supply Chain

### INNOVATOR

**Proposal 1.1 — MCP Tool Description Content Validator (P0)**

The analysis flags MCP tool poisoning as P0 — the first demonstrated attack on MCP, published by Invariant Labs. Our `mcp_config_validator.sh` (added 2026-04-04) validates JSON structure but not tool description content. A malicious MCP server can pass our gate with poisoned tool descriptions that inject instructions into the agent's context.

Concrete proposal: Extend `eval/mcp_config_validator.sh` with a content validation phase. For any `.mcp.json` referenced in a generated Skill:

1. Add a static pattern scanner: reject descriptions containing instruction injection phrases (`"ignore previous"`, `"you must"`, `"do not tell"`, `"send to"` + URL patterns).
2. Flag descriptions exceeding 500 characters (legitimate tool descriptions are concise).
3. Reject descriptions containing file path references, credential keywords (`password`, `token`, `secret`, `API_KEY`), or bare URLs.
4. Maintain an allowlist at `eval/mcp_server_allowlist.json` — servers not on the list emit a warning (not a block) in the CI/CD gate output.

This is a bash script extension — minimal blast radius, fits the existing CI/CD gate architecture.

**Proposal 1.2 — Dependency Pinning + Audit Gate (P1)**

The LiteLLM supply chain compromise (3 ADK patches in 4 days) and the npm axios RAT are both real-world supply chain attacks on agent frameworks. Our pipeline has two exposure surfaces: Python packages (eval tools) and npm (Claude Code distribution chain).

Concrete proposal:
1. Run `pip freeze > requirements.txt` to create a locked dependency manifest. Check this into git.
2. Add `pip install --require-hashes -r requirements.txt` to CI where applicable.
3. Add `npm audit --audit-level=high` as a check in `.claude/hooks/pre-deploy.sh`.
4. Both are single-line additions to existing scripts.

**Proposal 1.3 — MCP Server Allowlist Default in meta-agent-factory (P1)**

When `meta-agent-factory` generates a `.mcp.json` for a new Skill, it should default to servers from the allowlist. If a user specifies an unknown MCP server, the factory should emit a warning comment in the generated config: `// WARNING: Server not in allowlist — validate tool descriptions before deploying`.

File to change: `.claude/agents/meta-agent-factory.md` — add allowlist awareness to the MCP generation instructions.

---

### ENGINEER

**On 1.1 (MCP Content Validator):**

Agree this is the right priority. The implementation is low-cost — regex pattern matching in a bash script, extending an existing file. Two concerns:

- **False positives on legitimate descriptions**: A tool that legitimately handles passwords (e.g., a vault manager MCP server) would have "password" in its description. The allowlist addresses this — known-good servers bypass content scanning. Make sure the allowlist check runs *before* content scanning so we don't alert on trusted servers.
- **Fetch-at-validation-time is harder than it sounds**: The proposal says "fetch tool descriptions at validation time." This means the CI/CD gate needs network access to MCP servers, which introduces a new failure mode (server unreachable = gate blocked). For Phase 4, static scanning of `.mcp.json` content is sufficient. Dynamic fetching can wait.

**Verdict on 1.1**: **ADOPT (P0)** — static content scanning of `.mcp.json` tool descriptions, with allowlist bypass. Defer dynamic server fetching to Phase 5.

**On 1.2 (Dependency Pinning):**

This is textbook defense-in-depth with near-zero implementation cost. `pip freeze` and `npm audit` are both one-liners. The only question is enforcement — do we block deployment on `npm audit` findings, or just warn?

Given that we're downstream of Claude Code's npm distribution, a high-severity npm audit finding could block our entire pipeline on something we can't fix. Recommend: `npm audit` warns but doesn't block; `pip install --require-hashes` blocks (we control our Python deps).

**Verdict on 1.2**: **ADOPT (P1)** — `pip freeze` as blocking; `npm audit` as warning.

**On 1.3 (Allowlist in meta-agent-factory):**

This is a description-level change to an agent definition — the lightest possible modification. The warning comment approach is good — it's informational, not blocking, so it doesn't prevent legitimate use of custom MCP servers.

But: the meta-agent-factory already has a dense description. Adding allowlist logic to the agent's instructions risks pushing it over the description complexity threshold where the model starts ignoring instructions. Keep it to one sentence: "When generating .mcp.json configs, prefer servers from eval/mcp_server_allowlist.json and include a WARNING comment for servers not on the list."

**Verdict on 1.3**: **ADOPT (P1)** — one-sentence addition to meta-agent-factory.md.

## Round 2: Architecture & Efficiency

### INNOVATOR

**Proposal 2.1 — Progressive Disclosure for Skill Loading (P2)**

Gemini CLI v0.36.0 implements progressive disclosure — only skill metadata loaded initially, full instructions injected on activation. Our pipeline loads all Skill definitions into context at startup, which is fine with 5-10 Skills but scales poorly.

The analysis notes this as a cross-pollination opportunity. Our current architecture already has the pieces: `.claude/skills/*/SKILL.md` files with YAML frontmatter (name, description, trigger patterns). The frontmatter *is* the metadata layer.

Concrete proposal: Implement a two-tier loading strategy in our agent fleet:
1. **Tier 1 (always loaded)**: Name + description (first 200 chars) + trigger patterns from YAML frontmatter.
2. **Tier 2 (loaded on match)**: Full SKILL.md content injected only when a prompt matches the trigger pattern.

This is primarily a Claude Code hooks architecture change. The `PreToolUse` hook already does pattern matching — extend it to inject full Skill content on first match rather than loading everything at startup.

File to change: The Vercel plugin's `pretooluse-skill-inject.mjs` already implements exactly this pattern (session dedup, priority sorting, MAX_SKILLS cap). Our own skill loading should follow the same architecture.

**Proposal 2.2 — Graph-Based Closed-Loop Pipeline (P2)**

Our `scripts/closed_loop.sh` is a linear pipeline: factory → validate → optimize → deploy. The analysis cites ADK v2.0's graph-based workflow runtime as an alternative that supports:
- Parallel validation + security scanning before optimization
- Conditional branches (skip optimization if trigger rate already >= 0.95)
- Retry sub-graphs without restarting the full pipeline

Concrete proposal: Refactor `closed_loop.sh` into a state machine with explicit nodes and edges. Not a full graph runtime — just a bash case statement with state transitions:

```
START → GENERATE → VALIDATE ─┬─ pass(>=0.95) → DEPLOY
                              ├─ pass(>=0.90) → SECURITY_SCAN → DEPLOY
                              ├─ conditional(>=0.75) → OPTIMIZE → VALIDATE (loop)
                              └─ fail(<0.75) → REPORT_FAILURE
```

This eliminates unnecessary optimization runs and adds the security scan before deploy.

**Proposal 2.3 — Context Usage Monitoring for Long-Running Agents (P2)**

Agent SDK v0.1.56 adds `get_context_usage()` — programmatic context window reporting. Our 6 nightly agents run 30-90 minutes and rely on opaque model-side compaction.

This one is a *defer* from my side, actually. Our agents run in Claude Code, not the Agent SDK. `get_context_usage()` is a Python SDK feature. Unless we wrap our agent invocations in SDK-based orchestration, this isn't directly applicable. The pattern is right (proactive context management > reactive compaction) but the tool doesn't fit our runtime.

---

### ENGINEER

**On 2.1 (Progressive Disclosure):**

The idea is sound in principle, but there's a critical misunderstanding of our architecture. Claude Code's skill loading is *already* progressive in the Vercel plugin — the `pretooluse-skill-inject.mjs` hook does exactly this (metadata match → full inject, session dedup, MAX_SKILLS=3 cap). This is documented in the system prompt.

For our *own* agents (the nightly steward fleet), the issue is different. They run as Claude Code sessions with `.claude/agents/*.md` definitions. The model loads all *agent* descriptions at startup to decide routing — not all Skill contents. Agent descriptions are short (under 1024 chars each). With 12 agents, that's ~12KB of descriptions — well within budget.

The *real* scaling issue isn't Skill loading — it's that adding more agents with overlapping descriptions causes routing regression (Lesson L7: T=0.895 → T=0.658). Progressive disclosure doesn't fix routing competition; it's a different problem.

**Verdict on 2.1**: **DEFER** — the architecture already has progressive loading (Vercel plugin hooks). The actual problem (routing regression from agent fleet expansion) needs a different solution — description disambiguation, not lazy loading.

**On 2.2 (Graph-Based Closed Loop):**

I like the state machine concept but not the framing. ADK v2.0's graph runtime is alpha software from Google — we should not adopt their architecture. However, refactoring `closed_loop.sh` from a linear pipeline to a state machine with conditional branches is a good improvement independent of ADK.

Specific feedback:
- The "skip optimization if >= 0.95" shortcut is valuable — it saves significant API cost and time.
- Adding SECURITY_SCAN as an explicit node (running the MCP content validator + permission checker in parallel) before DEPLOY is a natural place for Proposal 1.1.
- A retry counter on the OPTIMIZE → VALIDATE loop is essential (currently the optimizer has a 50-iteration budget, but the closed loop has no explicit retry limit).

Rename: this isn't "graph-based" — it's just a proper state machine. Don't over-engineer it.

**Verdict on 2.2**: **ADOPT (P2)** — refactor `closed_loop.sh` into a state machine with conditional skip, parallel security scan, and retry limits. Not as a new system — as a rewrite of the existing script.

**On 2.3 (Context Usage Monitoring):**

The Innovator correctly self-deferred this. Our agents are Claude Code sessions, not Agent SDK processes. The pattern is good but we can't use the tool. When/if Claude Code exposes context usage via its own API or hooks, revisit.

**Verdict on 2.3**: **DEFER** — right idea, wrong runtime. Revisit when Claude Code exposes context usage programmatically.

## Round 3: Strategic & Forward-Looking

### INNOVATOR

**Proposal 3.1 — Agent Payment Protocol Tracking for Phase 7 (P2)**

Four competing payment protocols launched in 90 days: AP2 (Google, mandate-based), Visa TAP (identity certificates), x402 (HTTP micropayments), PayPal Agent Ready (protocol-agnostic). Our Phase 7 is Stripe-only.

Concrete proposal: Add a new task to ROADMAP Phase 7: "7.7 — Survey agent payment protocols (AP2, TAP, x402, PayPal) and design a billing adapter layer supporting multiple settlement rails." Don't implement anything now — just reserve the architectural space.

Also update the risk table: "Phase 7 billing architecture assumes Stripe-only; agent payment protocol war may require multi-rail support."

**Proposal 3.2 — Capybara/Mythos Model Contingency Plan (P1)**

Anthropic warned government officials that Mythos "makes large-scale cyberattacks significantly more likely in 2026." References to Opus 4.7 and Sonnet 4.8 found in internal configs. If a new model tier ships above Opus 4.6, our eval baselines, optimized descriptions, and routing behavior are all potentially invalidated.

Concrete proposal: Add a runbook to `eval/` — `eval/model_migration_runbook.md` — documenting the steps to re-baseline when a new model ships:
1. Run full eval suite (T+V) on new model with current descriptions
2. Compare posterior means and CIs against Opus 4.6 baselines
3. If regression > 5%, trigger AutoResearch optimizer with new model
4. Update CLAUDE.md and agent definitions to reference new model

This costs nothing to create now and saves significant scramble time when a new model drops.

**Proposal 3.3 — Conway Event-Driven Agent Architecture Prototype (P3)**

Conway leaked: webhook-driven agent wake, `.cnw.zip` extensions, persistent instances. Our cron model is batch-oriented. The analysis notes that `scripts/github_issue_triage.sh` (currently untracked in the repo) could be an early webhook-to-agent prototype.

Concrete proposal: Don't build for Conway (it's internal). Instead, prototype event-driven agent activation using what we have today: GitHub Actions webhooks → Claude Code CLI invocation. Specifically:
1. Wire `scripts/github_issue_triage.sh` to a GitHub Actions workflow triggered on `issues.opened`
2. The script invokes Claude Code with the issue context and routes to the appropriate agent
3. This gives us event-driven agent activation without waiting for Conway

---

### ENGINEER

**On 3.1 (Payment Protocol Tracking):**

A ROADMAP task and a risk table entry cost nothing and future-proof the architecture. No objection. Keep it as documentation only — no code, no design, no adapter interfaces. The protocol war hasn't produced a winner yet, and premature abstraction for 4 payment rails we don't understand is worse than no abstraction.

**Verdict on 3.1**: **ADOPT (P2)** — ROADMAP task + risk entry. Documentation only.

**On 3.2 (Model Migration Runbook):**

This is good operational hygiene. The analysis upgrades Capybara/Mythos from P2 to P1, and the leaked Opus 4.7/Sonnet 4.8 references suggest new models are coming within months. A runbook is cheap insurance.

One addition: the runbook should also cover the routing regression risk. A new model might route *better* (resolving our T=0.658 regression) or *worse*. Step 1 should explicitly check both positive and negative test sets separately, not just overall trigger rate.

**Verdict on 3.2**: **ADOPT (P1)** — create `eval/model_migration_runbook.md`.

**On 3.3 (Conway Event-Driven Prototype):**

I'm skeptical of the framing but agree with the underlying idea. We should *not* call this a "Conway prototype" — Conway is internal Anthropic software that may never ship or may ship in a completely different form. Building toward a leaked product is poor engineering.

However, the *actual proposal* — wiring `github_issue_triage.sh` to a GitHub Actions webhook — is a straightforward, useful improvement independent of Conway. Event-driven agent activation via GitHub Actions is a well-understood pattern. The script already exists (untracked).

Concerns:
- Cost: each GitHub issue triggers a Claude Code session ($2-5 at API rates). For high-volume repos, this needs a rate limiter or a filter (only issues with specific labels trigger the agent).
- Security: the issue body becomes agent input. Need to sanitize or at least scope the agent's permissions tightly (read-only analysis, no code changes).
- This is a Phase 4+ task — we shouldn't start it until the core closed loop is stable.

**Verdict on 3.3**: **DEFER** — the GitHub Actions webhook approach is good but premature. Complete Phase 4 stress test (50 Skills in 24 hours) first. Revisit as a Phase 4.5 task. Drop the Conway framing entirely.
