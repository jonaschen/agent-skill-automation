## Round 1: OTEL CLI-Native Discovery and S1 Automated Improvement Loop

### INNOVATOR

**Proposal 1.1: Design an OTEL Trace Analyzer Agent for S1 Automated Improvement (P1 — Strategic)**

The evening analysis uncovered the single most impactful finding for S1 today: OTEL tracing is CLI-native. The SDK doesn't produce telemetry — it passes env vars to `claude -p`, and the CLI emits traces directly. This collapses the "months of SDK migration" timeline to "days of env var addition."

But the evening discussion stopped at "add env vars now, deploy collector later." That's the infrastructure half. The strategic half — advancing S1's goal of *automatic agent improvement* — needs a design now, even if implementation is Phase 5.

The concept: an **OTEL trace analyzer agent** that reads span data from completed steward sessions and generates improvement suggestions. With structured traces, we can automatically detect:

- **Inefficient tool call patterns**: steward spent 40% of tokens on retried Bash calls (suggests prompt improvement)
- **Delegation anomalies**: factory-steward called meta-agent-factory 0 times in a session that should have created a skill (delegation regression)
- **Cost outliers**: researcher session consumed 3x median tokens with no additional commits (suggests runaway reasoning)
- **Convergence failures**: optimizer ran 30 iterations without CI improvement (suggests description is at local optimum)

This is the "capability diff at the behavioral level" the night analysis identifies as the first concrete path to S1. Not tracking platform capabilities (the researcher already does that), but tracking our own agents' behavior and auto-generating optimization targets.

Concrete deliverable: a design document at `knowledge_base/agentic-ai/experiments/otel-trace-analyzer-design.md` specifying:
1. What span attributes to capture (tool name, duration, token count, agent name, session ID)
2. What patterns to detect (the 4 above plus extensible pattern registry)
3. How to connect detection to action (generate steering notes, create ROADMAP tasks, trigger optimizer)
4. Phase 5 integration point (runs after each steward session, writes findings to `logs/trace-analysis/`)

**Proposal 1.2: Run a Controlled OTEL Pilot on Factory-Steward (P1 — Experimental)**

Before designing the analyzer, we need real trace data to understand the span structure. The evening discussion's A1 (add OTEL env vars) points to a configurable endpoint but no collector. Without a collector, traces are silently dropped — we learn nothing.

I propose a focused 1-day pilot:
1. Deploy a minimal OTEL collector (single Docker container, stdout JSON exporter — no Jaeger/Grafana needed yet)
2. Run factory-steward once with OTEL env vars active
3. Capture the raw trace JSON
4. Analyze: what spans does `claude -p` actually emit? What attributes are present? Is token count per-span or per-session? Are subagent invocations separate spans?
5. Write findings to `knowledge_base/agentic-ai/experiments/otel-pilot-results.md`

This is S1 experimental research — "design experiments and run controlled comparisons using the existing pipeline as a testbed" per the strategic priorities meta-goal. Total effort: ~2 hours. Compute cost: one factory-steward session + Docker container.

That said, the concept is strong and directly addresses S1. I'd reframe: write a *brief* experiment design document (1 page, not a full spec) that describes what we're trying to learn from the pilot, then expand to a full analyzer design after the pilot produces data.

**Response to 1.2 (OTEL Pilot):**

This is exactly the kind of experiment the strategic priorities meta-goal calls for. Low cost, high information value. Two adjustments:

1. **Don't deploy a Docker container.** Use `OTEL_TRACES_EXPORTER=console` (or `OTEL_EXPORTER_OTLP_PROTOCOL=http/json` with endpoint pointing to `file:///path/to/traces.jsonl`). **Verdict on 1.2**: ADOPT P1 with Engineer's adjustments — console exporter first, manual test session, then expand.

### ENGINEER

**Response to 3.1 (Shadow Eval Go/No-Go Criteria):**

Strong proposal. Having explicit numeric criteria prevents post-hoc rationalization ("the score dropped but maybe it's fine"). Two refinements:

1. The 0.05 tolerance on posterior mean is too tight. Our baseline is 0.829 with CI [0.702, 0.927] — that's a wide interval reflecting N=39 with non-trivial failure rate. A model change could shift the mean by 0.05 purely from routing noise. The CI overlap test (criterion 2) is the correct statistical gate — it's what our Bayesian framework was designed for. Drop criterion 1 (point estimate comparison) and rely on criterion 2 (CI overlap). If CIs overlap, the models are statistically indistinguishable at our sample size.

2. The 400-error criterion is good but should distinguish between eval infrastructure errors (network, rate limit) and model-returned errors (breaking change). A 400 from rate limiting is a retry condition, not a no-go signal.

3. Duration criterion: 1.5x is reasonable for first run. But note the night analysis estimates 1.35x from tokenizer alone — so 1.5x leaves only 11% headroom for variance. I'd raise to 2x for go/no-go and track the actual ratio.

On documenting in `model_migration_runbook.md`: yes, that's the right home. The runbook already has migration steps but lacks quantitative go/no-go gates.

**Verdict on 3.1**: ADOPT P0 — with CI overlap as primary gate (drop point-estimate criterion), 400-error check (model-returned only), and 2x duration threshold. Document in migration runbook.

**Response to 3.2 (Cost Control Architecture Document):**

The 4-layer table is clean and informative. But does it need a standalone document? The architecture is already implicitly documented across `cost_ceiling.sh`, the steward scripts (where `--max-budget-usd` and `--max-turns` will live), and CLAUDE.md's architecture section. A standalone doc adds a maintenance burden — when someone changes a cost parameter, they now need to update 2+ files.

Counter-proposal: add a 10-line "Cost Control Stack" subsection to CLAUDE.md's Architecture section. It stays in the single source of truth rather than becoming yet another file.

**Verdict on 3.2**: ADOPT P2 — as a CLAUDE.md subsection, not a standalone document. Keep it under 15 lines.

**Response to 3.3 (Programmatic Tool Calling permissions.deny):**

Absolute yes. This is a textbook defense-in-depth measure: block the capability at the permission layer until we've designed the audit layer. The analysis is clear — 4/5 hooks bypassed. The 1-line deny rule costs nothing and prevents accidental exposure.

One concern: verify the deny rule syntax. Is it `Tool(code_execution_20260120)` or `code_execution_20260120` or some other format? Check existing `permissions.deny` entries for the pattern.

**Verdict on 3.3**: ADOPT P1 — verify deny rule syntax, add the entry.
