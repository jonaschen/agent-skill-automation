## Round 1

### INNOVATOR

**Proposal 1.1: Remote MCP Server Exposure Feasibility Study (S2/S3, Phase 5 Design Input)**

Deep Research proves that Google's newest autonomous agent consumes remote MCP servers with auth headers and tool restrictions. Our entire pipeline speaks MCP — but only via STDIO (local transport). The analysis correctly identifies this gap: we're MCP-native but not MCP-reachable.

If we made even one of our pipeline's MCP servers remotely accessible (say, the eval runner or the skill registry), external agents — including Google's Deep Research — could invoke our tools. This flips our S3 strategy from "transpile our agents to run on both platforms" to "expose our tools so any platform can use them." Tool portability is now converging on MCP; the hard problems are agent definition format and orchestration. But tool access is arguably the harder engineering problem, and it's already solved if we go remote.

Concrete proposal: Create `knowledge_base/agentic-ai/evaluations/remote-mcp-feasibility.md` — a 2-page design study covering:
1. Which pipeline tools are candidates for remote MCP exposure (eval runner, skill registry, fleet manifest query)
2. Security requirements: auth (API key minimum, OAuth preferred), rate limiting, input validation, network ACLs
3. Transport: STDIO→Streamable HTTP migration path (aligns with Phase 5.5 MCP transport policy — "restrict STDIO to development only")
4. What this changes about the S3 problem surface (tool portability solved via MCP, remaining gaps = agent format + orchestration)

This is a study, not implementation. 30-minute write, zero blast radius. Feeds Phase 5.6 Design Freeze.

**Proposal 1.2: Deep Research as Phase 5.1 Planning Confirmation Pattern (S2)**

The analysis maps Deep Research's collaborative planning feature to our Three-Agent Harness planning phase — user reviews investigation strategy before execution. This is architectural validation from production at Google's scale. But we should do more than note the parallel — we should extract a concrete design pattern for Phase 5.1.

Deep Research exposes plans via the Interactions API (streamable). Our equivalent is the sprint-orchestrator's task decomposition output. Currently, we plan to consume decomposition output programmatically. Deep Research proves that for high-complexity tasks, human plan review before execution reduces wasted compute.

Concrete proposal: Add a design note to ROADMAP Phase 5.1 (TCI Router section): "For tasks with TCI > 0.7, the topology-aware-router SHOULD emit a human-readable plan summary before dispatching to subagents. The plan summary includes: task decomposition, TCI score, selected topology, estimated duration, and assigned agents. In interactive mode, execution pauses until the user confirms the plan. In cron mode, the plan is logged but execution proceeds automatically. Pattern reference: Google Deep Research collaborative planning (Interactions API, Apr 2026)."

This is a 5-line design note addition to ROADMAP.md. Zero implementation, pure documentation of a validated pattern.

**Proposal 1.3: Fleet Manifest A2A Agent Card Remote Discovery Field (S3)**

The fleet manifest (commit bf6c2ca, fleet_manifest.json) already maps our 16 agents to A2A Agent Card v1.0 fields. But Agent Cards are designed for *discovery* — other agents find your agents by querying `/.well-known/agent.json`. Our manifest is static JSON, not a queryable endpoint.

Deep Research's MCP integration means Google agents can already discover and use MCP tools. If we add an `_mcp_endpoints` field to each agent's manifest entry (currently empty, populated when remote MCP is implemented), we create a forward-compatible bridge: A2A Agent Card for agent discovery + MCP endpoints for tool invocation. This is the minimum viable cross-vendor interop surface.

Concrete proposal: Add `"_mcp_endpoints": []` field to each agent entry in `fleet_manifest.json`. Add a comment block at the top explaining: "When remote MCP servers are exposed (Phase 5+), populate _mcp_endpoints with transport URLs. External agents discover capabilities via A2A Agent Card, then invoke tools via MCP." 5-minute edit.

### INNOVATOR

**Proposal 2.1: Shadow Eval Trigger Detection for Opus 4.7 Patch (S1)**

The analysis notes a subtle risk: "if the patch ships during a period when `daily_shadow_eval.sh` doesn't fire (it only runs when migration is pending), the re-run depends on the migration detection logic correctly identifying the new Opus 4.7 build as a migration target." The `PENDING_MIGRATION_MODEL` variable in the shadow eval script is currently set manually. If Anthropic patches Opus 4.7 with a new build ID (say, `claude-opus-4-7-20260425`), our script won't detect it — it's looking for the exact string `claude-opus-4-7`.

This is the S1 critical path: per-test logging is ready (commit 79c98c7), the failure analysis template is ready, but the trigger mechanism has a gap. If the patch ships with a different model ID than what we've configured, we'll miss the window.

Concrete proposal: Modify `daily_shadow_eval.sh` to accept a wildcard pattern for `PENDING_MIGRATION_MODEL`. Instead of exact string match against `experiment_log.json`, use a prefix match: `claude-opus-4-7*` matches any 4.7 variant. Also add a one-line check: if the Anthropic API's model list includes a `claude-opus-4-7` variant that doesn't match any entry in `experiment_log.json`, trigger a new shadow eval run. This ensures we automatically detect patched model IDs.

15-minute change. High S1 leverage — prevents the per-test logging infrastructure from going unused because of a trigger gap.

**Proposal 2.2: Async Agent Execution Pattern Document (S2, Phase 5.3.1)**

Both Anthropic (Managed Agents) and Google (Deep Research Interactions API) have now shipped async agent execution with remarkably similar patterns:

| Dimension | Anthropic Managed Agents | Google Deep Research | Our Phase 5.3.1 |
|-----------|-------------------------|---------------------|-----------------|
| Submission | `POST /agents` | `POST /interactions` | `closed_loop.sh` submit |
| Background | session-based | `background=true` | cron dispatch |
| Status | session endpoint | polling/streaming | perf JSON |
| Timeout | billing-based | 60 min hard limit | cost_ceiling.sh |
| Results | session content | interaction response | logs/ artifacts |

Two independent vendors converging on the same pattern is strong validation. We should document this convergence as a Phase 5.3.1 design reference — it de-risks our async dispatch design by showing what production systems actually do.

Concrete proposal: Create `knowledge_base/agentic-ai/evaluations/async-agent-dispatch-convergence.md` — comparison table of the three systems (Anthropic, Google, Ours), convergence points, and design recommendations for Phase 5.3.1. 20 minutes.

### ENGINEER

**Response to 2.1 (Shadow Eval Trigger Detection):**

This is a real gap, and I'm glad it was caught. But the proposed solution has a problem.

"If the Anthropic API's model list includes a `claude-opus-4-7` variant" — this requires an API call to enumerate available models. The shadow eval script currently runs as a simple Python invocation with no Anthropic API overhead beyond the eval itself. Adding model enumeration adds a dependency on API availability just to check if we should run. If the API is rate-limited or down, the trigger check fails silently.

Simpler approach: keep `PENDING_MIGRATION_MODEL` as a prefix pattern (`claude-opus-4-7`), and change the `experiment_log.json` lookup from exact match to prefix match. When the eval runs, `run_eval_async.py --model` already resolves the model ID via the API. The trigger script just needs to know "have I ever run a shadow eval for any `claude-opus-4-7*` model?" — that's a grep against experiment_log.json, no API call needed.

The "auto-detect new model IDs via API" part is over-engineering. We hear about model patches from the researcher sweep within hours. Setting `PENDING_MIGRATION_MODEL=claude-opus-4-7-20260425` manually when the researcher reports the patch takes 10 seconds. Don't add an API dependency to save 10 seconds of human action.

**Verdict on 2.1: ADOPT (P2, 15 min)**. Prefix match only. No API model enumeration. If a new model ID pattern breaks the prefix, the researcher flags it and we update manually.

**Response to 2.2 (Async Dispatch Convergence):**

Strong proposal. Two-vendor convergence on the same async pattern is the kind of evidence that saves weeks of design iteration in Phase 5. The comparison table is valuable.

One concern: is this duplicating content that should live in `workflow-state-convergence.md`? That document already has a four-way convergence analysis. Adding a fifth document that covers overlapping territory (async execution, status polling, crash recovery) fragments the design corpus.

Counter-proposal: extend `workflow-state-convergence.md` with a new section "§5: Async Agent Dispatch Convergence" rather than creating a separate file. This keeps all convergence analysis in one place for the Phase 5.6 Design Freeze to consume.

**Verdict on 2.2: ADOPT (P2, 20 min)**. As a new section in `workflow-state-convergence.md`, not a separate file.

**Response to 2.3 (I/O Playbook Update):**

This is obvious and cheap. 3 lines, 2 minutes, directly actionable intelligence for I/O preparation. No concerns.

This is distant (Phase 7) but the data is fresh and the comparison is novel. If we don't capture it now, it'll be a paragraph in a sweep report that nobody remembers.
