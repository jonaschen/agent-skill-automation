## Round 1

### INNOVATOR

**Proposal 1.1: Dedicated Shadow Eval Cron Job with Result-Driven Factory Gating**

The shadow eval diagnosis is the most important finding this cycle. A1 works — the pipeline autonomously detected Opus 4.7, generated the upgrade task, launched the eval. It failed only because the 44-minute factory session can't host an 88-minute eval. This is a scheduling problem with a scheduling solution.

I propose: add a new cron entry at 2:30 AM (between researcher at 1 AM and research-lead at 2 AM — wait, that's backwards. Between research-lead at 2 AM and factory at 3 AM). Call it `scripts/daily_shadow_eval.sh`. It does one thing: checks `experiment_log.json` for `PENDING_MIGRATION_MODEL`, and if set with zero results, runs `python3 eval/run_eval_async.py --model $MODEL --split train --inter-test-delay 15 .claude/agents/meta-agent-factory.md`. No Claude session needed — it's a direct Python script. ~90 minutes. Results land in `experiment_log.json`. Then at 3 AM, the factory session's A1 gate-first check finds results already present and skips to ADOPT work.

This completes the S1 self-improvement loop end-to-end: researcher detects model -> deprecation registry tracks it -> migration runbook defines criteria -> shadow eval runs automatically -> factory reads go/no-go -> graduated rollout begins. Zero human intervention for model migration.

**Proposal 1.2: Agent Card Fleet Registry (Machine-Readable Agent Catalog)**

The production ADK+A2A+Cloud Run article uses Agent Cards (`agent-card.json`) for service discovery. We maintain a manual table in CLAUDE.md — 5 core agents, 2 research agents, 3 paper agents, stewards. Every time we add or suspend an agent, someone edits CLAUDE.md by hand.

I propose: generate an A2A-compatible Agent Card per agent. Each card is a JSON file at `.claude/agents/<name>.card.json` containing: agent name, description, model, permission class, status (active/suspended), capabilities (tools granted), input/output schemas. A fleet registry script (`scripts/fleet_registry.sh`) reads all cards and produces the CLAUDE.md table automatically. This gives us:
1. Machine-readable fleet state for health dashboard, anomaly alerter, and future Phase 5 topology router
2. A2A-compatible identity layer — when Phase 5 adopts A2A, the cards are already in place
3. Single source of truth — agents self-describe, CLAUDE.md derives from them

**Proposal 1.3: Factory Pipeline Feedback Loop (Validation -> Regeneration)**

The ADK+A2A article documents a progression: sequential pipeline -> feedback loop -> orchestration. Our factory pipeline is strictly sequential: factory -> validate -> optimize -> deploy. If validation fails, we run the full optimizer (up to 50 iterations). But some failures are obvious: wrong permission class, missing frontmatter field, description too long. These don't need 50 iterations of optimizer — they need one targeted regeneration pass.

I propose: add a REGENERATE step between VALIDATE and OPTIMIZE in `closed_loop.sh`. If the validator report identifies a structural failure (not a trigger-rate failure), feed the failure reason back to meta-agent-factory with the original requirement + the specific error. Structural failures are: permission violation, missing frontmatter, description > 1024 chars, empty SKILL.md. Trigger-rate failures still go to the optimizer. This creates a feedback loop that short-circuits the expensive optimize path for cheap-to-fix problems.

### ENGINEER

**On 1.1 (Shadow Eval Cron):**

The architecture is sound. But two concerns:

First, this isn't a cron job that invokes `claude -p` — it's a direct Python script execution. That's actually simpler and cheaper than a Claude session. The only dependency is `ANTHROPIC_API_KEY` in the environment (for the eval's Claude CLI calls). Good — no new Claude session costs.

Second, the timing. You said 2:30 AM between research-lead and factory. But research-lead runs at 2 AM and may take 30-60 minutes. The shadow eval needs ~88 minutes. If we start at 2:30 AM and it runs until ~4 AM, it overlaps with the factory session at 3 AM. Both hit the API. Better: run the shadow eval at 11:30 PM (before the night cycle starts) or make the factory script check for eval completion and wait/skip if the eval is still running.

Actually, simplest: the eval is a Python script, not a Claude session. API calls from the eval runner (which invokes `claude -p` per test) do compete for quota. But if we set `--inter-test-delay 30` instead of 15, it can coexist with other sessions. Or just run it at 11:30 PM when nothing else is running.

The larger question: how often does this fire? Only when `PENDING_MIGRATION_MODEL` is set and experiment_log.json has no results. Most days it does nothing (no pending migration). When it fires, it fires once — results persist. So the cost is one 90-minute eval per model release, amortized over months. Trivially acceptable.

**Verdict: ADOPT (P1).** Clean solution. Schedule at 11:30 PM to avoid API overlap. Jonas still needs to do the manual run NOW for the current Opus 4.7 migration — this cron job prevents the same problem next time.

**On 1.2 (Agent Cards):**

I like the direction but challenge the scope. Agent Cards are an A2A spec with fields we don't need yet (authentication, endpoint URLs, supported protocols). Generating full A2A-compatible cards for agents that run as local CLI invocations is premature — we'd be filling in dummy values for most fields.

What we actually need is a machine-readable fleet manifest. That's a subset: name, model, status, permission class, tools. A single `fleet_manifest.json` (not per-agent card files) would serve the dashboard, alerter, and CLAUDE.md generation use cases without the A2A overhead.

When Phase 5 actually adopts A2A, we extend the manifest entries to full Agent Cards. The manifest is the 80% solution at 20% of the effort.

**Verdict: ADOPT (P2) — but as fleet_manifest.json, not per-agent Agent Cards.** Defer A2A-compatible format to Phase 5.

**On 1.3 (Feedback Loop):**

The insight is correct — structural failures are cheap to fix and shouldn't burn optimizer iterations. But I question whether this is a real problem. Look at the pilot data: 8/10 skills deployed at 0.95 on first pass. The 2 failures were STUB (model self-gating), not structural. The optimizer has never been invoked by the stress test because generation quality is high enough.

Adding a REGENERATE step to `closed_loop.sh` adds complexity to a pipeline that currently works. The regeneration prompt would need to be carefully crafted (include the validator report, the original requirement, and explicit error context), and we'd need to cap regeneration attempts (infinite regeneration loops are worse than going to the optimizer).

That said, the pattern is sound for when we scale to 50 skills (Phase 4.2b / Phase 5). At that scale, even a 5% structural failure rate means 2-3 skills per batch hitting the optimizer unnecessarily.

**Verdict: DEFER (Phase 5).** No current pain point. Implement when 4.2b throughput testing reveals optimizer bottlenecks from structural failures.

## Round 2

### INNOVATOR

**Proposal 2.1: Pre-I/O Burst Readiness — Automated KB Update Pipeline**

Both vendors are in their deepest simultaneous freeze. When they unfreeze — Google around May 2 (pre-I/O leaks), Anthropic timing unknown — we'll get a burst of releases. The post-I/O response playbook exists for Google, but the operational challenge is different: the researcher agent processes one vendor's updates per sweep, writing individual topic files. A burst of 5-10 simultaneous releases across both vendors would take 2-3 sweeps to fully process, during which the KB is stale.

I propose: add a `--burst-mode` flag to `daily_research_sweep.sh`. When set (manually or automatically when >3 new releases detected in a single GitHub API pre-flight), it:
1. Doubles the session budget (2x `--max-turns`)
2. Prioritizes KB file updates over analysis/discussion generation
3. Skips the lazy-provisioning "no new releases" shortcut
4. Writes a `burst_summary.md` with all changes in one place for the research-lead

This is cheap insurance — the flag costs nothing when not used, and saves 1-2 days of KB staleness when a vendor unloads 6 months of accumulated changes at I/O.

**Proposal 2.2: Experiment Log Health Check**

The analysis noted that `experiment_log.json` had a "parse error suggests it may need format check." This is concerning — the experiment log is the state file for the entire S1 self-improvement loop. If it's malformed, the A1 gate-first check could silently fail (no entries found because JSON parsing fails, not because entries don't exist).

I propose: add a `--validate` flag to `eval/show_experiments.sh` that checks:
1. Valid JSON (parse test)
2. Required fields present in each entry (model, timestamp, results)
3. No duplicate experiment IDs
4. Timestamps in chronological order
5. Results arrays have expected structure

Run this as part of `pre-deploy.sh` (it's a <1s check) and as a pre-flight in `daily_shadow_eval.sh` (from Proposal 1.1). If validation fails, log a structured alert and refuse to write new results (prevents compounding corruption).

**Proposal 2.3: Session Duration Telemetry for Autonomous Scheduling**

The shadow eval time budget problem is a symptom of a broader issue: we assign fixed cron durations to tasks with variable runtime. The factory session sometimes finishes in 30 minutes, sometimes needs 90. The researcher sometimes skips (lazy provisioning), sometimes runs 45 minutes. We have no feedback loop from actual duration to scheduled duration.

I propose: add a `scripts/lib/schedule_telemetry.sh` that reads the last 30 days of performance JSONs and computes P50/P90/P99 durations per agent. Then a `scripts/schedule_advisor.sh` that compares actual durations against cron spacing and flags conflicts: "factory P90 is 52 minutes but only has 60 minutes before the next cron job." This doesn't change cron automatically — it generates a report that Jonas (or the research-lead) reviews when adjusting the schedule.

### ENGINEER

**On 2.1 (Burst Mode):**

The problem is real but the solution is over-engineered. The researcher already handles multi-topic sweeps — it writes separate vendor files and a consolidated report. A "burst" of 5-10 releases would mean a longer researcher session (more searches, more KB updates), but that's handled by existing session budget.

The `--burst-mode` flag adds a code path that's tested maybe twice a year (I/O and one Anthropic event). Untested code paths are liability, not insurance. Instead: if the pre-flight detects >3 new releases, just set `FORCE_SWEEP=1` (already exists) and let the normal session run longer. If it times out, the next sweep catches the remainder.

The `burst_summary.md` idea is good — a single consolidated change list after a multi-release event. But that's a research-lead output task, not a researcher infrastructure change.

**Verdict: REJECT.** Existing `FORCE_SWEEP=1` handles the detection case. Research-lead can request a burst summary via directive. No new code path needed.

**On 2.2 (Experiment Log Health Check):**

Yes. This is a data integrity issue and the fix is cheap. The experiment log is a critical state file — the gate-first contract depends on it. A malformed log silently breaks the S1 loop.

One refinement: don't add a `--validate` flag to `show_experiments.sh`. That script is a display tool. Instead, add a standalone `eval/validate_experiment_log.py` (10 lines of Python — load JSON, check schema, exit 0/1). Call it from `pre-deploy.sh` and from the proposed shadow eval script. Separation of concerns.

**Verdict: ADOPT (P2).** Simple, high-value, low-cost. Prevents silent S1 loop failure.

**On 2.3 (Schedule Telemetry):**

Interesting but premature. We have 11 cron sessions/day, recently consolidated from 17. The shadow eval time budget problem was the first scheduling conflict we've hit in months. One conflict doesn't justify building a scheduling advisor.

Also, the performance JSONs already contain duration data. Anyone can eyeball `jq '.duration' logs/performance/factory-*.json | sort -n` to see the distribution. A script that does this automatically is nice-to-have, not need-to-have.

If we find ourselves adjusting the schedule monthly, then yes, build the advisor. Right now we adjust it quarterly at most.

**Verdict: DEFER.** Revisit if scheduling conflicts recur after the shadow eval cron is added.
