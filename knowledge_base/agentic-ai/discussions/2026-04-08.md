## Round 1

### INNOVATOR

**Proposal 1A: CVE-2026-35020 Environment Sanitization + Version Check**

The analysis flags CVSS 8.4 OS command injection via the `TERMINAL` env var in Claude Code CLI. Our 8 daily cron scripts are all exposed — they invoke `claude` without sanitizing the environment first. Concrete fix: add a 3-line preamble to all 8 `scripts/daily_*.sh` files:

```bash
# CVE-2026-35020 mitigation: sanitize TERMINAL before invoking claude
export TERMINAL="${TERMINAL_SAFE:-xterm}"
unset -v TERMINAL_UNSAFE_CHARS  # defense-in-depth
```

Plus add a version assertion that checks `claude --version` output against a minimum safe version. If below threshold, abort the run and log to `logs/security/`. This is P0 — we should do it today.

**Proposal 1B: Initiator-Type Permission Context (AWS IAM Pattern)**

AWS just shipped `aws:ViaAWSMCPService` and `aws:CalledViaAWSMCP` — IAM context keys that distinguish agent actions from human actions. Our `post-tool-use.sh` currently treats all tool calls identically. Proposal: add an `initiator_type` field to the hook's permission context with three values:

- `cron-automated` — from daily scripts (restrict destructive git ops like `push --force`, `reset --hard`)
- `human-interactive` — from a live Claude Code session (current behavior)
- `orchestrator-delegated` — from subagent delegation (intermediate restrictions)

Implementation: each `daily_*.sh` script exports `CLAUDE_INITIATOR_TYPE=cron-automated` before invoking claude. `post-tool-use.sh` reads this env var and applies the appropriate policy. This directly addresses the analysis finding that "automated environments have different threat surfaces."

**Proposal 1C: Model Deprecation Guard in Pre-Deploy Gate**

Four model deprecation deadlines are converging within 95 days (Haiku 3: Apr 19, Sonnet 3.7: May 11, etc.). While our agents use Opus 4.6 / Sonnet 4.6, we should add a `model_deprecation_check.sh` to the pre-deploy gate. It greps all agent definitions and eval configs for known deprecated model IDs and fails the gate if any are found within 30 days of retirement. Low effort, high safety value — prevents a silent breakage if someone accidentally references an old model.

---

### ENGINEER

**Response to 1A (CVE Sanitization): ADOPT — but simplify.**

The env var sanitization is a no-brainer. However, the version assertion is fragile — `claude --version` output format may change, and parsing version strings in bash is error-prone. Simpler: just `unset TERMINAL` in the preamble. This completely neutralizes the attack vector without needing to know the patched version. If the user's system is unpatched, the `unset` still protects us. The version check adds complexity without marginal security benefit given the `unset` already closes the hole.

Verdict: **ADOPT the `unset TERMINAL` preamble across all 8 scripts. Skip the version assertion — the unset is sufficient defense-in-depth.**

**Response to 1B (Initiator-Type): ADOPT — phased.**

This is sound and mirrors an industry-proven pattern. But the full 3-tier policy matrix doesn't need to ship today. Phase it:
1. **Immediate (this week)**: Add `CLAUDE_INITIATOR_TYPE` export to all daily scripts + read it in `post-tool-use.sh`. Log it in structured output. No enforcement yet — just visibility.
2. **Next week**: Define the restricted operation set for `cron-automated` (block force-push, reset --hard, branch -D) and enforce.
3. **Phase 5**: Add `orchestrator-delegated` when subagent delegation actually exists.

This avoids speculative complexity (we don't have orchestrator delegation yet) while getting the hardest part — plumbing — done now.

Verdict: **ADOPT Phase 1 (env var + logging) this week. ADOPT Phase 2 (enforcement) next week. DEFER Phase 3 (orchestrator tier) to Phase 5.**

**Response to 1C (Deprecation Guard): ADOPT.**

Minimal effort, real protection. One grep-based script, a static list of deprecated model IDs with dates, and a 30-day warning window. Only concern: where does the deprecation list live? It should be a simple JSON file (`eval/deprecated_models.json`) updated by the researcher agent when it detects new deprecation announcements, rather than hardcoded in the script.

Verdict: **ADOPT. Implement as `eval/model_deprecation_check.sh` + `eval/deprecated_models.json`. Wire into pre-deploy gate.**

## Round 2

### INNOVATOR

**Proposal 2A: Security Orchestrator Agent (CodeMender-Inspired)**

Google's CodeMender shipped 72 upstream security fixes using a multi-agent pattern: orchestrator + specialized sub-agents for static analysis, dynamic testing, fuzzing, and SMT solving. Our security stack has grown to 7+ discrete scripts (`check-permissions.sh`, `mcp_config_validator.sh`, `cmd_chain_monitor.sh`, `cost_ceiling.sh`, MCP depth monitor in `post-tool-use.sh`, dependency audit in `pre-deploy.sh`, content scanning). A security orchestrator agent could:

1. Run all security checks in a coordinated sequence
2. Reason about gaps between tools (e.g., "the MCP validator passed but the depth monitor was never tested against this config")
3. Generate coherent security assessment reports instead of scattered log entries
4. Automatically triage findings by severity

Implementation: promote the `security-auditor` Changeling role to a full agent at `.claude/agents/security-orchestrator.md` with Read/Glob/Grep/Bash tools (review-class permissions). It orchestrates the existing scripts — no new security tooling needed.

**Proposal 2B: Cross-Platform SKILL.md Distribution Layer**

The analysis confirms CLI architectural convergence is complete: `.claude/skills/` and `.gemini/skills/` now share virtually identical structures. Proposal: write a `scripts/skill_export.sh` that:

1. Takes a `.claude/skills/<name>/SKILL.md` as input
2. Generates a `.gemini/skills/<name>/SKILL.md` equivalent (format mapping)
3. Produces a distribution manifest (`skill_manifest.json`) with checksums for both formats

This makes our Skills consumable by both Claude Code and Gemini CLI users — a Phase 7 distribution opportunity we can prototype now at near-zero cost.

**Proposal 2C: Qwen 3.6 Plus Evaluation Track for Phase 3 Distillation**

Qwen 3.6 Plus hits 78.8% SWE-bench at ~3x Opus speed with a free preview. Instead of waiting for Phase 6, we could test it *now* as a Phase 3 distillation candidate. Run our existing eval suite against Qwen 3.6 Plus via the `meta-agent-factory` skill — if it achieves >85% of Opus trigger rate at dramatically lower cost, it could replace Haiku as the distillation target in Phase 3.5.

---

### ENGINEER

**Response to 2A (Security Orchestrator): DEFER.**

The current 7-script security stack works. Each script has a clear responsibility and fails independently. An orchestrator agent adds:
- Latency: running an LLM to decide *which scripts to run* is slower than just running all of them
- Complexity: the agent definition, tool permissions, and failure modes all need testing
- Cost: every agent invocation burns tokens

The value proposition ("reason about gaps between tools") is real but premature. We have 7 scripts, not 70. A senior engineer can review the output of 7 scripts faster than an agent can reason about them. The right trigger is when the security stack exceeds the point where a human can't efficiently review all outputs — maybe 15+ tools.

Counter-proposal: instead of an agent, write a simple `eval/security_suite.sh` that runs all 7 checks in sequence, aggregates pass/fail, and outputs a single structured JSON report. 90% of the orchestration benefit at 10% of the cost.

Verdict: **DEFER the full agent to Phase 5 (when the stack is larger). ADOPT the `security_suite.sh` aggregator script as a simpler alternative now.**

**Response to 2B (Cross-Platform Distribution): DEFER.**

The convergence is real, but the format mapping is speculative. We haven't verified that `.gemini/skills/` uses identical YAML frontmatter fields, identical trigger mechanisms, or identical tool names. The "virtually identical" claim from the analysis is about *architectural patterns*, not byte-level format compatibility. Before writing an export script, someone needs to:

1. Create a Gemini CLI project and inspect the actual `.gemini/skills/` format
2. Document the exact field mappings and incompatibilities
3. *Then* write the converter

Shipping a converter without verified format specs will produce broken output and give a false sense of cross-platform readiness.

Verdict: **DEFER. Add a research task to Phase 5 planning: "Verify .gemini/skills/ format compatibility and document field mapping." The export script follows the research, not the other way around.**

**Response to 2C (Qwen Distillation): REJECT for Phase 3; DEFER to Phase 6.**

Three problems:
1. **Free preview won't last.** Production pricing is unknown. Building a distillation pipeline around a free tier is building on sand.
2. **Qwen 3.6 Plus is a server-side model.** Our Phase 3 distillation target (3.5) is about reducing *cost* by running on cheaper models. We'd need to benchmark Qwen's production pricing against Haiku 4.5 — and that pricing doesn't exist yet.
3. **Phase 3.5 isn't active.** We haven't collected the 100 successful Opus output cases yet (task 3.5 line 1). Starting a distillation evaluation without the training data is premature.

The right action: add Qwen 3.6 Plus to the Phase 6 model evaluation matrix (as the analysis already recommends). When Phase 6 begins and production pricing exists, evaluate it alongside Gemma 4 variants.

Verdict: **REJECT for Phase 3 distillation. Already tracked as P2 for Phase 6 in the analysis.**

**Proposal 3B: Pre-Deploy Security Suite Aggregator**

Per the Engineer's counter-proposal in Round 2: implement `eval/security_suite.sh` that runs all security checks in sequence and produces a unified JSON report. Structure:

```json
{
  "timestamp": "2026-04-08T12:00:00Z",
```

Wire it into `pre-deploy.sh` as a single call replacing the individual script invocations. This simplifies the pre-deploy gate and makes security check results machine-parseable for the reviewer agent.
