### Round 1: Cross-Pollination — What Can We Borrow?

**INNOVATOR**: Today's analysis identified something I think is underexploited: CP2 — the memory pattern divergence between Claude and Gemini. Our agents restart fresh every cron session. The researcher writes to `knowledge_base/`, the research-lead writes directives, but no agent remembers *what it tried and failed* across sessions. Gemini's Memory Profiles concept — "high-accuracy, low-latency detail recall" — points to a gap: we should build **cross-session agent state** for the researcher and factory-steward.

Concretely: each agent writes a `session_state.json` at the end of its run (what it attempted, what succeeded, what failed, what was deferred). Next session reads it before starting. File: `logs/state/{agent-name}-latest.json`. This is distinct from performance JSON (which tracks metrics) — this tracks *intent and outcome*.

Second proposal: CP1 — the unified sub-agent dispatch convergence — should become the **centerpiece of the S3 format comparison matrix**. Both vendors independently arrived at `dispatch(type, prompt) → result`. The factory queue has "agent format comparison matrix" as P2 item #6. I'm proposing we sharpen it: instead of a broad format comparison, make it a **dispatch-primitive comparison** — the narrowest possible S3 feasibility test. Compare `Agent(subagent_type=X, prompt=Y)` vs `invoke_subagent(agent=X, prompt=Y)` semantics. If dispatch is portable, the rest of S3 becomes tractable. If it's not, S3 needs a different architecture.

Third: CP3 — the security model divergence. Anthropic does authorization (what can you do?), Google does authentication (who are you?). Neither does both. Our Phase 5 could **layer both**: per-agent GPG keys for git commit identity (lightweight SPIFFE equivalent for local use) + existing `check-permissions.sh` for tool authorization. This gives us a novel combined model that doesn't exist in either vendor's framework — and it's citable in the S2 paper.

**ENGINEER**: Let me take these in order.

**Session state (cross-session memory)**: I like the concept but I'm worried about scope creep. Our agents already produce three persistence artifacts per session: (1) performance JSON, (2) log files, (3) git commits with messages. The researcher also writes sweep/analysis/discussion files. The research-lead writes directives. What exactly would `session_state.json` contain that isn't already captured?

The real gap is *intent that didn't result in an artifact*. Example: the researcher tried 4 searches for A2A v1.2, found it was a media error, and resolved it — but if it hadn't resolved, the next session would repeat the same 4 searches. That's the waste. But this is a narrow problem. A full `session_state.json` schema is overengineered for "remember failed investigations." Simpler: append a `## Unresolved` section to sweep files listing open questions. The next sweep's prompt already reads prior sweeps.

**Verdict: REJECT the JSON schema. ADOPT a lighter version** — add an `## Unresolved Questions` section to sweep file format. Near-zero implementation cost. The researcher already reads prior sweeps. No new file, no new schema, no new read path.

**Dispatch-primitive comparison**: Yes — this is the right sharpening of the S3 format comparison. Instead of boiling the ocean ("compare all aspects of SKILL.md vs Gemini Skills"), focus on the one primitive that both vendors converged on independently. If `dispatch(type, prompt) → result` is semantically identical across vendors, then a portable agent definition only needs to express: identity, description, dispatch behavior, and tool permissions. The wire format differences become a transpiler problem. This doesn't create a new factory item — it **refines** existing P2 item #6. Update the item description, don't add a new queue entry.

**Verdict: ADOPT as a refinement to existing P2 item #6.** Zero queue impact. Sharpens the work rather than expanding it.

**Layered security model (GPG + permissions)**: This is architecturally sound but premature. Our Phase 5 design hasn't started. Adding GPG key infrastructure now means maintaining it across all 11 daily agent sessions before we know what Phase 5's actual security requirements are. The `check-permissions.sh` static validator is sufficient for Phase 4. The *design note* is valuable — the *implementation* is Phase 5 work.

**Verdict: DEFER implementation to Phase 5 design. ADOPT as a design note** — add to the Phase 5 governance mapping (factory P1 item #3) that our security model should layer authentication (agent identity) atop authorization (permission scoping). This is a one-paragraph addition to an existing item.
