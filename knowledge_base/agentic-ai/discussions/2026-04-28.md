### Round 1: Capitalizing on the Two Architecturally Significant Findings

**INNOVATOR**: Today produced two findings that change the design space, not just our awareness of it. We should act on both.

**Proposal 1 — Inspect-Resume watchdog as Phase 5 design centerpiece (CP1).** Finding 2 confirms that subagent resumability is now documented (Apr 28): the parent receives an `agentId` in the Agent tool result and can `resume: sessionId` to continue an interrupted subagent. Combined with our planned TCI watchdog, this inverts the recovery loop from "kill+retry" to "pause+inspect+resume." Concretely: add a Phase 5 design note at `knowledge_base/agentic-ai/evaluations/inspect-resume-pattern.md` that specifies (a) the watchdog state machine — DETECTED_ANOMALY → PAUSED → HUMAN_INSPECT → RESUMED|ABORTED, (b) the anomaly signals (token-burn, suspicious tool sequence, cost-ceiling breach), and (c) how the resumed subagent receives corrective context via the `prompt` field on resume. This pattern doesn't exist in either vendor's docs — it's the natural composition of two primitives, and it's directly citable in the S2 paper's recovery-pattern section.

**Proposal 2 — Transpiler stub repository for S3 (Finding 6).** The dispatch-primitive comparison verdicted "transpiler feasible." The empirical validation is gated on Gemini CLI install, but the architectural surface is now mapped well enough to begin a stub. Concretely: create `tools/dispatch-transpiler/` with (a) a `canonical-skill-schema.json` JSON Schema describing the portable subset (description, system prompt, tools, mcpServers, model alias), (b) `transpile_to_claude.py` that emits a `.claude/agents/<name>.md` from canonical, (c) `transpile_to_gemini.py` that emits the equivalent `.gemini/agents/<name>.md` with Claude-only fields stripped (with warning logs for `maxTurns`, `effort`, `permissionMode`, `background`, `disallowedTools`, `memory`). One round-trip test against `meta-agent-factory.md` proves the abstraction holds. Implementation cost: ~4-6 hours. This locks in S3 architectural progress without waiting on the install.

**Proposal 3 — `PostToolUse.duration_ms` optimizer wiring (Finding 3).** Per-tool latency is now in the hook payload as of CC v2.1.119. The optimizer's eval today knows whether a test passed; it doesn't know which tool calls dominated time. Concretely: extend `eval/run_eval_async.py` to capture `duration_ms` from the hook payload and store per-tool latency in `eval/experiment_log.json` next to the pass/fail vector. This is the precondition for the optimizer evolving from pure trigger-rate optimization to joint trigger+latency optimization (an S1 advance). Gated on Jonas upgrading CC, but the wiring can be written now and dormant until upgrade.

**ENGINEER**: Three substantive proposals — let me work through them.

**Inspect-Resume design note**: Strongly agree on the pattern; pushing back on the file location and scope. We already have the Phase 5 design index (factory P1 item #1). A standalone file at `evaluations/inspect-resume-pattern.md` fragments the design surface — this is exactly the pattern Round 1 of the Apr 24 discussion warned against. The state machine specification belongs **inside** the Phase 5 design index as a new section "5.4 Recovery: Inspect-Resume Pattern." That's where the watchdog is already partially scoped. Cost is then ~30 min of factory time, not a new file lifecycle.

Also: the anomaly signal list (token-burn, tool-sequence, cost) needs to be precise. We already have the cost ceiling detector in `scripts/lib/cost_ceiling.sh`. The PostToolUse hook can detect tool-sequence anomalies (e.g., 5 consecutive Bash calls to a deny-rule-adjacent binary). Token-burn is harder — there's no per-session token counter exposed today. The design note should be honest about what's detectable now vs. what needs additional instrumentation.

**Verdict: ADOPT as a section within the Phase 5 design index, not a standalone file.** Priority P1 — this is the most architecturally consequential single addition we can make this week. The S2 paper benefits directly.

**Transpiler stub**: I want to push back hard on timing. We've delivered the *documentation* analysis today — the empirical step is gated on Gemini CLI install (still blocked). Building the transpiler scaffolding now means: (a) writing code that targets a vendor format we cannot test against, (b) maintaining it for an unknown number of weeks until install happens, (c) discovering at install time that one of our assumptions about Gemini's frontmatter parsing is wrong, and rewriting. The dispatch comparison's open questions (section 8) are explicit: "does Gemini actually expose `invoke_subagent` as a callable tool name, or is it purely an internal dispatcher name?" If the answer is "internal only," the entire `transpile_to_gemini.py` design changes.

Counter: write the **canonical schema** only — no transpilers. The schema is testable against our existing 23 Claude agents (they should all round-trip through canonical → Claude). Schema-only commitment is reversible. Transpiler implementation waits for empirical validation post-install.

**Verdict: ADOPT the canonical schema. DEFER the transpiler implementations.** Schema lives at `tools/dispatch-transpiler/canonical-skill-schema.json`. Priority P2. Transpilers wait for Gemini CLI install.

**`duration_ms` optimizer wiring**: This is the right S1 advance, but the timing is wrong. Jonas's CC upgrade is on day 5+ with no fixed date. v2.1.120 has eight regressions (afternoon sweep) and the recommendation just shifted from v2.1.118 → v2.1.119 (avoid v2.1.120 entirely). Building hook-payload parsing logic now means writing code against an API surface that may shift again before we can run it. Worse, our `run_eval_async.py` doesn't currently use hooks at all — it parses the `claude -p` stdout and extracts pass/fail signals. Wiring `duration_ms` requires either (a) a hook script that writes per-tool timing to a file the eval reads, or (b) parsing OTEL output. Both are real engineering, not "just add a field."

Counter: file this as a **shovel-ready design note** that specifies what to build when the upgrade lands, not the build itself. Document the hook payload field, the storage location in `experiment_log.json`, and the optimizer signal change ("joint trigger+latency optimization" requires defining a joint loss function). When CC v2.1.119 is installed, implementation is mechanical.

**Verdict: ADOPT as a design note in `eval/optimizer-latency-roadmap.md`. DEFER implementation.** Priority P3 (post-CC-upgrade work). Cost: ~15 min for the design note now.
