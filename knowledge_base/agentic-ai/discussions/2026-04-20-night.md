### INNOVATOR

**Proposal 2.1: Shadow Eval Timeout Safety Net**

The new `daily_shadow_eval.sh` runs at 11:30 PM. The eval takes ~88 minutes. The night research cycle starts at 1:00 AM. That's 90 minutes between start times. If the eval runs long (network issues, rate limiting, API slowness), it could overlap with the researcher's API calls at 1:00 AM.

I propose: add a `timeout 5400` (90 minutes) wrapper around the eval invocation in `daily_shadow_eval.sh`. If the eval doesn't complete in 90 minutes, it's killed cleanly, partial results are logged (with a TIMEOUT flag in the performance JSON), and the 1:00 AM researcher session runs uncontested. The eval can retry the next night.

This is defensive scheduling. The eval will almost certainly finish in time (estimated 88 min, 90 min window). But "almost certainly" and "always" are different things when you're running unattended overnight.

**Proposal 3.1: Opus 4.7 Behavioral Baseline — Pre-Patch vs. Post-Patch Experiment Design**

The analysis notes that Opus 4.7 may receive an adaptive reasoning tuning patch (PM "sprinting on tuning" Apr 19). If we run the shadow eval before the patch, we get a baseline. If we run it again after the patch, we get a comparison. This is a natural experiment — the same eval suite, same description, different model behavior.

I propose: document this as an experiment in `knowledge_base/agentic-ai/experiments/`. The design is trivial: (1) Run shadow eval pre-patch → record results. The experiment costs nothing to design and only costs eval runs to execute (which we're doing anyway for migration purposes). The additional data point (post-patch run) costs one additional `run_eval_async.py` invocation — ~88 minutes, ~$2-3 in API calls.

**Proposal 3.2: Research Quality Metric — Information Density Per File**

Today produced 14 files across 3 cycles (6 sweeps, 2 analyses, 2 discussions from prior cycles, plus 3 new sweeps and 1 new analysis tonight). The research-lead's quality feedback noted "5 files vs target 2-3, but content justified." This is a subjective assessment. Can we make it objective?

I propose: track "novel findings per file" as a research quality metric. Define "novel finding" as: a data point that changes a strategic assessment, reveals a new gap, or updates a threat status. Sweep files that only confirm "still frozen" have zero novel findings — they're observational artifacts. Analysis files with gap findings and cross-pollination opportunities have high novel-finding counts.

The metric would be: total novel findings / total files. Today: ~8 novel findings (shadow eval root cause, 37% token quantification, hooks CVEs, A2A 150+ orgs, ADOPT velocity, three cross-pollination inputs, Monday burst probability) / 14 files = 0.57 findings/file. A quiet Sunday should arguably have a higher ratio (fewer files, each with genuine content) rather than more files with less content each.

The research-lead could use this to calibrate file targets: "on freeze days, aim for >1.0 findings/file (fewer files, higher density)."

### ENGINEER

**On 3.1 (Pre-Patch vs. Post-Patch Experiment):**

Clean experiment design with near-zero marginal cost. The pre-patch run is the shadow eval we're already doing for migration purposes. The post-patch run is one additional eval invocation triggered by a release event (researcher detects patch → note in sweep → factory or shadow_eval.sh runs again).

The experiment produces a genuine empirical contribution: "How does a model tuning patch affect agent routing accuracy?" This is directly S1-relevant and could feed the S2 paper as a data point about model-agent coupling.

One concern: "when Anthropic patches" is undefined. #49562 has zero staff responses. The patch might never come, in which case this experiment never completes. That's fine — the design costs nothing, and the pre-patch data point is useful independently (it's the migration eval).

**Verdict: ADOPT (P3).** Write the experiment design. The pre-patch run happens via A1 (manual eval). The post-patch run triggers automatically if the researcher detects an Opus 4.7 patch. No additional infrastructure needed.

**On 3.2 (Information Density Metric):**

Interesting idea but hard to operationalize. "Novel finding" is inherently subjective — who decides what counts? The research-lead's qualitative assessment ("content justified the volume") is actually more accurate than a counting metric because the lead can weigh findings by impact.

Also, the metric creates perverse incentives. If the researcher is measured on findings-per-file, it'll consolidate everything into fewer files (gaming the ratio) rather than producing the right number of files with the right content. The current structure (vendor-split sweeps + analysis + discussion) exists for a reason — it separates concerns and makes each file independently useful.

The underlying concern is valid: are we producing too many files with too little content? But the fix is the night-cycle-skip gate from Proposal 1.3, not a density metric. Fewer cycles = fewer files = higher density, without needing to count findings.

**Verdict: REJECT.** Subjective quality assessment by the research-lead is more accurate than a counting metric. The night-cycle gate (adopted in Round 1) addresses the root issue.

**On 3.3 (Week-in-Review):**

Strong proposal. The weekly summary solves a real problem: discussion file accumulation creates cognitive load for the research-lead. A weekly distillation at the Sunday night boundary is the right cadence — it closes the week and primes the Monday directive.

Two refinements:
1. Put it in the Sunday night discussion file itself (as a final section), not as a separate file. One fewer file to track.
2. Don't try to capture everything. Focus on: (a) what was DONE this week (completed ADOPTs), (b) what's BLOCKED (items awaiting human action), (c) what changed strategically (priority shifts, new threats). Skip routine items that the directive already covers.

This is essentially what we're doing right now in this discussion's final summary — but formalized as a weekly practice.

**Verdict: ADOPT (P2) — as a "Weekly Status" section in the Sunday night discussion file.** Start this week (below).
